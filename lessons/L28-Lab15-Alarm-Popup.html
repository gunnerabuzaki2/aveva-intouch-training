Lesson 28: Alarm Historization & Historical Alarm Display

Module 5 — Alarms and Events Visualization | Pages 363–368

28.1 Enabling Historization at the Object Level

In AVEVA System Platform, alarms and events are generated continuously as the process runs. However, once an alarm returns to normal or an event completes, it disappears from the live alarm display. To preserve a permanent record for analysis, auditing, and regulatory compliance, you need to historize — that is, log — these alarms and events to a database. Historization happens at two levels: the individual object level and the system-wide Alarm DB Logger level.

Key Concept: Object-Level Historization
At the object level, events can be historized by checking the Log system events in addition to user events checkbox in the Log Change feature. Alarms can be historized by enabling the State alarm feature. Together, these two settings ensure that both user-initiated actions (like setpoint changes) and system-generated alarms (like high-temperature conditions) are recorded.

To enable historization on a specific attribute, open the Object Editor in the IDE and navigate to the attribute you want to track. In the Feature Controls section, you will find two key checkboxes:

  • State alarm — When checked, this enables alarm logging for the attribute. You can configure alarm details such as category, priority (0–999), alarm message, and active alarm state.
  • Log change — When checked, this enables event and attribute change logging, recording every time the value changes and who initiated the change.
Object Viewer — the LSClosed attribute has State alarm enabled (priority 500) and Log change checked, ensuring both alarms and events for this limit switch are historized.

28.2 Configuring the Alarm Client for Historical Mode

The Alarm Client tool is the primary control used to display alarms in InTouch HMI graphics. By default, it shows live (current) alarms. However, you can switch it to Historical mode to retrieve and display alarms and events that have been stored in the alarm database. To do this, double-click the Alarm Client element on your canvas to open the Edit Animations dialog, then configure the following under Alarm Mode:

  • Client Mode — Select one of the historical options from the drop-down list: Historical Alarms, Historical Events, or Historical Alarms and Events.
  • Authentication Mode — Typically set to Windows Integrated for domain environments.
  • Server Name — The hostname or IP of the SQL Server hosting the alarm database.
  • Database Name — The name of the alarm database (e.g., History Blocks or WWAlmDb).
Edit Animations — AlarmClient configured for Historical Alarms mode with Windows Integrated authentication and a Test Connection button to verify database connectivity.
Tip: Always click Test Connection after entering your database settings. This verifies that the Alarm Client can reach the SQL Server and that the authentication credentials are valid before you proceed.

28.3 Alarm DB Logger and Database Configuration

The Alarm DB Logger Manager is a legacy utility that manages the continuous logging of alarms and events from the Galaxy to a SQL Server database. It can run either as a Windows service (recommended for production) or as a normal desktop application. The Logger includes a Smart Cache that buffers alarm records when the SQL Server connection is temporarily unavailable or when alarms arrive faster than the Logger can write them.

Configuring alarm database logging involves four steps:

  1. Configure the database connection — The Alarm DB Logger requires SQL Server authentication in mixed mode. Windows Integrated authentication is not supported for the Logger itself.
  2. Select which alarms to log — Define one or more queries using Galaxy alarm provider syntax. For remote nodes, use \\NodeName\Provider!AlarmGroup; for local nodes, use \Provider!AlarmGroup. For example, \\ProdSvr\InTouch!$System selects all alarms from the $System group on the ProdSvr node.
  3. Set the logging interval — Controls how frequently the Logger writes cached records to the database.
  4. Select the run method — Choose whether to run as a service or as a desktop application.
Warning: The Alarm DB Logger only works with SQL Server authentication set to mixed mode. If your SQL Server is configured for Windows Integrated authentication only, the Logger will fail to connect. Ensure mixed mode is enabled before proceeding.

28.4 Historical Display Modes

Once alarm data is logged to the database, the Alarm Control can be configured to display it in three modes:

ModeWhat It ShowsClientMode Value
Historical AlarmsOnly alarm records from the database3
Historical EventsOnly event records (user actions, system events)4
Historical Alarms & EventsBoth alarms and events combined5

When configuring the Alarm Control for any historical mode, you need to specify:

  • The server name hosting the Alarm Database
  • Authentication information to connect to the database
  • The maximum number of records to retrieve
  • The time range or duration to display
  • Whether the Alarm Control should auto-update to the current client time
Key Concept: ClientMode Values
The Alarm Client's ClientMode property is an integer that controls which display mode is active: 3 = Historical Alarms, 4 = Historical Events, 5 = Historical Alarms & Events. You can switch modes at runtime by writing these values to the ClientMode property via pushbutton animations — a pattern you will implement in Lab 16.

Quiz — Test Your Understanding

Q1: At the object level, which two checkboxes must be enabled to historize both alarms and events for an attribute?

You must enable State alarm (to historize alarms) and Log system events in addition to user events in the Log Change feature (to historize events). Both must be checked for complete historization.

Q2: What authentication mode does the Alarm DB Logger require for its SQL Server connection?

The Alarm DB Logger requires SQL Server authentication in mixed mode. Windows Integrated authentication is not supported by the Logger itself — this is a common pitfall in production environments.

Q3: What is the ClientMode value used to display combined Historical Alarms & Events?

The ClientMode value is 5 for combined Historical Alarms & Events. (3 = Historical Alarms only, 4 = Historical Events only.)

Q4: What is the purpose of the Smart Cache in the Alarm DB Logger Manager?

The Smart Cache buffers alarm records when the SQL Server connection is down or when alarms arrive faster than the Logger can write them to the database. The Logger displays the percentage fill of the cache so operators can monitor its status.
