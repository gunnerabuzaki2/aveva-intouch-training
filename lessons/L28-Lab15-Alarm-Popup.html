Lesson 28: Object-Level Historization and the Alarm DB Logger

Source: AVEVA InTouch for System Platform 2023 Training Manual, Module 5 – Alarms and Events Visualization, Section 3 (pages 363–368)

This lesson covers how to enable alarm and event historization at the object level, introduces the Alarm DB Logger Manager for legacy alarm logging, and explains how to configure the Alarm Client for historical data display.

28.1 Object-Level Historization

Reference: Module 5, Section 3 (page 363)

At the object level, events can be historized by checking the Log system events in addition to user events checkbox in the Log Change feature. Alarms can be historized by enabling the State alarm feature on a Boolean attribute.

When you enable the State alarm feature on an attribute, you can configure:

  • Category — e.g., Discrete
  • Priority — 0 to 999 (higher numbers = higher priority)
  • Alarm message — a text string or expression (e.g., me.LSClosed.Description = "Close limit switch")
  • Active alarm state — which state triggers the alarm (e.g., Closed)
Configuring alarm historization on the LSClosed attribute — State alarm enabled with priority 500, Log change checked (page 363)

Configuring the Alarm Client for Historization

When you select one of the historical options in the Client Mode dropdown (Historical Alarms, Historical Events, or Historical Alarms and Events), additional database connectivity options appear:

SettingDescription
Authentication ModeWindows Integrated or SQL Server authentication
Server NameThe SQL Server or Historian server hosting the alarm database
Database NameThe database to query (e.g., WWAlmDb, A2ALMDB, or History Blocks)
Configuring the Alarm Client for Historical Alarms mode with Windows Integrated authentication (page 363)

28.2 The Alarm DB Logger Manager

Reference: Module 5, Section 3 (page 364)

The legacy Alarm DB Logger application provides a way to continuously log alarms and events to a SQL Server database. It can run as either a system service or a normal desktop application.

Key Concept: The Alarm DB Logger Manager includes a Smart Cache that stores alarm records when the SQL Server connection is down or when alarms arrive faster than the logger can write them. The Smart Cache fill percentage is displayed in the manager's interface.

Configuring Alarm Database Logging

To configure the Alarm DB Logger, you use its configuration wizard to:

  1. Configure the connection to the alarm database (must use SQL Server authentication in mixed mode)
  2. Select which alarms to log using query syntax
  3. Set the logging interval for writing records to the database
  4. Select the run method (service or desktop application)

Alarm Query Syntax for the Logger

The Alarm DB Logger uses the same query syntax as other alarm clients:

SyntaxDescription
\\NodeName\Provider!AlarmGroupAlarms from a remote node's specific alarm group
\Provider!AlarmGroupAlarms from the local node's alarm group
\\Galaxy!AreaAll alarms from a specific area using the Galaxy provider
Important: The Alarm DB Logger only works with SQL Server authentication — Windows Integrated authentication is not supported. SQL Server must be configured in mixed mode.

28.3 Historical Alarm Display Modes

Reference: Module 5, Section 3 (pages 365–366)

The Alarm Client control supports three historical display modes:

ModeWhat It Shows
Historical AlarmsOnly alarm records from the Alarm Database
Historical EventsOnly event records from the Alarm Database
Historical Alarms and EventsBoth alarm and event records combined

To configure any of these modes, you need:

  • The server name hosting the Alarm Database
  • Authentication information to connect
  • The maximum number of records to retrieve
  • The time range or duration to display
  • Whether the Alarm Control should auto-update to the current client time

Knowledge Check

1. What are the two ways to historize data at the object level?

Click to reveal answer

Events are historized by checking the "Log system events in addition to user events" checkbox in the Log Change feature. Alarms are historized by enabling the State alarm feature on a Boolean attribute.

2. Why does the Alarm DB Logger require SQL Server mixed mode authentication?

Click to reveal answer

The Alarm DB Logger only supports SQL Server authentication — it does not support Windows Integrated authentication. Therefore, SQL Server must be configured in mixed mode to allow both authentication types.

3. What is the Smart Cache in the Alarm DB Logger Manager?

Click to reveal answer

The Smart Cache is a buffer that stores alarm records when the SQL Server connection is down or when alarms arrive faster than the logger can write them to the database. It prevents data loss during connectivity issues.

4. What is the difference between Historical Alarms mode and Historical Alarms and Events mode in the Alarm Client?

Click to reveal answer

Historical Alarms mode shows only alarm records from the database. Historical Alarms and Events mode shows both alarm records and event records combined in a single view.
