29.1 Lab Introduction and Objectives

Lab 16 — Building a Historical Alarm Display

In a previous lab, you created a display for current alarms and events. In this lab, you will build a display for historical alarms and events — those that have already occurred and been logged to the alarm database. You will use the Alarm Client tool configured in historical mode and the ShowGraphic() script function to present the data in a popup window.

Objectives:
  • Configure the Alarm Client to retrieve and display historical alarms and events from the database
  • Create runtime filters to narrow the displayed data by group or other criteria

29.2 Creating the Historical Alarm Display Symbol

Start by creating a new symbol called AlarmHistDisplay in the Training toolset on the Graphics tab. Open it for editing, then from the Tools pane, place an Alarm Client element on the canvas. Leave some space above the Alarm Client — you will add text labels and mode-switching buttons there later. Name the element AlarmClient.

Double-click AlarmClient to open Edit Animations. Under Alarm Mode, set the following:

  • Client Mode: Historical Alarms
  • Authentication Mode: Windows Integrated
  • Server Name: Your Historian server name (your instructor will provide this)
  • Database Name: History Blocks
Edit Animations — AlarmClient configured for Historical Alarms with Windows Integrated authentication and the History Blocks database.
Warning: When you change the Database Name to History Blocks, a Configuration Warning dialog appears stating that the current configuration (including any saved password) will be lost. In a real-world environment, make sure you have recorded your existing authentication information before proceeding. For this lab, no previous configuration exists.

Click Test Connection to verify the database connection succeeds, then set the Time Range to 15 minutes and enable Update to Current Time.

Configuration Warning — this dialog appears when switching databases; the previous configuration will be overwritten.
Successful connection test — confirms the Alarm Client can reach the History Blocks database on S01ENG.
Time Range settings — 15-minute duration with auto-update enabled ensures the display refreshes with current data.

29.3 Building the Mode-Switching Buttons

To let operators switch between historical modes at runtime, you will create three pushbuttons: one for Historical Alarms, one for Historical Events, and one for both. Each button writes a specific value to AlarmClient.ClientMode and uses an Element Style animation to visually highlight when its mode is active.

First, add a text label "History Alarm Mode:" above the Alarm Client, then create the first button:

29.6 Creating Runtime Filters

One of the most powerful features of the Alarm Client in historical mode is the ability to create filters at runtime. Right-click inside the Alarm Client and select Queries and Filters to open the Query and Filter Favorites dialog. Click the Add new filter button to create a new filter.

Name the filter Line1 and construct the condition Group = 'Line1'. After clicking OK, enable the filter by checking its checkbox. The Alarm Client immediately refreshes to show only alarms belonging to the Line1 group.

Filter construction — selecting the Group field and setting the value to 'Line1' filters the alarm history to that production line.
Filtered view — only Line1 alarms are displayed after applying the Group = 'Line1' filter.
Key Concept: Runtime Filtering
Runtime filters are created directly in the Alarm Client by operators — no design-time changes required. This lets operators focus on specific areas, groups, or alarm types during an incident. Filters persist for the current session but do not modify the underlying database query.

You can also generate user events by changing the agitator speed setpoint and starting/stopping the agitator, then switch to Historical Events mode to see those events appear. The Historical Alarms & Events mode shows both types interleaved chronologically.

Historical Events — user-initiated actions like setpoint changes and command writes are recorded here.
Combined Historical Alarms & Events — both process alarms and user events appear together, giving a complete timeline.

Quiz — Test Your Understanding

Q1: What ClientMode value must you write to display Historical Events (not alarms)?

You must write the value 4 to AlarmClient.ClientMode. (3 = Historical Alarms, 4 = Historical Events, 5 = Both.)

Q2: What happens when you change the Database Name from WWAlmDb to History Blocks in the Alarm Client configuration?

A Configuration Warning dialog appears, informing you that the current configuration — including any saved password — will be deleted. You must confirm with Yes to proceed. In production, always record your existing authentication info first.

Q3: How do you create a runtime filter to show only alarms from a specific production line?

Right-click in the Alarm Client, select Queries and Filters, click Add new filter, name the filter (e.g., "Line1"), and construct the condition Group = 'Line1'. After clicking OK, enable the filter by checking its checkbox.

Q4: Why does the Element Style animation use the expression AlarmClient.ClientMode == 3 for the Historical Alarms button?

The expression evaluates to True when ClientMode is 3, which applies the Intensity4 (highlighted) style to the button. This provides visual feedback to the operator indicating which mode is currently active. When the expression is False, the button returns to its unchecked appearance.
