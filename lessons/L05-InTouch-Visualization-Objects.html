Lesson 5: InTouch Visualization + Automation Objects

Module 2 — Getting Started | Pages 41–46

5.1 Visualization Concepts

Application Server provides tools for developing interactive and animated graphical displays. The behavior of these displays — including animations and visual representations — can be linked to properties of processes and equipment through real-time data provided by System Platform. This means your graphics are not static images; they are live, dynamic windows into your plant's operation.

System Platform visualization clients support multiple user interface capabilities, including keyboard, mouse, tablet, and touch-screen inputs, as well as multi-screen outputs. This flexibility ensures that operators can interact with the system using whatever input method is most appropriate for their environment.

Key Concept: Two Visualization Clients
System Platform offers two visualization clients that can coexist in the same solution and share data and graphics: InTouch for System Platform (traditional development tools for creating displays from scratch) and Operations Management Interface (OMI) (a modern visualization engine for OT/IT convergence with rapid-design capabilities). Note that InTouch is not included with System Platform Enterprise.

5.2 What is InTouch for System Platform?

InTouch for System Platform is configured and licensed as a visualization client that works alongside System Platform, displaying data from objects defined in an Application Server Galaxy. It uses traditional development tools to create displays and navigation, giving you full control over the look and feel of your operator interfaces.

A critical distinction is that of a Managed InTouch Application. A managed application is one that is created from within the System Platform IDE and deployed as an object in the Application Server application. This is different from a standalone InTouch application. Creating a managed application allows you to use symbols (industrial graphics), and the application exhibits the same behavior as other IDE objects — it can be checked out, checked in, and deployed through the standard System Platform workflow. However, it opens in an external development environment called WindowMaker.

5.3 InTouch Components

InTouch consists of three main components that work together:

ComponentPurposeDescription
WindowMakerDevelopment environmentWhere you create animated, click-sensitive display windows using object-oriented graphics. Graphics are embedded in InTouch windows.
WindowViewerRuntime environmentDisplays the graphic windows created in WindowMaker. This is what operators interact with during production.
Application ManagerOrganization toolOrganizes the applications that are created.

5.4 Visualization Content Types

Several types of content can be used to build visualization client applications:

Industrial Graphics (also called symbols) allow you to create customized graphical representations of processes and integrate other visualization tools. They can be embedded or associated with automation objects in Application Server, meaning alarms, history, logic, and graphics can all be defined with the same automation object. This enables two major benefits: object reuse (objects derived from templates are preconfigured to reference applicable attributes) and centralized development (graphical representations are managed centrally).

Client Controls provide functionality through .NET controls that you can embed in symbols. External .NET controls can be imported into a Galaxy and used in symbols — symbols with embedded .NET controls are fully supported by InTouch for System Platform.

Widget Controls are HTML5 widget controls included in Application Server, providing modern web-standard UI components.

Key Concept: AVEVA Connect for Graphics
AVEVA Connect provides a cloud-based repository for sharing graphics between locations, Galaxies, and nodes. Graphics are stored in stores (which can be global, tenant, or user-specific) with configurable access levels. This allows teams to build centralized graphic libraries that multiple users and projects can access. An AVEVA Connect account is required, and users can have read-only or read/write access depending on their role.

5.5 InTouchViewApp Object

The InTouchViewApp object is the automation object that represents an InTouch application within the Application Server environment. It manages the check-in, check-out, and deployment of the InTouch application, integrating it fully into the Galaxy's management framework.

The InTouchViewApp object inherits capabilities from the Application Server InTouchProxy object, which means InTouch tags can be referenced by Application Server Message Exchange clients as InTouchViewApp attributes. This is how other objects in the Galaxy can interact with InTouch tag data as if it were native Application Server data.

Through the InTouchViewApp, Application Server manages:

  • Check-in and check-out for multiple users
  • Security restrictions on who can deploy the application and when
  • Backup and restore
  • Deployment and undeployment

5.6 InTouchViewApp Object Rules

There are several important rules that govern how InTouchViewApp objects can be created and used:

  • InTouchViewApp instances cannot be created directly from the base template — you must first create a derived template
  • Only one level of template derivation is allowed from the base InTouchViewApp template
  • An instance cannot be derived from an existing instance — only from templates
  • InTouchViewApp objects cannot host other objects
  • Only ViewEngine objects can host and deploy InTouchViewApp objects
  • An InTouchViewApp can be assigned to an Area Object if its alarms should belong to that area
  • Undeployment is not permitted if the object is currently loaded on a target node
Important: When you make changes to the InTouch application through WindowMaker launched from the IDE, the original InTouch application files are not modified. However, you should never access files in the file repository directory directly or make assumptions about the repository structure.

5.7 ViewEngine Object

The ViewEngine object is a simplified version of the AppEngine object. It is the runtime host for InTouchViewApp objects (and ViewApp objects). Think of the ViewEngine as the engine that makes your InTouch application actually run within the System Platform ecosystem.

Key characteristics of the ViewEngine object:

  • It is assigned to and deployed to a WinPlatform object
  • It hosts and executes InTouchViewApp objects; the ViewEngine's scan rate determines the scan rate of all hosted InTouchViewApp objects
  • It contains initialization logic for setting up InTouchViewApp objects so they can communicate with other Galaxy objects
  • It does not need to be running for WindowViewer to execute scripts or access process data
  • It can host Application Server scripts and UDAs that continue running when WindowViewer is shut down
  • It provides configuration and runtime attributes
  • It contains runtime diagnostic attributes that can be monitored, alarmed, and historized
  • It can include Historian configuration — all hosted InTouchViewApp objects use this historian
  • It does not support redundancy
Tip: Because InTouchViewApp objects serve as proxies for InTouch tags, using a shorter ScanPeriod on the ViewEngine object can minimize the latency of tag value updates when referencing tags as InTouchViewApp attributes. However, the historization and alarm options of the ViewEngine may not always be needed depending on your architecture.

Quiz — Test Your Understanding

Q1: What are the three main components of InTouch?

WindowMaker (development environment), WindowViewer (runtime environment), and Application Manager (organizes applications).

Q2: Why must you create a derived template before creating an InTouchViewApp instance?

InTouchViewApp instances cannot be created directly from the base template. You must first create a derived template (one level of derivation allowed), then create instances from that derived template.

Q3: What type of object can host and deploy InTouchViewApp objects?

Only ViewEngine objects can host and deploy InTouchViewApp objects.

Q4: What is the relationship between the ViewEngine's scan rate and its hosted InTouchViewApp objects?

The ViewEngine's scan rate determines the scan rate of all its hosted InTouchViewApp objects. A shorter ScanPeriod on the ViewEngine can minimize tag value update latency.

Q5: What is the difference between industrial graphics, client controls, and widget controls?

Industrial graphics (symbols) are customized graphical representations that can be associated with automation objects. Client controls are .NET controls that can be embedded in symbols. Widget controls are HTML5-based controls provided by Application Server.
