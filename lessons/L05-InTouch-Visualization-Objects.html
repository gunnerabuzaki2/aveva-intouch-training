Lesson 5: InTouch Visualization + Automation Objects

Source: AVEVA InTouch for System Platform 2023 Training Manual, Module 2, Section 1–2 (pages 41–46)

This lesson introduces InTouch as a visualization client for Application Server, covering visualization content types, the InTouchViewApp object, and the ViewEngine object that hosts InTouch applications within the Galaxy.

5.1 Introduction to InTouch Visualization

Reference: Module 2, Section 1 (pages 43–44)

Application Server provides tools for developing interactive, animated graphical displays that represent industrial processes. These displays animate and react based on real-time data from System Platform objects — allowing operators to monitor equipment status, see alarms, and control processes from a single screen. System Platform visualization clients support multiple input methods (keyboard, mouse, tablet, touch-screen) and can drive multi-screen outputs for control room environments.

System Platform offers two visualization clients that can coexist in the same solution and share data and graphics:

Visualization ClientDescription
InTouch for System PlatformProvides traditional development tools for creating displays and navigation. Not included with System Platform Enterprise — licensed separately.
Operations Management Interface (OMI)A modern visualization engine that leverages HMI and SCADA for OT/IT convergence. Provides rich, modern user experiences across platforms.
Key Concept: InTouch for System Platform is configured and licensed as a visualization client that displays data from objects defined in an Application Server Galaxy. It is the traditional toolset for building operator displays.

InTouch for System Platform consists of three main components:

  • WindowMaker — The development environment where you create animated, click-sensitive display windows using object-oriented graphics. Graphics are embedded in InTouch windows.
  • WindowViewer — The runtime environment that displays the graphic windows created in WindowMaker. This is what operators see on the plant floor.
  • Application Manager — Organizes and manages the InTouch applications that have been created.

5.2 Managed InTouch Applications

Reference: Module 2, Section 1 (pages 43–44)

A Managed InTouch application is one that is created from within the System Platform IDE and deployed as an object in the Application Server Galaxy. Unlike a standalone InTouch application, a managed application is fully integrated into the Galaxy — it supports check-out and check-in workflows, multi-user collaboration, and centralized deployment. Creating a managed application also enables the use of symbols (industrial graphics).

When you open an InTouchViewApp object from the IDE, WindowMaker launches automatically. There is no separate object editor — WindowMaker is the editor for InTouch applications.

Key Concept: A Managed InTouch application behaves like any other Galaxy object: it can be checked out, checked in, backed up, restored, deployed, and undeployed through the IDE.

5.3 Visualization Content Types

Reference: Module 2, Section 1 (pages 44)

Several types of content can be used to build visualization client applications in Application Server:

  • Industrial Graphics (Symbols) — Customized graphical representations of processes that can be embedded or associated with automation objects. Industrial graphics support object reuse (deriving from templates) and centralized development. They are also referred to as symbols in Application Server.
  • Client Controls — .NET controls that can be used within symbols. External .NET controls can be imported into a Galaxy and embedded in symbols. Supported by InTouch for System Platform.
  • Widget Controls — HTML5 widget controls included in Application Server for modern web-compatible visualizations.
  • AVEVA Connect for Graphics — A cloud-based repository for sharing graphics between locations, Galaxies, and nodes. Access it through the Graphics tab in the IDE.

AVEVA Connect stores graphics in stores (global, tenant, or user-specific), and each store can contain multiple drives with different access levels (read-only or read/write). Only one user at a time can edit or save a graphic to a given drive, but multiple users can view it simultaneously.

5.4 InTouchViewApp Object

Reference: Module 2, Section 2 (pages 45–46)

The InTouchViewApp object is the Galaxy object that represents an InTouch application within the Application Server environment. It manages the check-in, check-out, and deployment lifecycle of the InTouch application. It also inherits from the InTouchProxy object, meaning InTouch tags can be referenced by other Application Server Message Exchange clients as attributes of the InTouchViewApp object.

The Application Server system uses the InTouchViewApp to manage:

  • Check-in and check-out for multiple users
  • Security restrictions on who can deploy the application and when
  • Backup and restore
  • Deployment and undeployment
Key Concept: Objects running on a ViewEngine can access InTouch tags as if they were native Application Server attributes of the InTouchViewApp object. Tags are browsed through the standard Application Server attribute browser.

5.5 InTouchViewApp Object Rules

Reference: Module 2, Section 2 (pages 45–46)

Several important rules govern how InTouchViewApp objects can be created and used:

RuleExplanation
Cannot create from base templateInTouchViewApp instances cannot be created directly from the $InTouchViewApp base template — you must first derive a template.
One level of derivationOnly one level of template derivation is allowed from the base $InTouchViewApp template.
No instance derivationYou cannot derive a new InTouchViewApp from an existing instance.
Cannot host other objectsInTouchViewApp objects cannot host child objects in the Galaxy hierarchy.
Only ViewEngine hosts itOnly a ViewEngine object can host and deploy InTouchViewApp objects.
Can assign to AreaIn the Model view, you can assign an InTouchViewApp to an Area Object if its alarms should belong to that area.
Undeployment restrictionYou cannot undeploy an InTouchViewApp that is currently loaded on a target node.
Pitfall: Do not access files in the file repository directory directly or make assumptions about the repository structure. The Galaxy manages the repository automatically.

When you create an InTouchViewApp from an existing InTouch application, the application files are copied to the repository — the originals are not modified or deleted. Once the InTouchViewApp is created, deleting the original application has no effect on the managed copy.

5.6 ViewEngine Object

Reference: Module 2, Section 2 (pages 46)

The ViewEngine is a simplified version of the AppEngine object. It hosts InTouchViewApp objects (as well as ViewApp objects) and provides the execution environment for InTouch applications. The ViewEngine supports common engine features — deployment, undeployment, startup, and shutdown — but notably does not support redundancy.

Key characteristics of the ViewEngine object:

  • Is assigned to and deployed to a WinPlatform object
  • Hosts and executes InTouchViewApp objects; the ViewEngine's scan rate determines the scan rate of all its hosted InTouchViewApp objects
  • Contains initialization logic so InTouchViewApp objects can communicate with other Galaxy objects when first deployed
  • Does not need to be running for WindowViewer to execute scripts or access process data
  • Can host Application Server scripts and UDAs that continue running even when WindowViewer is shut down
  • Provides configuration and runtime attributes, including runtime diagnostic attributes that can be monitored, alarmed, and historized
  • Can include a Historian as part of its configuration — all hosted InTouchViewApp objects share this historian
Tip: Because InTouchViewApp objects serve as proxies for InTouch tags, a shorter ScanPeriod on the ViewEngine minimizes the latency of tag value updates when referencing tags as InTouchViewApp attributes.
Key Concept: Only one ViewEngine per node is needed, but it can host multiple InTouchViewApp objects. However, only one InTouch application per node can actually run at a time.

Knowledge Check

1. What are the three main components of InTouch for System Platform?

Click to reveal answer

WindowMaker (development environment), WindowViewer (runtime environment), and Application Manager (application organizer).

2. What is the difference between InTouch for System Platform and a standalone InTouch application?

Click to reveal answer

A Managed InTouch application is created from within the System Platform IDE and deployed as a Galaxy object. It supports check-out/check-in workflows, multi-user collaboration, symbol usage, and centralized deployment — features not available in standalone InTouch applications.

3. Can an InTouchViewApp object be created directly from the $InTouchViewApp base template?

Click to reveal answer

No. InTouchViewApp instances cannot be created from the base template. You must first derive a template from $InTouchViewApp, then create instances from that derived template.

4. What is the role of the ViewEngine object in the Application Server Galaxy?

Click to reveal answer

The ViewEngine hosts and executes InTouchViewApp objects. It is assigned to a WinPlatform, provides initialization logic for Galaxy communication, and determines the scan rate for all hosted InTouchViewApp objects. It does not support redundancy.

5. How do other Application Server objects access InTouch tags?

Click to reveal answer

Objects running on a ViewEngine can access InTouch tags as if they were native Application Server attributes of the InTouchViewApp object. Tags are browsed through the standard Application Server attribute browser, thanks to the InTouchProxy functionality inherited by InTouchViewApp.
