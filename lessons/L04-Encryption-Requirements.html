Lesson 4: Encrypted Communication & System Requirements

Source: AVEVA InTouch for System Platform 2023 Training Manual, Module 1, Sections 3–4 (pages 31–40)

Understanding encrypted communication between server and client applications, plus the software and hardware requirements for running System Platform.

4.1 Encrypted Communication Overview

Reference: Module 1, Section 3 (pages 29–30)

The end-to-end communication between software applications (server and client) in System Platform can be encrypted and secured to prevent eavesdropping and malicious tampering — also known as Man-in-the-Middle (MitM) attacks. This is an important security feature for industrial environments where process data must be protected.

Key Concept: To enable encrypted communication, one node in the network must host the System Management Server (SMS). The SMS generates, manages, and distributes secure digital certificates used for establishing and maintaining encrypted communications between all nodes.

How It Works

  • One node hosts the SMS, which issues digital certificates to all nodes in the network.
  • All other nodes must be configured to connect to the SMS to become part of the same community.
  • Once a node joins the SMS community, the SMS does not need to be available at runtime — nodes will not need access to the SMS until it is time to renew certificates.
Tip: The SMS is only needed during initial setup and certificate renewal. It can be offline during normal operation without affecting encrypted communications.

Protocols Secured by Encryption

With encrypted communication enabled, all of the following protocols are secured:

4.2 Software Supporting Encrypted Communication

Reference: Module 1, Section 3 (page 1-30)

The following AVEVA software products support secured communication:

  • Application Server 2017 Update 3 and later
  • InTouch Operations Management Interface 2017 Update 3 and later
  • InTouch for System Platform 2017 Update 3 and later
  • OI Server (all versions — generic encryption support)
  • InTouch Web Client 2017 Update 3 and later
  • Historian 2017 Update 3 and later
  • Sentinel System Monitor 1.1 for System Platform 2017 Update 3 and later
Important: Communications from a specific OI Server to any other listed product can be encrypted. This applies generically to all OI Server versions.

4.3 Node-to-Node Communication

Reference: Module 1, Section 3 (page 1-31)

Node-to-node communication is unsecured under certain scenarios. Understanding these scenarios is critical for ensuring your deployment maintains proper encryption:

  • Software on one or more nodes is older than 2017 Update 3 (Communication Driver version has no impact)
  • One node points to an SMS, but the other node has no SMS configured
  • Neither node has an SMS configured
Critical Warning: If one node points to one SMS and another node points to a different SMS, there will be NO network connection between those nodes. All nodes in a community must point to the same SMS.
Key Concept: Nodes that point to the same SMS form a secured community. Nodes without an SMS or with different SMS configurations cannot communicate with each other. Legacy nodes (older software or no SMS) operate in unsecured mode.

4.5 Licensing

Reference: Module 1, Section 4 (pages 34–35)

AVEVA System Platform uses an activated licensing framework. Contact your local distributor for details about licensing and pricing.

Licensing Models

  • Perpetual licensing — permanent licenses with no expiration date. Applies to a specific purchased software version.
  • Flex licensing — subscription-based licenses renewed for a specific term. Credits are redeemable to access a wide range of software based on current needs. Visit https://sw.aveva.com/flex-subscription for details.

Activated Licensing Components

  • License Server — acquires, stores, maintains, and serves licenses. Supports redundancy and failover.
  • License Manager — web-based user interface to access and maintain licenses.

License activation can be performed online (Internet access on the License Server network) or offline (activate off-site). Both methods require Internet access at some point in the process.

4.6 Azure Active Directory Integration

Reference: Module 1, Section 4 (page 1-35)

Starting with System Platform 2023, user identity authentication and single sign-on (SSO) via Azure Active Directory (AD) is supported. After configuring Galaxy security to use an external identity provider, Azure AD can authenticate users for:

  • System Platform IDE
  • Operations Control Management Console
  • Operations Management Interface ViewApps
  • Secured and verified writes
Key Concept: Azure AD integration means users can leverage their Microsoft-managed identities to log in across all System Platform tools, eliminating the need for separate credential management and enabling enterprise-grade SSO.

Knowledge Check

1. What is the role of the System Management Server (SMS) in encrypted communication?

Click to reveal answer

The SMS generates, manages, and distributes secure digital certificates used for establishing and maintaining encrypted communications between nodes. All nodes must connect to the same SMS to form a community.

2. Why is it safe for the SMS to be offline during normal operation?

Click to reveal answer

Once a node has joined the SMS community and received its certificates, it does not need access to the SMS until certificate renewal. The encrypted communication continues working without the SMS being available.

3. What happens if two nodes point to different System Management Servers?

Click to reveal answer

There will be NO network connection between those nodes. All nodes that need to communicate must point to the same SMS to be part of the same community.

4. What is the minimum RAM requirement for a large server node handling more than 50K I/O points?

Click to reveal answer

The minimum is 16 GB RAM, with a recommendation of 24 GB. The minimum CPU is 16 logical processors with a recommendation of 32.

5. What new authentication feature was introduced in System Platform 2023?

Click to reveal answer

Azure Active Directory integration for user identity authentication and single sign-on (SSO). Users can use Microsoft-managed identities to log into the IDE, OMC, OMI ViewApps, and perform secured/verified writes.
