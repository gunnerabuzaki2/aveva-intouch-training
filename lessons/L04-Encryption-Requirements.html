Lesson 4: Encrypted Communication + System Requirements

Module 1 — Introduction | Pages 31–40

4.1 End-to-End Encrypted Communication

The end-to-end communication between software applications in a System Platform deployment — between servers and clients — can be encrypted and secured. This protection prevents eavesdropping and malicious tampering, commonly known as Man-in-the-Middle (MitM) attacks. In an industrial environment where data integrity is critical, encrypted communication is an essential security measure.

Key Concept: System Management Server (SMS)
To enable encrypted network communication, one node in your network must be configured to host and run the System Management Server (SMS). The SMS is responsible for generating, managing, and distributing secure digital certificates that are used for establishing and maintaining secure communications between all nodes.

All other nodes in the network need to be configured to connect to the SMS so they become part of the same security community. A crucial detail: the SMS does not need to be available at runtime for secure communications to function. Once a node has joined the SMS community, it will not need to contact the SMS again until it is time to renew the certificates.

4.2 Protocols Secured by Encryption

When encrypted communication is enabled, all of the following protocols benefit from the protection:

  • SuiteLink — AVEVA's proprietary communication protocol
  • Message Exchange (MX) — Used for inter-application communication
  • iData — Data access protocol
  • iBrowse — Browsing protocol for tag access
  • HCAL — Historian data communication
The architecture of secured communication across the System Platform ecosystem, showing how encryption protects data flowing between all major components.

4.3 Software Supporting Secured Communication

The following software products support secured communication. Note that most require version 2017 Update 3 or later:

  • Application Server 2017 Update 3 and later
  • InTouch Operations Management Interface 2017 Update 3 and later
  • InTouch for System Platform 2017 Update 3 and later
  • OI Servers (all versions — encryption is generic across OI Server versions)
  • InTouch Web Client 2017 Update 3 and later
  • Historian 2017 Update 3 and later
  • Sentinel System Monitor 1.1 for System Platform 2017 Update 3 and later

4.4 Node-to-Node Communication Rules

Understanding when communication is secured versus unsecured is important for planning your deployment. Node-to-node communication will be unsecured under the following scenarios:

  • The software version on one or more nodes is earlier than 2017 Update 3 (Communication Driver version does not impact this)
  • One node points to an SMS, but the other node has no SMS configured
  • Neither node has an SMS configured
Critical Warning: If one node points to one System Management Server and another node points to a different System Management Server, there will be NO network connection between those nodes at all. All nodes that need to communicate must belong to the same SMS community.
Three communication communities: two secured communities (each pointing to their own SMS) and one legacy unsecured community with no SMS. Note that secured communities cannot communicate with each other if they use different SMS instances.

4.7 Licensing

AVEVA System Platform uses an activated licensing framework. There are two licensing models available:

Perpetual licensing provides permanent licenses with no expiration date, tied to a specific purchased software version.

Flex licensing offers subscription-based licenses that are renewed for a specific term. The license provides credits redeemable for access to a range of software based on your current needs.

Licenses are managed through two components: the License Server (which acquires, stores, maintains, and serves licenses with support for redundancy and failover) and the License Manager (a web-based interface for managing licenses). Activation can be performed online or offline, though both methods require Internet access.

4.8 Azure Active Directory Integration

Starting with System Platform 2023, user identity authentication and single sign-on via Azure Active Directory (AD) is supported. After configuring Galaxy security to use an external identity provider, Azure AD can be used to authenticate users. This allows System Platform users to log into the System Platform IDE, Operations Control Management Console, Operations Management Interface ViewApps, and to perform secured and verified writes using their Microsoft-managed identities.

Quiz — Test Your Understanding

Q1: What is the role of the System Management Server (SMS) in encrypted communication?

The SMS generates, manages, and distributes secure digital certificates used for establishing and maintaining encrypted communications between nodes.

Q2: What happens if two nodes point to different System Management Servers?

There will be NO network connection between those nodes. All nodes that need to communicate must belong to the same SMS community.

Q3: Does the SMS need to be running at runtime for encrypted communication to work?

No. Once a node has joined the SMS community and received its certificates, it does not need access to the SMS until certificates need to be renewed.

Q4: What are the two licensing models available for System Platform?

Perpetual licensing (permanent, no expiration, version-specific) and Flex licensing (subscription-based with redeemable credits).

Q5: What new authentication feature was introduced in System Platform 2023?

Azure Active Directory (AD) integration for user identity authentication and single sign-on, allowing Microsoft-managed identities for logging into System Platform tools.
