Lesson 20: Scripts In Graphics

Module 3 — Symbols | Pages 249–254

20.1 Associating Scripts with Graphics

In an InTouch application, static graphics alone rarely tell the full story. You need your displays to respond to live data — to change color when a valve opens, to recalculate a value when a sensor reading shifts, or to pop up a window when an operator clicks a button. This is where scripts in graphics come in. By associating a script with a symbol placed in your InTouch window, you can animate graphics, modify their elements, and drive dynamic behavior while the application is running.

After you select a graphic embedded in an InTouch window, you choose the expression or tag reference whose value change will trigger the script. The Graphic Editor provides the interface for selecting this trigger and writing the associated script code.

Key Concept: Two Types of Symbol Scripts
InTouch recognizes two fundamental categories of symbol scripts: predefined scripts and named scripts. A predefined script runs based on the lifecycle state of the symbol itself (e.g., when it opens, while it is shown, or when it closes). A named script runs in response to a specific expression or tag reference changing state — you define the trigger expression, and the script fires whenever that expression evaluates to true, false, or transitions between the two.

20.2 Predefined Scripts

Predefined scripts work much like InTouch window scripts, but they are scoped to an individual symbol rather than to a window. Depending on how you configure the script trigger, a predefined symbol script can execute in three different scenarios:

  • Once on open — The script runs a single time after the symbol appears or is shown in the running application.
  • Periodically while shown — The script runs repeatedly at a defined interval as long as the symbol remains visible on screen.
  • Once on close — The script runs a single time after the symbol is hidden or removed from the display.

The periodic mode is particularly useful for symbols that need to update continuously — for example, a gauge that must refresh its needle position every second based on a changing tag value.

20.3 Named Scripts

Named scripts offer more flexible triggering. Instead of relying on the symbol's visibility lifecycle, you define a specific expression or tag reference as the trigger. A named symbol script can execute when:

  • The trigger expression transitions from true to false or false to true.
  • The data value associated with the trigger expression changes.
  • The quality state of the trigger expression changes (for example, a tag going from "good" to "uncertain" quality).

This makes named scripts ideal for event-driven logic — such as logging a value when a threshold is crossed, or updating a calculation whenever a downstream tag changes.

Tip: Use predefined scripts for simple visibility-driven behavior (show/hide/periodic refresh). Use named scripts when you need fine-grained control over exactly which data change triggers your logic.

20.4 Execution Types and Execution Order

An important subtlety of scripting in graphics is that the existence and execution order of scripts are inherently locked at each stage of the template hierarchy. When you create a set of scripts in a template, those scripts form an ordered block. In any downstream derived template or instance, new scripts can be added before or after that block, but the template's scripts remain grouped together in their original order.

Furthermore, scripts cannot trigger faster than the scan period of the Application Engine. If your Application Engine scans at 1-second intervals, even a script configured to run "periodically" will not execute more frequently than once per second, regardless of the interval you set in the script configuration.

Pitfall: Scripts cannot execute faster than the Application Engine's scan period. If you set a script to run every 100 milliseconds but your scan period is 1 second, the script will only run once per second. Always verify your scan period when debugging timing issues.

20.5 The ShowGraphic() and HideGraphic() Functions

One of the most powerful scripting capabilities for managed InTouch applications is the ShowGraphic() / HideGraphic() pair of functions. These allow you to display any Galaxy graphic as a popup window at runtime — without needing to embed the graphic directly in the application. By associating all Galaxy graphics with the InTouchViewApp template, you ensure that every graphic is deployed and available, whether or not it is referenced by a specific application.

Key Concept: ShowGraphic() vs. ShowSymbol Animation
InTouch has two mechanisms for displaying popups: the built-in Show/Hide Symbol animation feature and the Show/Hide Graphic script functions. They can be used independently or together. When both are active simultaneously, two independent popup windows open. The key difference is that ShowGraphic() can reference any graphic in the Galaxy (not just embedded symbols), giving you greater flexibility for building dynamic navigation systems.

When you configure a ShowGraphic() call, you can control several properties of the resulting popup window:

  • Which symbol appears as the popup
  • Whether the window includes a title bar
  • The initial position of the popup on screen
  • Whether the window can be resized
  • Whether the window is modal (blocks interaction with other windows) or modeless
  • The relative position of the popup window
  • Passing the OwningObject to the displayed symbol
  • Values of custom properties on the symbol

ShowGraphic() Syntax

The basic pattern for displaying a graphic at runtime follows this structure:

Dim graphicInfo as aaGraphic.GraphicInfo;
graphicInfo.Identity = "<Identity>";
graphicInfo.GraphicName = "<SymbolName>";
ShowGraphic( graphicInfo );

The Identity string acts as a unique identifier for the popup. You use this same identity to later close the popup with HideGraphic().

HideGraphic() Syntax

Closing a displayed graphic is straightforward — simply pass the identity string to the hide function:

HideGraphic(string identity);

You can also close a graphic using HideSelf from within the graphic itself.

Tip: Both ShowGraphic() and HideGraphic() are available only in managed or published InTouch applications. They will not function in unmanaged (standalone) applications.

20.6 Scripting the OwningObject

When you pass an OwningObject through a ShowGraphic() script function, the OwningObject resolves only relative references within the displayed symbol. Absolute references are not affected. For example, if your symbol references me.S1 (a relative reference) and you set OwningObject = "Obj2", the reference resolves to Obj2.S1. But an absolute reference like Obj3.S1 remains unchanged regardless of the OwningObject.

The OwningObject value can be a concatenation of constant strings and reference strings, and you can browse for it using the Display Automation Object Browser or type the name directly.

graphicInfo.OwningObject = "Mixer_001";
Pitfall: The OwningObject only affects relative references (e.g., me.TagName). Absolute references like ObjName.TagName are completely unaffected. If your popup graphic uses absolute references and isn't behaving as expected, check whether you should be using relative references instead.

20.7 Runtime Behavior and Popup Management

Graphics displayed by ShowGraphic() behave like ShowSymbol animation popups rather than traditional InTouch popup windows. Several important runtime behaviors apply:

  • Popup graphics always remain in front of InTouch windows — even if you click an InTouch window, it stays behind the popup.
  • You can open and close popups from any symbol in any InTouch window, managing them across the entire application.
  • If you enable in-memory graphics caching in WindowViewer memory properties, popup symbols are cached in memory for faster display. The system tracks closure order to determine symbol age, and automatically removes the oldest cached popups when the memory limit is exceeded (except those in high-priority windows).

Handling Duplicate Identities

If you attempt to open a popup with an Identity that is already displayed, the new ShowGraphic() call is silently blocked — no error or warning appears in the logger. This applies to calls from OnShow, OnHide, and WhileShowing scripts alike. Additionally, if a symbol is modal, calling ShowGraphic() cannot change it to modeless.

Tip: When the in-memory graphic cache is enabled, WindowViewer performs a memory health check each time you display a symbol with ShowGraphic() or ShowSymbol animation. This helps prevent memory exhaustion in long-running applications with many popup graphics.

Quiz — Test Your Understanding

Q1: What is the difference between a predefined script and a named script in a symbol?

A predefined script runs based on the symbol's lifecycle (on open, while shown, on close). A named script runs when a specific expression or tag reference changes state — you define the trigger expression yourself.

Q2: Can scripts execute faster than the Application Engine's scan period?

No. Scripts cannot trigger faster than the scan period of the Application Engine. Even if you configure a script to run every 100ms, it will only execute at the scan rate (e.g., once per second).

Q3: What is the key difference between ShowGraphic() and ShowSymbol animation?

ShowGraphic() can reference any graphic in the Galaxy (not just embedded symbols), giving you greater flexibility. ShowSymbol animation can only display symbols that are already embedded in the application.

Q4: How does the OwningObject resolve references in a ShowGraphic popup?

The OwningObject only resolves relative references (e.g., me.TagName). Absolute references (e.g., ObjName.TagName) are completely unaffected by the OwningObject.

Q5: What happens when you call ShowGraphic() with an Identity that is already displayed?

The new ShowGraphic() call is silently blocked. No error or warning appears in the logger. Only one popup with a given Identity can be displayed at a time.
