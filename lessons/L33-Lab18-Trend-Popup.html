Lesson 33: Lab 18 – Trend Popup Symbol

Module 6 — Trend Visualization | Pages 423–430

When operators need to investigate historical data, they typically want a flexible tool where they can pick tags on the fly, choose time ranges, and see results in a table or chart. In the previous lessons you worked with InTouch's built-in Trend object, which is great for fixed, predefined trends. But what if you need a more interactive, ad-hoc query experience directly inside your HMI? That is exactly what the Historian Client .NET controls provide. In this lesson, you will learn what these controls do, and then build a reusable Trend Popup symbol that embeds the Historian Client Trend for drag-and-drop tag exploration at runtime.

33.1 The Historian Client .NET Controls

AVEVA ships a set of .NET controls that can be embedded in InTouch windows (or in any .NET container like Visual Basic .NET or a web browser). Each control handles a specific piece of the historical-data workflow: picking tags, choosing time ranges, running queries, and displaying results. Understanding each control will make it easier to assemble them into a complete trend popup.

aaHistClientQuery

The aaHistClientQuery control runs the Historian Client Query program — or a functional subset of it — right inside InTouch. Query retrieves data from a Historian database (or any Microsoft SQL Server database) and returns the results in a table format. If you are querying the Historian database, you can choose from several predefined query types and select options through a user interface, eliminating the need to know SQL syntax. For more advanced use cases, you can also write custom SQL queries as long as you know the SQL syntax and the database schema.

The aaHistClientQuery control provides a full query interface with tag selection, time range, and a results grid — all without writing SQL.

aaHistClientTagPicker

The aaHistClientTagPicker control displays the hierarchy of objects in the Historian database — tags, InTouch nodes, events, and so on — in a tree-style browser. Unlike the other controls, it does not include a built-in user interface for connecting to a Historian server. You need to add a programmatic connection. For example, to connect to the local Historian, you would call:

aaTagPicker1.Servers.Add("localhost");

The Tag Picker presents a hierarchical view of all available tags, with filter tabs for Analog, Discrete, String, and Event types.

aaHistClientTimeRangePicker

The aaHistClientTimeRangePicker control lets users select date and time parameters for a query. It offers a calendar popup for picking start and end dates, plus preset duration shortcuts (1 minute through 4 hours) so operators can quickly zoom into a recent window without manually calculating timestamps.

The Time Range Picker offers both a calendar interface and preset durations for quick time window selection.
Preset duration options let operators jump to common time windows with a single click.

aaHistClientActiveDataGrid

The aaHistClientActiveDataGrid control can run any SQL query against a Historian or SQL Server database and display the results in a grid. It does not support data-definition or data-manipulation queries (INSERT, UPDATE, DELETE) — only SELECT queries. Like the Tag Picker, it requires programmatic setup. Here is an example that connects to the local Historian and queries historical data for the SysTimeSec tag:

aaHistClientActiveDataGrid1.ServerName = "localhost";
aaHistClientActiveDataGrid1.UserName = "aaUser";
aaHistClientActiveDataGrid1.Password = "pwUser";
aaHistClientActiveDataGrid1.DatabaseName = "Runtime";
aaHistClientActiveDataGrid1.SQLString = "SELECT * FROM History WHERE Tagname = 'SysTimeSec'";
aaHistClientActiveDataGrid1.Connected = 1;

The Active DataGrid displays query results with DateTime, TagName, Value, and validated-Value columns.

aaHistClientSingleValueEntry

The aaHistClientSingleValueEntry control provides a simple interface for manually adding a single tag value to the Historian database. This is useful for recording operator-entered observations or manual data points that are not connected to a PLC.

The Single Value Entry control lets operators manually insert data points into the Historian database.
Key Concept: The Historian Client controls are .NET components. They must be embedded in InTouch graphics using the Toolbox search. Unlike native InTouch objects, they often require script-based initialization (calling methods like AddServerEx()) rather than simple property binding.

33.2 Building the TrendHistDisplay Symbol

In this lab, you will create a symbol called TrendHistDisplay that embeds the aaTrendControl. The resulting symbol will let operators pick any tag from the Historian at runtime, drag it onto a chart, and compare mixer levels with alarm highlights. You will also wire up navigation so that this trend can be opened from the main navigation bar.

Lab Objectives: Upon completion, you will be able to embed the Historian Client Trend .NET control in a symbol, use the AddServerEx() script function to initialize the trend server connection, and use the Historian Client Trend Tag Picker to add tags to the trend at runtime.

Step 1: Create the Symbol and Place the Trend Control

Open the IDE and navigate to the Graphics tab. In the Training toolset, create a new symbol named TrendHistDisplay and open it for editing. Switch to the Toolbox tab, search for aatrend, and drag the aaTrendControl onto the canvas. Name the graphic element Trend.

After dragging the aaTrendControl to the canvas, you will see the empty trend with its Tag Picker panel and time range controls ready for configuration.

Step 2: Add a Custom Property and Initialization Script

Right-click the canvas and open Custom Properties. Add a new custom property called HistServer with the following settings:

SettingValue
Data TypeString
Default ValueYour Historian node name (e.g., S01ENG)
VisibilityPrivate

The Private visibility means this property is hidden when the symbol is embedded in another graphic — a good practice for internal configuration values.

Next, right-click the canvas and open Scripts. In the Predefined Scripts section, make sure the Trigger type is set to On Show. Enter the following script body:

Trend.AddServerEx(HistServer, "", "", false);

Key Concept: AddServerEx() initializes the Historian connection at runtime. The On Show trigger fires when the symbol becomes visible, ensuring the server is connected every time the trend window opens. The false parameter specifies that integrated (Windows) security is used for the connection.
Tip: Your instructor will provide the correct Historian node name for your environment. Do not hard-code the server name in the script — using the custom property makes the symbol portable across different installations.

Step 3: Add Navigation

Save, close, and check in the TrendHistDisplay symbol. Then open the SystemNavigation symbol for editing. Duplicate the Historical Alarms button and place the copy to the right. Use Substitute Strings to change the button text to "Historian Client Trend".

Open the custom properties of the new button and change the GraphicName default value to TrendHistDisplay. Save, close, and check in the graphic.

The navigation bar now includes a Historian Client Trend button alongside the existing navigation options.

Step 4: Test in Runtime

Click Runtime in WindowMaker and navigate to the Historian Client Trend. After a moment the trend appears. Open the Tags panel, select Level_001.PV, and drag it onto the chart. The chart displays the mixer level with alarm limits highlighted — orange for HiHi and LoLo, yellow for Hi and Lo.

A single mixer level trend showing cyclic fill-and-drain behavior with alarm limit highlights.

Now add more tags. In the Tags panel, select Level_002.PV through Level_008.PV (hold Ctrl or Shift for multi-select) and drag them to the chart. The overlapping colored lines show how each mixer fills and drains at slightly different times — you can see the latency between them.

Multiple mixer levels displayed as overlapping trend lines reveal the time offsets between different mixers.

Use the time range dropdown to select 1 minute, then click the Enable/Disable live mode button. The chart begins scrolling continuously, showing real-time data as it arrives.

The preset time range menu lets you quickly zoom into different historical windows.
In live mode, the chart scrolls continuously, showing real-time mixer levels as they change.
Watch Out: The Historian Client Trend is a .NET control. If the Historian server is not running or the node name in the HistServer property is incorrect, the Tag Picker will show an empty server list and no tags will be available. Always verify connectivity before troubleshooting the trend itself.

Quiz — Test Your Understanding

Q1: What script function initializes the Historian connection when the TrendHistDisplay symbol is shown at runtime?

The Trend.AddServerEx(HistServer, "", "", false) function is called in an On Show script. It takes the server name from the custom property and uses integrated (Windows) security.

Q2: Why is the HistServer custom property set to Private visibility instead of Public?

Private visibility hides the property when the symbol is embedded in another graphic. Since HistServer is an internal configuration value (the Historian node name), operators do not need to see or modify it — only developers need access during design time.

Q3: Which Historian Client control would you use to manually record an operator's field observation into the Historian database?

The aaHistClientSingleValueEntry control. It provides a simple interface for manually inserting a single tag value into the Historian, which is ideal for operator-entered observations that are not connected to a PLC.

Q4: In the TrendHistDisplay, what do the orange and yellow alarm highlights on the trend chart represent?

Orange represents the extreme alarm limits (HiHi and LoLo), while yellow represents the warning alarm limits (Hi and Lo). These highlights let operators instantly see when a value is approaching or exceeding alarm thresholds.
