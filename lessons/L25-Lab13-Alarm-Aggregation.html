Lesson 25: Lab 13 — Alarm Aggregation Overview

Module 5 — Alarms and Events Visualization | Pages 321–340

Building a Hierarchical Alarm Overview

In the previous lesson, you learned how individual alarms appear on graphic elements through Alarm Border animations. But when a production facility has dozens or even hundreds of automation objects, an operator cannot rely solely on visual borders to understand the overall alarm situation. What is needed is a higher-level view — a single screen that summarizes alarm status across the entire production hierarchy, from the Plant level down through Lines and into individual Mixers.

This is exactly what the Alarm Aggregation Overview graphic provides. It uses the SA_AlarmSeverity symbol from the Situational Awareness Library to display severity counts, active and unacknowledged counts, and the current alarm mode at each level of the hierarchy. By embedding these widgets at every node — Production, Line1, Line2, and each Mixer — you create a living dashboard that rolls up alarm information from the bottom of the hierarchy to the top.

Key Concept: Alarm Aggregation Roll-Up
Alarm aggregation works by automatically rolling up alarm counts from child objects to parent objects. If Mixer100 has 12 Critical alarms and Mixer200 has another 12 Critical alarms, Line1 will show 24 Critical alarms. This hierarchical aggregation continues up to the Production level, giving operators an instant summary of where problems are concentrated. The roll-up happens automatically at runtime based on the Galaxy containment hierarchy — no scripting is required.

Adding SA_AlarmSeverity to Templates

The first step is to add alarm severity symbols to the $gArea template, so that every area object in your Galaxy automatically includes alarm visualization. This is a hallmark of template-based design: you configure the alarm display once in the template, and every instance inherits it. Open the $gArea template and add a Content symbol named AlarmSeverityLabels. From the Toolbox, search for "severity" and drag the SA_AlarmSeverity symbol to the canvas. Set the Label property to False for a compact display that shows only the severity counts without a text label.

Now duplicate this symbol to create AlarmSeverityPanel and reconfigure it with a richer set of options. This second widget serves a different purpose: it provides a full panel display with the object tagname, a background, and an alarm border — making it suitable for overview graphics where the operator needs to see not just the counts but also which object they belong to. Set the Wizard Options as follows:

PropertyValue
TypeAlarmPanel
LabelTrue
LabelTypeObjectTagname
BackgroundTrue
AlarmBorderTrue

Copy this graphic to the clipboard — you will reuse it in the $Mixer template. Add a similar AlarmSeverityPanel to the $Mixer template as well, but set IsAreaObject to False since $Mixer is not an area object in the Galaxy hierarchy. This distinction matters because it tells the widget to look for alarm data on the object's own attributes rather than aggregating alarms from child areas and objects.

The SA_AlarmSeverity widget configured as an AlarmPanel with full severity display, mode controls, and active/unacknowledged counts.

Creating the Alarm Aggregation Overview Graphic

With the templates configured, the next step is to create the overview graphic itself. Create a new symbol called AlarmAggregationOverview and embed the AlarmSeverityPanel instances for Production, Line1, Line2, and Mixer100 through Mixer400. Arrange them in a hierarchical tree layout — Production at the top, Lines in the middle, and Mixers at the bottom — and use the Connector tool to draw lines showing parent-child relationships. These connector lines are purely visual; the actual alarm roll-up is determined by the Galaxy containment hierarchy.

The complete alarm aggregation hierarchy — connector lines visually show the relationship between Production, Lines, and Mixers.

Adding Navigation and Runtime Controls

An overview graphic is only useful if operators can access it easily. To make the Alarm Aggregation Overview accessible from anywhere in the application, duplicate the NavigationButton symbol to create NavigationButton2. Remove the Element Style and Disable animations, leaving only the Action Scripts that control window navigation. Set the WindowModal and WindowResizable custom properties to True so the overview opens as a resizable popup window, and configure GraphicName to AlarmAggregationOverview. Leave WindowName blank — this ensures the graphic opens in a new popup window rather than replacing the current window.

You can enhance the Line 1 and Line 2 navigation buttons even further by embedding the AlarmSeverityLabels symbols directly into them. This puts live alarm counts right on the navigation buttons, so operators can see at a glance whether a line has active alarms without even opening the overview.

At runtime, alarm severity counts appear directly on the Line 1 and Line 2 navigation buttons.

Runtime Testing and Alarm Mode Control

At runtime, clicking the Alarm Overview button opens the aggregation popup as a modal window — you cannot interact with the background until you close it. This modal behavior is intentional: it forces the operator to focus on the alarm overview before returning to other tasks. The Production panel includes a Command Panel dropdown that lets you change the alarm mode for the entire area, with four options:

  • Enable — Restores normal alarm operation for the area and all its children.
  • Disable — Turns off all alarms in the area and every object it contains. This is useful during major maintenance windows.
  • Silence — Suppresses audible alarms while keeping visual alarms active. Useful when an alarm condition is known and expected.
  • Inhibit — Prevents new alarms from triggering while allowing existing alarms to continue aging. This is useful when a sensor is being calibrated and would otherwise generate spurious alarms.
Tip: Disabling alarms at the Production level cascades to all child objects — Lines and Mixers alike. This is a powerful way to silence an entire plant section during maintenance, but it carries a significant responsibility: always remember to re-enable alarms when maintenance is complete. The built-in shelving timeout mechanism (covered in Lesson 24) can help prevent this scenario for shorter maintenance windows.
When Production alarms are disabled, all child nodes automatically show Mode: Disable — the change cascades down the hierarchy.
Pitfall: Changing alarm mode at a parent level affects all descendants without exception. Always verify you are targeting the correct level before making changes. Disabling alarms at the Plant level will silence the entire facility — including critical safety alarms. In production environments, alarm mode changes are typically governed by role-based permissions and logged for audit purposes.

Quiz — Test Your Understanding

Q1: What is the difference between AlarmSeverityLabels and AlarmSeverityPanel in the SA_AlarmSeverity widget?

AlarmSeverityLabels (Type: AlarmSeverityIndicator) shows a compact display of severity counts with Mode selector but no background or label. AlarmSeverityPanel (Type: AlarmPanel) adds a background, object tagname label, and alarm border — providing a richer, more contextual display suitable for overview graphics.

Q2: Why must IsAreaObject be set to False for the $Mixer template?

The $Mixer automation object is not an area object in the Galaxy hierarchy. Setting IsAreaObject to False tells the widget to look for alarm data on the object's own attributes rather than aggregating alarms from child areas and objects. Using the wrong setting would cause the widget to display incorrect or no alarm data.

Q3: What happens when you disable alarms at the Production level in the aggregation overview?

All alarm panels in the hierarchy (Line1, Line2, Mixer100-400) automatically switch to Mode: Disable. Disabling alarms at a parent level cascades to all descendant objects. The alarm counts reset to zero and no new alarms will trigger until alarms are re-enabled.

Q4: How do alarm severity counts roll up through the hierarchy?

Each parent node sums the alarm counts from all its children. For example, if Line1 has 12 Critical alarms and Line2 has 12 Critical alarms, Production shows 24 Critical. This aggregation happens automatically at runtime based on the Galaxy containment hierarchy.
