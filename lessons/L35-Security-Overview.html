Lesson 35: Security Overview

Source: AVEVA InTouch for System Platform 2023 Training Manual, Module 7, Section 1 (pages 439–472)

Comprehensive overview of the AVEVA security model — authentication modes, user roles and permissions, object-level security classifications, the security audit trail, and InTouch security script functions.

35.1 Introduction to Application Security

Reference: Module 7, Section 1 (pages 441–442)

Security in AVEVA's System Platform is a global, relationship-based system that governs every object in the Galaxy database. Rather than relying on simple file-level permissions, the security model defines explicit relationships between users, roles, and the objects and functions they are permitted to interact with. This gives system administrators fine-grained control over who can perform which operations — from routine day-to-day tasks to critical configuration changes.

The security system is designed so that administrators can easily define users, assign them to roles, and grant those roles specific operational permissions on particular objects. Permissions are defined in terms of what operations a user can perform on Automation objects, and these permissions flow through a hierarchy of security roles, security groups, and access levels.

Key Concept: The InTouch security model is deeply integrated with Application Server. When you configure security for a node, InTouch uses methods and dialog boxes from Application Server for log-on and log-off operations. Users are configured in the System Platform IDE, not directly in InTouch.

Security includes two advanced write-protection mechanisms that specifically affect InTouch:

  • Secured Write: When writing data to a Galaxy attribute from InTouch, the operator must re-enter their password to complete the write. This adds an authentication confirmation step for routine operations.
  • Verified Write: Writing data requires authentication from two different users — the operator who initiates the write, and a second user who verifies it. This implements a "four-eyes" principle for critical changes.

35.2 Configuring Security for the Galaxy

Reference: Module 7, Section 1 (pages 441–442)

Before you can open the security editor for a Galaxy, several conditions must be met. First, no other user should be connected to the Galaxy — if others are connected, they will be denied access while security is being configured. Second, all objects in the Galaxy must be checked in. You can use the Find button on the Home ribbon in the IDE to search for checked-out objects (enable "Only checked-out objects" and search in both templates and instances), then right-click and select check-in. Third, your user profile must have the configuration permissions required to change Framework Configuration/Modify Security Model if security was previously configured.

Warning: Do not configure security settings in the IDE while a Managed InTouch application is open for editing in WindowMaker. Also, only change the System Administrator username or password using the Change Network Account Utility — the administrator account information is cached, and engines may need to be redeployed after the account is changed.
The IDE Configure panel — access the Security settings from Galaxy > Configure > Security (page 448)

To open the Security editor, go to Galaxy menu > Configure > Security in the IDE. This is where you will set the authentication mode, define roles, create users, and assign passwords — all of which we will walk through in Lab 20.

35.3 Authentication Modes

Reference: Module 7, Section 1 (page 442)

The authentication mode determines how user credentials are validated when someone logs on to the system. AVEVA offers four authentication options, each suited to different deployment scenarios:

ModeHow It Works
GalaxyUses the local Galaxy configuration to authenticate users. All security is contained at the Galaxy level. When a user logs on, their credentials are checked against the Galaxy's own user database and access is granted accordingly. This is the simplest option and ideal for standalone installations.
OS User basedLeverages the operating system's user authentication on an individual user level. Security credentials are checked against the OS user accounts. Useful when you want a single sign-on experience tied to Windows login.
OS Group basedUses OS authentication at a group level. OS groups are mapped to security roles in the Galaxy. When a user logs on, their OS group membership is verified and mapped to Galaxy roles. Each node in the Galaxy must have the same OS Users, Groups, and user-group mappings for consistent access.
Authentication providers (AIM)Uses AVEVA Identity Manager (AIM) to create a unified security infrastructure across local nodes and Azure VMs, leveraging Azure Active Directory (AD) as an identity provider.
The Authentication mode tab — select Galaxy, OS User, OS Group, or Authentication providers (page 449)
Key Concept: The Galaxy authentication mode is the most commonly used for training and standalone deployments. It keeps all user credentials within the Galaxy database itself, making it self-contained and easy to manage.

35.4 General and Operational Permissions

Reference: Module 7, Section 1 (pages 443–444)

The security model organizes permissions into two categories. General permissions relate to application configuration and administration tasks — things like starting the IDE, importing/exporting, and general configuration. Operational permissions relate to runtime operations on objects, such as modifying attributes or acknowledging alarms.

Two roles are defined by default in every Galaxy:

  • Administrator — has all permissions and cannot have its General permissions modified
  • Default — all users belong to this role automatically; this relationship cannot be changed
Best Practice: Since every user automatically belongs to the Default role, you should remove all permissions from the Default role and instead grant permissions through other, more specific roles. This prevents unauthorized users from gaining excessive access simply by being in the Galaxy.

Operational Permissions

The following operational permissions can be associated with a role:

PermissionDescription
Can Modify "Operate" AttributesAllows day-to-day tasks like changing setpoints, outputs, and control modes for PID objects, or commanding Discrete Device objects.
Can Modify "Tune" AttributesAllows tuning attributes in the runtime environment, such as adjusting alarm setpoints and PID sensitivity.
Can Modify "Configure" AttributesAllows configuration changes that require the object to first be placed Off scan. This is for significant changes like modifying PLC register definitions.
Can Acknowledge AlarmsAllows manual acknowledgment of alarms in the runtime environment.
Can Verify WritesAllows providing an authentication signature for Verified Write attributes. Only users with this permission can verify a task performed by operators with Can Modify "Operate" Attributes.
Can Shelve AlarmsAllows manually shelving and unshelving alarms.
Can Modify Alarm ModesAllows modifying the mode of an alarm (e.g., enabling or disabling alarm groups).
Can Modify Plant StatesAllows modifying plant states for state-based alarming.
The Roles tab showing Administrator with full permissions — note the General and Operational permissions trees (page 449)
Tip: For Galaxies migrated from earlier versions to Application Server, the "Can Modify Operate Attributes" setting is copied to "Can Verify Writes." In new Application Server Galaxies, Can Verify Writes is disabled by default.

35.5 Setting Object Security Classifications

Reference: Module 7, Section 1 (page 444)

Operators interact with automation objects through individual attributes. Each attribute that can be modified at runtime can have an associated security classification — a setting that controls who may write to it and under what conditions. Clicking the security control (shield icon) next to an attribute lets you select from one of seven possible security states:

ClassificationDescription
FreeAccessNo restrictions — anyone can write to this attribute without logging in. Used for safety-critical tasks like halting a failing process where a login request could cause dangerous delays.
OperateRequires Operate permissions for the object's security group. Used for normal tasks like writing to Setpoint or Command attributes.
SecuredWriteRequires username/password authentication each time the attribute is written to, plus Operate permissions. Adds an explicit confirmation step.
VerifiedWriteRequires two users: the operator (with Operate permissions) and a verifier (with Can Verify Writes permission). Implements a four-eyes principle.
TuneRequires Tune operational permissions. Used for tuning alarm setpoints and PID sensitivity at runtime.
ConfigureRequires Configure operational permissions. The object must first be placed Off scan before writing. Used for significant configuration changes.
ViewOnlyRead-only access — users can view the attribute but can never write to it at runtime, regardless of permissions.
Tip: If an attribute's security is shown in gray, its classification is locked in the parent object and cannot be changed, or it requires enabling a group attribute first.

35.6 Group Locking and Security

Reference: Module 7, Section 1 (page 445)

The lock and security controls associated with option groups allow you to set conditions for all options in the group at once. The group control typically reflects the uniform setting for every option in the group. However, if even one option differs from the others, the group control shows an indeterminate icon — alerting you that the group has mixed settings.

Key Concept: Group controls behave identically to individual option controls. If you change the security classification at the group level, it propagates to all options within the group — unless an individual option has been explicitly overridden.

35.7 Configuring Security for Symbols

Reference: Module 7, Section 1 (page 445)

Security extends beyond runtime operations to protect design-time activities as well. You can set permissions so that users cannot:

  • Import or export symbols
  • Create, modify, or delete symbols on the Graphics tab
  • Create, modify, or delete symbols in any AutomationObject template or instance
  • Create, modify, or delete View Applications (such as InTouch View Applications)
  • Deploy or undeploy View Applications
  • Edit the configuration of the Quality and Status Display

35.9 Security Script Functions

Reference: Module 7, Section 1 (page 446)

InTouch provides several script functions for working with security programmatically. These are essential for building custom logon displays, handling credentials, and managing user sessions from within your application scripts.

FunctionDescription
Logoff()Logs off the currently logged on user and sets the current user status to the default "None" operator. Available in both scripts and scripting.
AttemptInvisibleLogon()Logs on a user with supplied credentials without requiring manual entry of a password or user ID. Returns True on success (updating $OperatorName, $AccessLevel, and $Operator) or False on failure.
ChangePassword()Displays the Change Password dialog box, allowing the currently logged on operator to change their password.
InvisibleVerifyCredentials()Verifies a user's credentials without logging them on. Returns the associated access level as an integer, or -1 if invalid. Does not change the currently logged on user.
LogonCurrentUser()Logs on using the Windows account currently logged in to the operating system. Use only with OS-based security. Returns -1 on failure.
PostLogonDialog()Displays the InTouch Logon dialog box and returns True. Available in both scripts and scripting.
Key Concept: The PostLogonDialog() and Logoff() functions are the most commonly used in building custom logon panels. They handle the standard login/logout dialogs while giving you control over when and how they appear in your application.

Knowledge Check

1. What are the four authentication modes available in AVEVA System Platform security, and which is the simplest for standalone deployments?

Click to reveal answer

The four modes are Galaxy, OS User based, OS Group based, and Authentication providers (AIM). Galaxy is the simplest — it stores all user credentials within the Galaxy database itself, making it self-contained and ideal for standalone installations.

2. Why is it a best practice to remove all permissions from the Default role?

Click to reveal answer

All users automatically belong to the Default role, and this relationship cannot be changed. If the Default role has permissions, every user inherits them. Removing permissions from the Default role and granting them only through specific roles ensures that users only receive the access they explicitly need.

3. What is the difference between Secured Write and Verified Write security classifications?

Click to reveal answer

Secured Write requires a single user to authenticate with their username and password each time they write to an attribute. Verified Write requires two different users: the primary operator (with Operate permissions) and a verifier (with Can Verify Writes permission) — implementing a four-eyes principle for critical changes.

4. In the lab, what does the expression InTouch:$AccessLevel >= 1000 achieve when used with a Visibility animation?

Click to reveal answer

The expression makes the element visible only when the logged-in user has an access level of 1000 or higher. Since the Default role has access level 0 and the Training role has access level 1000, this ensures that elements like the Select Mixer dropdown are only visible to users with the Training role or higher — not to unauthenticated users.

5. Which InTouch script function displays the standard logon dialog box, and which function logs off the current user?

Click to reveal answer

PostLogonDialog() displays the InTouch Logon dialog box and returns True. Logoff() logs off the currently logged on user and sets the current user status to the default "None" operator.
