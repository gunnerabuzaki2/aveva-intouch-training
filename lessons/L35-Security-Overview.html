Lesson 35: Security Overview

Module 7 — Security | Pages 439–472

Every industrial control system has a fundamental tension: operators need quick, unimpeded access to process controls during normal operations, but unauthorized changes can have catastrophic safety and financial consequences. AVEVA's security system resolves this tension by tying every action in the Galaxy to a user's identity and permissions. In this lesson, you will learn how authentication modes work, how roles and permissions control what operators can do, and how to use InTouch's security tags and script functions to build a secure but usable HMI application.

35.1 How Galaxy Security Works

The security system is a global function that applies to every object in the Galaxy database. It is relationship-based: each user is mapped to roles, and each role determines what operations that user can perform on specific objects. Think of it as a chain: User → Role → Permissions → Object Actions.

Key Concept: Security is configured at the Galaxy level, not per-application. When you enable security, every object in the Galaxy inherits the security model. Users are defined in the System Platform IDE, and InTouch HMI uses the Application Server's logon/logoff methods. This means a single set of users and permissions governs both the automation engine and the visualization layer.

Prerequisites for Configuring Security

Before opening the security editor, you must ensure three conditions:

  1. No other user is connected to the Galaxy.
  2. All objects are checked in — the security editor will refuse to open if any object is checked out.
  3. Your user profile has configuration permissions — specifically the "Framework Configuration/Modify Security Model" permission.
Warning: Do not configure security settings while a Managed InTouch application is open for editing in WindowMaker. Other users who try to open the Galaxy during security configuration will be denied access. Also, only change the System Administrator username or password using the Change Network Account Utility — the administrator account is cached, and engines may need to be redeployed after the change.

35.2 Authentication Modes

System Platform supports four authentication modes. Each one determines how user credentials are verified when someone logs on to the Galaxy:

ModeHow It WorksBest For
GalaxyUses the Galaxy's own user database. All security is contained within the Galaxy.Standalone systems, training environments, small deployments.
OS User basedUses the Windows operating system's user authentication on an individual user basis.Facilities that already manage user accounts centrally via Active Directory.
OS Group basedUses OS groups mapped to Galaxy security roles. Users inherit permissions through group membership.Large organizations with existing group-based access policies.
Authentication ProvidersUses AVEVA Identity Manager (AIM) to create a unified security infrastructure across local nodes and Azure VMs, leveraging Azure Active Directory.Cloud-hybrid deployments, enterprises using Azure AD.
Key Concept: Galaxy authentication mode determines the entire security architecture. Once you select a mode and enable security, you cannot switch modes without disabling security first. Choose carefully based on your organization's existing identity infrastructure.

35.3 Roles and Permissions

Security in System Platform is organized around roles. Two roles exist by default:

  • Administrator (access level 9999) — has all permissions, and its General permissions cannot be modified.
  • Default (access level 0) — every user belongs to this role automatically. This relationship cannot be removed. It is strongly recommended that you remove all permissions from the Default role so that unauthenticated users have no access.

You create additional roles (such as "Training" or "Engineer") and assign access levels to each. Higher access levels grant broader permissions.

The Roles tab shows Administrator (9999) and Default (0) roles with their respective permission trees.

General vs. Operational Permissions

Permissions are split into two categories:

General permissions relate to application configuration and administration — things like modifying the Galaxy, deploying objects, and managing security itself. These control what a user can do in the IDE.

Operational permissions relate to runtime actions on objects. These are the permissions that matter most for operators at the HMI. The available operational permissions are:

PermissionWhat It Allows
Can Modify "Operate" AttributesNormal day-to-day tasks: changing setpoints, output values, control modes (e.g., PID setpoint, Discrete Device commands).
Can Modify "Tune" AttributesAdjusting alarm setpoints, PID sensitivity, and other tuning parameters.
Can Modify "Configure" AttributesSignificant configuration changes that require the object to be Off-scan first (e.g., changing a PLC register that defines a device input).
Can Acknowledge AlarmsManually acknowledging alarms in the runtime environment.
Can Verify WritesProviding an authentication signature for attributes configured with Verified Writes security — a two-person authorization mechanism.
Can Shelve AlarmsTemporarily suppressing (shelving) and unsuppressing alarms.
Can Modify Alarm ModesChanging the mode of an alarm (e.g., disabling or enabling it).
Can Modify Plant StatesModifying plant states used for state-based alarming.
Tip: The recommended security design pattern is to remove all operational permissions from the Default role and assign them only to purpose-built roles. This ensures that every user must authenticate before performing any action.

35.4 Object-Level Security Classifications

Each attribute on an object can have its own security classification. When an operator interacts with an object at runtime, the system checks whether the operator's role grants permission for that specific classification. The possible classifications are:

ClassificationBehavior
FreeAccessAnyone can write to this attribute without restriction, even without defined permissions. Use this for safety-critical or time-critical tasks (e.g., emergency stop) where a logon prompt could cause dangerous delays.
OperateRequires "Operate" permissions for the security group. Used for normal tasks like changing setpoints or commanding devices.
TuneRequires "Tune" permissions. Used for adjusting alarm setpoints and PID tuning parameters.
ConfigureRequires "Configure" permissions. The object must be taken Off-scan before these attributes can be written. Used for significant configuration changes.
Warning: Using FreeAccess on process-critical attributes defeats the purpose of security. Reserve FreeAccess strictly for emergency functions where requiring a logon would create a safety hazard. Every other attribute should be classified at Operate, Tune, or Configure.

Group Locking

When attributes are organized into option groups (like a set of radio buttons or checkboxes), the lock and security controls on the group apply to all options within it. If at least one option has a different security setting than the others, the group control shows an indeterminate icon — a visual cue that the group's permissions are not uniform.

35.5 Securing Symbols and Graphics

Beyond object attributes, you can also set security permissions on the design-time operations for symbols. This prevents unauthorized users from importing, exporting, creating, modifying, or deleting symbols on the Graphics tab. You can also protect View Applications (like InTouch applications) from being deployed or undeployed.

At runtime, the Security Audit Trail records which users logged on, from which domain, and when. This audit capability is a primary function of any industrial security system — it provides the accountability trail that regulators and safety managers require.

35.6 Security System Tags and Script Functions

InTouch exposes security information through system tags that your graphics and scripts can reference:

TagnameTypeAccessPurpose
$AccessLevelIntegerRead OnlyThe numeric access level of the currently logged-on user (0 = no user).
$OperatorMessageRead OnlyThe username of the currently logged-on user (max 16 characters).
$OperatorNameMessageRead OnlyThe full name of the currently logged-on user (max 131 characters).
$OperatorEnteredMessageWrite OnlyWrite the username here to trigger a logon attempt.
$PasswordEnteredMessageWrite OnlyWrite the password here to trigger a logon attempt.
$ChangePasswordDiscreteRead/WriteSet to 1 to open the Change Password dialog.
$InactivityTimeoutDiscreteRead OnlyIndicates whether the inactivity timeout has triggered.
$InactivityWarningDiscreteRead OnlyIndicates whether the inactivity warning is active.
$VerifiedUserNameMessageRead OnlyThe username of the person who last verified a write operation.

InTouch also provides several script functions for security operations:

  • Logoff() — Logs off the current user and resets $AccessLevel to 0. Available in both scripting and the Action Scripts animation.
  • PostLogonDialog() — Displays the standard logon dialog. The operator enters credentials and the system validates them against the configured authentication mode.
  • AttemptInvisibleLogon(UserId, Domain) — Attempts to log on a user silently using supplied credentials. Returns True on success, False on failure. No dialog is shown. Useful for automated logon scenarios.
  • InvisibleVerifyCredentials(UserId, Password, Domain) — Verifies credentials without logging the user on. Returns the access level if valid, or -1 if invalid. Useful for confirming a user's identity before performing a sensitive operation.
  • LogonCurrentUser() — Logs on using the Windows account of the currently logged-on Windows user. Requires OS-based authentication.
  • ChangePassword() — Opens the Change Password dialog for the current user.
Key Concept: The $AccessLevel tag is the backbone of runtime security in InTouch. You bind Visibility and Disable animations to expressions like InTouch:$AccessLevel >= 1000 to hide or disable UI elements for users who lack sufficient permissions. A value of 0 means no user is logged in.

Configuring Security in the IDE

In the IDE, navigate to Galaxy > Configure > Security. On the Authentication Mode tab, select Galaxy. On the Roles tab, add a new role called "Training" with access level 1000. On the Users tab, add a user called "TUser" with full name "Training User", assign the Training role, and set a password.

The Security configuration section is accessed from Galaxy > Configure in the IDE.
Select Galaxy as the authentication mode for standalone or training environments.

Securing Graphics with Animations

To protect interactive elements, you add Visibility and Disable animations bound to InTouch:$AccessLevel. For example, the Select Mixer text gets a Visibility animation with the expression InTouch:$AccessLevel >= 1000 — it only appears when a user with at least access level 1000 is logged in. The MixerDropDown combo box gets a Disable animation with the same expression — it is disabled when no authorized user is present.

The Select Mixer text is only visible when a user with access level 1000 or higher is logged in.
The MixerDropDown is disabled when no authorized user is logged in (access level below 1000).

For equipment controls (valves, pumps, agitators), you add Disable animations with the expression InTouch:$AccessLevel == 0 and set "Disabled When" to True. This means the controls are disabled whenever no user is logged in, but become available once any authenticated user logs on.

The Valve command panel is disabled when $AccessLevel is 0 (no user logged in).

Building the SecurityDisplay Symbol

The SecurityDisplay symbol contains three text boxes showing the current user's name, full name, and access level — each bound to the corresponding system tag ($Operator, $OperatorName, $AccessLevel). It also has Logon and Logoff buttons with Action Scripts calling PostLogonDialog() and Logoff() respectively.

The Logon button calls PostLogonDialog() to display the standard authentication dialog.
The Logoff button calls Logoff() to clear the current user and reset $AccessLevel to 0.

The Logon button also has an Element Style animation that changes its appearance based on InTouch:$AccessLevel <> 0 — when a user is logged in, the button uses Intensity4 style (highlighted) to indicate an active session.

Testing at Runtime

At runtime, the SecurityDisplay shows User: None and Access Level: 0. The Select Mixer text is invisible, the drop-down is disabled, and equipment command panels are disabled. After clicking Logon and entering TUser / ww, the display updates to show the logged-in user with access level 1000. The Select Mixer text appears, the drop-down becomes active, and equipment controls are enabled. Logging in as the administrator (access level 9999) provides the same access. Clicking Logoff returns everything to the locked state.

Tip: Use the $AccessLevel tag value to create tiered security. For example, access level 1000 might allow setpoint changes, while access level 5000 is required for alarm configuration, and 9999 for system administration. This gives you fine-grained control over who can do what.

Quiz — Test Your Understanding

Q1: What is the value of $AccessLevel when no user is logged in, and what does this mean for secured UI elements?

When no user is logged in, $AccessLevel is 0. Any Visibility animation bound to an expression like InTouch:$AccessLevel >= 1000 will evaluate to False, hiding the element. Any Disable animation bound to InTouch:$AccessLevel == 0 will evaluate to True, disabling the element. This is the default locked state.

Q2: Why is it recommended to remove all permissions from the Default role?

Every user automatically belongs to the Default role and this relationship cannot be removed. If the Default role has permissions, then even unauthenticated users inherit those permissions. By removing all permissions from Default, you ensure that every user must authenticate and be assigned to a specific role before they can perform any action.

Q3: When would you use FreeAccess security classification instead of Operate?

FreeAccess should only be used for safety-critical or time-critical functions where requiring a logon prompt could cause dangerous delays — for example, an emergency stop button. For all other process operations, Operate (or higher) classification should be used so that only authorized users can make changes.

Q4: What is the difference between AttemptInvisibleLogon() and PostLogonDialog()?

PostLogonDialog() displays a visible logon dialog where the operator enters their username and password. AttemptInvisibleLogon() attempts to log on a user silently using supplied credentials as parameters — no dialog is shown. InvisibleLogon is useful for automated logon scenarios or when credentials are already known.

Q5: What are the four authentication modes available in System Platform, and which one would you choose for a cloud-hybrid deployment using Azure AD?

The four modes are Galaxy, OS User based, OS Group based, and Authentication Providers. For a cloud-hybrid deployment using Azure AD, you would use Authentication Providers, which leverages AVEVA Identity Manager (AIM) to create a unified security infrastructure across local nodes and Azure VMs using Azure Active Directory.
