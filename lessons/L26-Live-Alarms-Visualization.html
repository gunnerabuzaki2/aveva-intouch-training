The Alarm Client Control: From Tables to Interactive Grids

In the previous lessons, you saw how alarms can be visualized through border animations on individual symbols and through severity count widgets in an aggregation overview. But there is a third, equally important way to interact with alarms: the Alarm Client control. This is a powerful graphic element that replaces the legacy Alarm Viewer and Alarm DB View controls, providing a rich, interactive grid for displaying current and historical alarms and events directly within your InTouch graphics.

Unlike static displays, the Alarm Client supports full runtime configuration. Operators can refresh, acknowledge, filter, sort, and freeze alarm data without ever returning to development mode. This interactivity is what transforms a passive alarm list into an active tool for alarm management. The Alarm Client control is not merely a table — it is an interactive workspace where operators can drill into alarm details, apply ad-hoc filters, and acknowledge alarms with comments that become part of the audit trail.

Key Concept: Five Client Modes
The Alarm Client control supports five distinct client modes, grouped by their data source:
  • Current Alarms — Live alarms from the InTouch Database (WWALMDB) or ArchestrA Database (A2ALMDB). This is the most commonly used mode for day-to-day operations.
  • Recent Alarms and Events — Recently occurred alarms and events that have not yet been archived. Useful for reviewing what happened in the last few minutes or hours.
  • Historical Alarms — Archived alarms from Historian history blocks. Used for post-incident analysis and compliance reporting.
  • Historical Events — Archived events from Historian. Provides a timeline of system changes and state transitions.
  • Historical Alarms and Events — A combined historical view showing both alarms and events in a single chronological feed.

You can switch between modes at runtime using the ClientMode integer property in scripting. This means a single alarm display symbol can adapt to different operator needs throughout a shift — showing current alarms during normal operations, switching to recent alarms during an incident investigation, and pulling up historical alarms for a compliance review, all without leaving the same graphic.

Configuring the Alarm Client: Eight Pages of Control

To embed the Alarm Client in your graphics, open the Graphic Editor and select the Alarm Client tool from the Tools pane. Click the canvas to place the control, then double-click to access its configuration dialog. The configuration is organized into eight pages, each controlling a different aspect of the alarm display's behavior and appearance.

Alarm ModeSet client mode (Current, Recent, or Historical) and configure the alarm query path that determines which alarms the client retrieves.

Alarm Query Syntax: Telling the Client What to Show

The alarm query is the heart of the Alarm Client's configuration. It specifies which alarms the client should retrieve from the distributed alarm infrastructure. The basic syntax follows a hierarchical pattern that mirrors the Galaxy containment hierarchy:

\\nodename\Galaxy!area

This retrieves all alarms and events from the specified area and all its sub-areas and contained objects. You can narrow results further by specifying objects or using wildcards:

  • \\nodename\Galaxy!area!object — Alarms from a specific object only.
  • \\nodename\Galaxy!area!object.* — All alarms from a specific object (the wildcard matches all alarm attributes).
  • \\nodename\Galaxy!area!*.attribute — Alarms from all objects in the area for a specific attribute.

This query syntax gives you precise control over what the Alarm Client displays. In a large facility, you might configure different Alarm Client instances with different queries — one showing only Critical alarms across the entire plant, another focused on a specific production line, and a third showing alarms for a single piece of equipment.

Pitfall: When using time-based filter criteria (TimeLCT, TimeOAT, TimeLCTOAT) for historical alarm modes, ensure the TimeSelector.StartDate is set earlier than any time filtering requirement and TimeSelector.EndDate is set later. Otherwise, the Alarm Client may not return all alarm and event records. This is a common issue when debugging "missing alarms" in historical views.

Color Coding and Priority Ranges

Visual differentiation is essential when operators are scanning a long list of alarms. The Alarm Client supports priority-based color coding with configurable breakpoints. By default, alarms are grouped into four priority ranges (1–249, 250–499, 500–749, 750–999), and each range has distinct background colors for acknowledged and unacknowledged states. This means an unacknowledged Critical alarm stands out immediately from an acknowledged Low-priority alarm, even in a crowded grid.

Beyond priority-based coloring, you can also set the control background color, grid lines, and heading colors to match your HMI design standards. Consistent color coding across your entire HMI — from alarm borders on symbols to the Alarm Client grid — reinforces the operator's mental model and reduces the time needed to assess a situation.

Creating the AlarmLiveDisplay Symbol

Lab Objectives:
  • Create a symbol with the Alarm Client control configured for current alarms
  • Configure alarm queries and custom filters for different production areas
  • Deploy and interact with live alarms at runtime

Step 1: Configure the Alarm Client for Current Alarms

Create a new symbol named AlarmLiveDisplay and place the Alarm Client on the canvas. Open the configuration dialog and navigate to the Alarm Mode page. Set the Client Mode to Current Alarms and configure the Alarm Query to point at your Production area:

  • Client Mode: Current Alarms
  • Alarm Query: \\<NodeName>\Galaxy!Production

This tells the client to retrieve all active alarms from the Production area and everything it contains. The node name should match the Platform node where your automation objects are deployed.

Alarm Mode configuration showing Current Alarms mode and the alarm query path pointing to the Production area.

Step 2: Create Custom Filters

Navigate to the Queries and Filters page and add three custom filters using the "Add new filter" button. Each filter narrows the alarm display to a specific production area. For each filter, you will name it, select a filter criterion, set an operator, and enter a value:

  1. Name the first filter (e.g., "Line1"). Select Group from the criteria list and click the right arrow to add it. Set the Operator to = and enter the group name "Line1". Click Set to confirm, then OK.
  2. Repeat for "Line2" (Group = "Line2") and "Production" (Group = "Production").

These filters let operators quickly switch between viewing alarms for different production areas at runtime, without needing to modify the alarm query or return to development mode.

Three custom filters configured — operators can check/uncheck these at runtime to filter the alarm display.

Step 3: Deploy and Test at Runtime

Set the AlarmClient dimensions (Width: 1920, Height: 200), save the graphic, and embed AlarmLiveDisplay in the AlarmDisplay window. At runtime, you will see a live grid of current alarms with columns for State, Node, Group, Name, Type, Time, and more. Use the context menu to acknowledge alarms individually or in bulk, apply the custom filters you created, and sort the grid by any column. The grid updates in real time as alarms occur, return to normal, and are acknowledged.

Live alarm display at runtime showing current alarms with color-coded priority, state information, and timestamps.

Quiz — Test Your Understanding

Q1: What replaced the legacy Alarm Viewer and Alarm DB View controls?

The Alarm Client control replaces both legacy controls and extends alarm visualization to the graphics environment. It supports current, recent, and historical alarm modes with rich runtime configuration capabilities.

Q2: How do you switch the Alarm Client between Current Alarms and Historical Alarms at runtime?

Use the ClientMode integer property in scripting. The five modes are: Current Alarms, Recent Alarms and Events, Historical Alarms, Historical Events, and Historical Alarms and Events. This allows a single alarm display to adapt to different operator needs throughout a shift.

Q3: What filter criteria is required for current alarm queries?

Current alarms and recent alarms and events require at least a Provider and a Group as filter criteria, using the equals sign (=). For example, a filter might specify Group = "Line1" to show only alarms from that production line.

Q4: Can operators acknowledge alarms directly from the Alarm Client at runtime?

Yes. Operators can right-click to access the context menu and use Ack Selected, Ack Others, or Ack All options. An optional acknowledgment comment can be entered, creating an audit trail. The alarm then moves to the ACK state and may disappear from the Current Alarms view if the value has returned to normal.
