Lesson 6: Lab 2 – Creating a Managed InTouch Application

Source: AVEVA InTouch for System Platform 2023 Training Manual, Module 2, Lab 2 (pages 47–54)

In this lab, you will create a Managed InTouch application for your Galaxy using the $InTouchViewApp template. This application will be used throughout the rest of the labs in this course.

Lab Objective: Create a Managed InTouch application by deriving from the $InTouchViewApp base template, organizing it in a custom toolset, and initializing it with the correct settings.

6.1 Create a Derived InTouchViewApp Template

Reference: Module 2, Lab 2 (pages 48–49)

In the System Platform IDE, the $InTouchViewApp base template lives under the System toolset. Because InTouchViewApp instances cannot be created directly from the base template, you must first derive a project-specific template from it. This derived template will define the settings and characteristics of your InTouch application.

Follow these steps to create the derived template:

  1. In the IDE, on the Templates tab, expand TrainingGalaxy, and then expand the System template toolset.
  2. Right-click the $InTouchViewApp template and select New > Derived Template.
  3. Name the new derived template $MixerView.
The IDE Templates tab showing the System toolset with $InTouchViewApp selected. The right-click context menu offers New > Derived Template (page 48)

6.2 Organize Templates with a Subtoolset

Reference: Module 2, Lab 2 (pages 49)

To keep your project organized, create a dedicated subtoolset under Training for all InTouchViewApp templates. This makes it easy to find and manage your visualization templates as the project grows.

  1. On the Templates tab, right-click the Training toolset and select New Template Toolset.
  2. Name the new toolset ViewApps.
  3. Drag the $MixerView template into the Training\ViewApps toolset.
Right-click the Training folder and select New Template Toolset to create the ViewApps subtoolset (page 49)
After organizing: the $MixerView template now resides under Training > ViewApps (page 49)

6.3 Initialize the InTouchViewApp

Reference: Module 2, Lab 2 (pages 50–51)

Double-clicking the $MixerView template opens the InTouchViewApp initialization wizard. This wizard walks you through the options for creating a new InTouch application or importing an existing one.

  1. Double-click $MixerView to launch the initialization wizard.
  2. Leave the default selection as Create new InTouch application and click Next.
The InTouchViewApp initialization wizard — Step 1: Select "Create new InTouch application" and click Next (page 50)

The wizard continues with application-specific settings:

  1. Check the InTouchView application checkbox. This tells InTouch to rely on built-in Application Server features (I/O access, alarms, history, and security) rather than legacy InTouch HMI features.
  2. In the Description field, enter: InTouchView application for the InTouch for System Platform course
  3. Click Next.
Key Concept: The "InTouchView application" checkbox determines which product features are used. When checked, InTouch uses Application Server's built-in I/O, alarms, history, and security. When unchecked, it falls back on legacy InTouch HMI features (AccessNames, InTouch alarms, etc.).
Tip: The Custom resolution feature allows you to set a production resolution different from your development resolution. The default "Screen resolution" uses your current development resolution as the production target.
Configure the application type (InTouchView checked), enter a description, and click Next (page 51)

6.4 WindowMaker Opens

Reference: Module 2, Lab 2 (pages 52–53)

After a few moments, AVEVA InTouch HMI WindowMaker opens, displaying the backstage New window area. This is where you can create new windows for your application — but for now, you will exit the backstage view.

  1. In the top-left corner of the backstage view, click the left arrow button to move to the main WindowMaker view.
WindowMaker backstage view after initialization — click the back arrow to exit (page 52)
The Available templates section in the backstage view (page 52)

The main WindowMaker interface appears with all its panels — Windows, Scripts, Industrial Graphics, and Properties. Your Managed InTouch application is now created and ready for window development in the next lab.

The WindowMaker main interface — the Managed InTouch application is now created and ready for development (page 53)
Key Concept: Opening an InTouchViewApp object from the IDE automatically launches WindowMaker. There is no separate object editor — WindowMaker is the development environment for InTouch applications.

Knowledge Check

1. Why can't you create an InTouchViewApp instance directly from the $InTouchViewApp base template?

Click to reveal answer

InTouchViewApp instances cannot be created from the base template. You must first derive a template from $InTouchViewApp, then create instances from that derived template. This enforces a consistent template hierarchy.

2. What does the "InTouchView application" checkbox do in the initialization wizard?

Click to reveal answer

When checked, it tells InTouch to use Application Server's built-in features for I/O access, alarms, history, and security. When unchecked, it falls back on legacy InTouch HMI features like AccessNames, InTouch alarms, and InTouch security.

3. What is the purpose of creating a ViewApps subtoolset under the Training toolset?

Click to reveal answer

Creating a dedicated ViewApps subtoolset provides organizational structure for all InTouchViewApp templates, making them easy to find and manage as the project grows.

4. What happens when you double-click an InTouchViewApp template in the IDE?

Click to reveal answer

It launches the InTouchViewApp initialization wizard, which guides you through creating a new InTouch application or importing an existing one. There is no separate object editor — WindowMaker is the editor.
