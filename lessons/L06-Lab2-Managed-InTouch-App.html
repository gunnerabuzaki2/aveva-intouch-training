steps, you will create a Managed InTouch application. 1. In the IDE, on the Templates tab, expand TrainingGalaxy, and then the System template toolset. 2. Right-click the $InTouchViewApp template and select New | Derived Template. The new derived template appears in the System toolset. 3. Name the new derived template as $MixerView.

Screenshot of the IDE Templates tab showing the TrainingGalaxy project hierarchy with expanded System folder listing templates ($AppEngine, $Area, $InTouchViewApp, etc.), with a right-click context menu on $InTouchViewApp offering New Derived Template and other options

Next, you will create a subtoolset for organizing your InTouchViewApp. 4. On the Templates tab, right-click the Training toolset and select New Template Toolset. 5. Name the new toolset as ViewApps. The new ViewApps toolset appears at the bottom of the Templates tab. 6. Drag the $MixerView template to the Training\ViewApps toolset.

7. Double-click $MixerView. The first part of the InTouchViewApp initialization dialog box appears. 8. Leave the default as Create new InTouch application and click Next.

Screenshot of the InTouchViewApp initialization wizard dialog with Create new InTouch application selected, showing Galaxy overwrite settings with checked options for Industrial graphics, galaxy style library, script function library, client controls, and languages, plus a backup checkbox

continues. 9. Check the InTouchView application check box. Note: This check box reflects which InTouch product was licensed. In this InTouch for System Platform course, we check this box to have InTouch rely on built-in Application Server features, such as I/O access, alarms, history, and security. Leaving the box unchecked relies on InTouch HMI built-in features, including legacy InTouch AccessNames, InTouch alarms, InTouch historical logging, and InTouch security. 10. In Description, enter: InTouchView application for the InTouch for System Platform course Note: The Custom resolution feature allows the InTouch application designer to set the resolution that will be used on the production node, even when their development resolution may differ. The default Screen resolution takes the current development resolution and uses that as the target resolution on the production node. 11. Click Next.
