Acknowledging Alarms: The Operator's Daily Workflow

When operators encounter alarms in the Alarm Client grid, they need a systematic way to acknowledge them. Acknowledgment is not just a formality — it is a critical step in the alarm management lifecycle. An unacknowledged alarm continues to demand attention through visual and audible indicators, and the acknowledgment event is recorded in the audit trail with the operator's identity, timestamp, and optional comment. This creates accountability and provides valuable data for post-incident analysis.

The right-click context menu provides several acknowledgment options that form the backbone of proper alarm management. The workflow is straightforward: identify the alarm by its color and state in the grid, right-click to access the context menu, choose the appropriate acknowledgment action, and optionally enter a comment. The acknowledged alarm then either disappears (if the value has already returned to normal) or transitions to the ACK state, where it remains visible but no longer triggers attention-getting indicators.

Key Concept: Alarm Acknowledgment Workflow
The standard operator workflow for handling alarms follows this pattern:
  1. Identify the alarm in the grid — the color and state column indicate its urgency and whether it has been acknowledged.
  2. Right-click and select Ack Selected to acknowledge individual alarms, or use Ack Others > Ack All to acknowledge everything at once.
  3. Enter an acknowledgment comment (e.g., "ticket issued", "maintenance notified") — this creates an audit trail that is invaluable for compliance and incident review.
  4. The acknowledged alarm either disappears (if the value returned to normal) or changes to the ACK state in the grid.

Beyond acknowledgment, the context menu provides a rich set of additional options: Shelve (temporarily hide the alarm with a timeout), Hide (remove from the current view), Sort (reorder the grid by any column), Queries and Filters (apply named filters), Freeze (pause the live update so you can study a snapshot), Statistics (see alarm counts by severity and state), Requery (refresh the data from the alarm database), and Reset (clear all filters and restore the default view).

Using Custom Filters at Runtime

In Lesson 26, you configured custom filters (Line1, Line2, Production) in the Alarm Client's Queries and Filters page. Now it is time to use them at runtime. Right-click the AlarmLiveDisplay graphic and select Queries and Filters. The Query and Filter Favorites dialog appears, showing your pre-configured filters with checkboxes.

Simply check the filter you want — for example, "Line1" — and click OK. The Alarm Client immediately updates to show only alarms matching that filter. This is invaluable during an incident: when an operator receives a call about a problem on Line 2, they can instantly narrow the alarm display to Line 2's alarms, cutting through the noise of alarms from other areas. To see all alarms again, open the dialog and uncheck the filter, or select the Default query.

The Query and Filter Favorites dialog — check a filter to instantly narrow the alarm display to a specific area.
Tip: You can combine multiple filters or switch between them quickly during an incident. The Default query always shows all Production alarms — use it as your baseline view, then apply specific filters when drilling down into a problem area. During shift handover, resetting to the Default view ensures the incoming operator sees the complete picture.

Creating a Context-Aware Alarm Display

The most powerful alarm display is one that automatically shows alarms relevant to the specific object the operator is viewing. Imagine an operator looking at Mixer100's detail graphic — they should see only Mixer100's alarms, not alarms from every mixer in the plant. This is achieved through scripting that dynamically builds alarm queries based on the current context, eliminating the need for operators to manually filter.

Key Concept: Dynamic Alarm Queries with Scripting
By embedding an Alarm Client in a template symbol and using VBScript to build the alarm query at runtime, you can create alarm displays that automatically show only the alarms for the currently selected object. The script constructs the query by combining the platform node name, the Galaxy area, and the specific object's tagname — all resolved dynamically through relative references. This eliminates manual filtering and ensures the operator always sees the right alarms for the right context.

Lab 14 — Building a Mixer-Specific Alarm Display

Lab Objectives:
  • Use scripting to dynamically build alarm queries based on the selected mixer
  • Use the MyPlatform relative reference to access node information without hardcoding
  • Create a Show Symbol animation to display context-aware alarm popups

Step 1: Create the AlarmPopup Symbol

Open the $Mixer template and add a new Content symbol named AlarmPopup. Place an Alarm Client on the canvas (800×400 pixels). Now add a custom property named NodeName with the following settings:

PropertyValue
Data TypeString
Default ValueMyPlatform.NodeName (Expression mode)
VisibilityPrivate

The MyPlatform relative reference is the key to making this symbol work across all mixer instances. It automatically resolves to the WinPlatform hosting the mixer, giving you the correct node name at runtime without hardcoding a specific platform name. When Mixer100 runs on SO1ENG, MyPlatform.NodeName resolves to "SO1ENG". When Mixer200 runs on a different platform, it resolves to that platform's name instead.

Step 2: Write the Dynamic Query Script

Add a script named UpdateQuery with a DataChange trigger on the NodeName property. This script builds a multi-object alarm query by concatenating the tagnames of all alarmed attributes in the mixer:

'Build alarm query prefix
dim prefix as string
prefix = "\\" + NodeName + "\Galaxy!" + Me.Area + "!"

'Build alarm query
dim almQry as string
almQry = prefix + Me.Tagname + ".* "
almQry = almQry + prefix + Me.Agitator.Tagname + ".* "
almQry = almQry + prefix + Me.Inlet1.Tagname + ".* "
almQry = almQry + prefix + Me.Inlet2.Tagname + ".* "
almQry = almQry + prefix + Me.Outlet.Tagname + ".* "
almQry = almQry + prefix + Me.Temperature.Tagname + ".* "
almQry = almQry + prefix + Me.Level.Tagname + ".* "

'Update alarm client with alarm query
AlarmClient.AlarmQuery = almQry
The UpdateQuery script dynamically builds an alarm query by combining the node name, area, and all child object tagnames.

The script uses the Me. relative reference to access each child object's Tagname property. This is what makes the symbol reusable: when Mixer200 opens the alarm popup, Me.Tagname automatically resolves to "Mixer200" instead of "Mixer100". The wildcard .* after each tagname captures all alarm attributes for that object, so you do not need to list each attribute individually. This design principle — write once, deploy everywhere — is the foundation of template-based development in System Platform.

Step 3: Add the Alarms Button

In the MixingProcess_Detailed graphic, draw a button labeled "Alarms" and apply the Intensity2 ElementStyle for consistent visual appearance. Configure a Show Symbol animation with Reference set to Me.AlarmPopup. This binding makes the button open the alarm popup for whichever mixer instance the operator is currently viewing — the Me. reference ensures the correct instance is used.

The Show Symbol animation uses Me.AlarmPopup to open the alarm popup for whichever mixer instance the operator is viewing.

Step 4: Test Context-Aware Alarms at Runtime

At runtime, click the Alarms button on Mixer100. You should see only alarms related to Mixer100's components — Level, Temperature, Agitator, Inlets, and Outlet. Close the popup, switch to a different mixer using the Select Mixer dropdown, and click Alarms again. The alarm display now shows only that mixer's alarms. The script re-executed because the context changed, building a new query specific to the newly selected mixer. This is the power of dynamic alarm queries combined with relative references.

The context-aware alarm popup for Mixer100 — showing only alarms relevant to this specific mixer and its child objects.
Pitfall: The alarm query assumes the WinPlatform hosting the alarmed objects is configured as the alarm provider. If your platform is not set up as an alarm provider, the Alarm Client will display no results even though the script executes correctly. Always verify the alarm provider configuration on the Platform object before debugging "empty alarm grid" issues.

Historizing Alarms and Events for Long-Term Analysis

Real-time alarm displays are essential for day-to-day operations, but they only show you what is happening right now. To preserve alarm and event data for historical analysis, trend reporting, and compliance documentation, you need to configure Historian storage. This is a two-step process. First, enable storage at the AppEngine level by checking Enable storage to historian and specifying the Historian node. Then, in the Alarms and Events Configuration dialog, check the Historize box for each alarm severity level and event type you want to archive.

Alarm and event historization shares the same Historian infrastructure as process data, including Store and Forward support, Application Engine fail-over, and dual Historian capability. This means your alarm history benefits from the same redundancy mechanisms that protect your process data — ensuring that no alarm record is lost even during network interruptions or server failures.

Quiz — Test Your Understanding

Q1: What does the MyPlatform relative reference resolve to at runtime?

MyPlatform resolves to the WinPlatform object that hosts the current AutomationObject. In the context of a mixer, it gives you the platform node name hosting that mixer — essential for building correct alarm queries without hardcoding node names.

Q2: Why does the script use Me.Tagname instead of a hardcoded string like "Mixer100.Level"?

Using Me.Tagname makes the script reusable across all mixer instances. When Mixer200 opens the alarm popup, Me.Tagname automatically resolves to "Mixer200.Level" instead of "Mixer100.Level". This is the power of relative references — write once, works for every instance.

Q3: What is the wildcard pattern used to capture all alarm attributes for an object?

The pattern is "objectname.*" — the dot separates the tagname from the attribute wildcard. The asterisk matches all alarm attributes (Level, Temperature, Agitator, etc.) for that specific object, so you don't need to list each attribute individually.

Q4: How do you enable alarm historization in System Platform?

Two steps: (1) On the AppEngine, check "Enable storage to historian" and specify the Historian node. (2) In the Alarms and Events Configuration dialog (Galaxy > Configure > Alarms and Events Configuration), check the "Historize" box for each alarm severity level and event type you want to archive.

Q5: What happens when you click the Alarms button on a different mixer?

The Show Symbol animation opens Me.AlarmPopup for that specific mixer instance. The UpdateQuery script re-executes because the NodeName context changes, building a new alarm query that references only the alarms for the newly selected mixer's components. The operator sees only alarms relevant to that mixer.
