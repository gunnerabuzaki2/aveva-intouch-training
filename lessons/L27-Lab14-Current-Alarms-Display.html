Lesson 27: Lab 14 – Current Alarms Display (continued)

Source: AVEVA InTouch for System Platform 2023 Training Manual, Module 5 – Alarms and Events Visualization, Lab 14 (pages 351–362)

This lesson continues Lab 14 — acknowledging alarms and using runtime filters — then introduces the concept of context-aware alarm popups for Lab 15, and provides an overview of historical alarm data sources.

27.1 Acknowledging Alarms at Runtime

Reference: Module 5, Lab 14 (pages 351–352)

In the previous lesson, you created the AlarmLiveDisplay symbol and tested it at runtime. Now you will practice acknowledging alarms and using the filter context menu.

  1. In the Alarm Client, select a row, right-click, and select Ack Selected. An Ack Comment dialog appears.
  2. In Comment, enter ticket issued and click OK. The acknowledged alarm disappears from the Current Alarms list (since it is no longer in an active alarm state after acknowledgment + return to normal).
  3. Right-click in the Alarm Client and select Ack Others → Ack All.
  4. Enter ticket issued as the comment and click OK.

The Ack Comment dialog where operators enter a reason for acknowledging the alarm (page 351)
After acknowledging all alarms, the Alarm Client displays all alarms in ACK (Acknowledged) state (page 351)

27.2 Using Runtime Filters

Reference: Module 5, Lab 14 (pages 352–353)

The filters you created at design time (Line1, Line2, Production) can be activated at runtime through the context menu. This allows operators to dynamically narrow the alarm view without needing to modify the graphic.

  1. Right-click the AlarmLiveDisplay graphic and select Queries and Filters. The Query and Filter Favorites dialog appears.
  2. In Filters, check Line1 and click OK.
  3. Verify that only Line1 alarms are displayed.
  4. Try additional filters (Line2, Production) to confirm they work correctly.
  5. Click Development! to return to WindowMaker.
Accessing the Queries and Filters dialog from the Alarm Client's context menu at runtime (page 352)
Activating the Line1 filter at runtime to show only Line1 alarm group alarms (page 352)
The Alarm Client displaying only Line1-grouped alarms after applying the filter (page 353)
Key Concept: Filters can be created at design time and reused at runtime, or created at runtime for immediate use. This bidirectional capability makes it easy to adapt alarm views to changing operational needs.

27.3 Lab 15 – Creating an Alarm Popup Symbol

Reference: Module 5, Lab 15 (pages 355–360)

Lab Objectives:
  • Use scripting to dynamically change the Alarm Client's AlarmQuery property for context-aware alarms
  • Use the MyPlatform. relative reference in an expression
  • Show a symbol using the Me. relative reference with Show Symbol animation

In this lab, you will create an alarm display symbol embedded within each mixer object. The key innovation is that the alarm query is built dynamically using scripting — so when you view Mixer100's popup, you see only Mixer100's alarms; when you view Mixer200's popup, you see only Mixer200's alarms.

Create the Context-Aware Alarm Display

  1. In the IDE, on the Templates tab, in the Training\Project template toolset, open the $Mixer template for editing.
  2. Add a symbol named AlarmPopup and open it for editing.
  3. In Tools, click Alarm Client and place it on the canvas.
  4. On the Properties tab, set Name to AlarmClient, Width to 800, Height to 400.
Configuring the Alarm Client properties for the AlarmPopup symbol (page 356)

Add a Custom Property and Script

  1. Right-click the canvas and select Custom Properties.
  2. Add a custom property named NodeName with Data Type String, Default Value in Expression mode: MyPlatform.NodeName, Visibility Private.
  3. Click OK to close Edit Custom Properties.
  4. Right-click the canvas and select Scripts.
  5. Click Add Script. Name it UpdateQuery, set Expression to NodeName, Trigger to DataChange.
The NodeName custom property configured with the MyPlatform.NodeName expression reference (page 357)
  1. In the script body, enter the following VBScript:
Tip: This script builds the alarm query dynamically using Me. references. Because the symbol is embedded within each $Mixer instance, Me.Area and Me.Tagname automatically resolve to the correct mixer's area and tagname at runtime. 
'Build alarm query prefix.
dim prefix as string;
prefix = "\\" + NodeName + "\Galaxy!" + Me.Area + "!";

'Build alarm query.
dim almQry as string;
almQry = prefix + Me.Tagname + ".* ";
almQry = almQry + prefix + Me.Agitator.Tagname + ".* ";
almQry = almQry + prefix + Me.Inlet1.Tagname + ".* ";
almQry = almQry + prefix + Me.Inlet2.Tagname + ".* ";
almQry = almQry + prefix + Me.Outlet.Tagname + ".* ";
almQry = almQry + prefix + Me.Temperature.Tagname + ".* ";
almQry = almQry + prefix + Me.Level.Tagname + ".* ";

'Update alarm client with alarm query.
AlarmClient.AlarmQuery = almQry;
  1. Click OK to close Edit Scripts.
  2. Save and close the graphic.
The UpdateQuery script that dynamically constructs an alarm query for each mixer instance (page 358)

Add the Alarms Button to the Mixer Display

  1. In the Object Editor for the $Mixer template, open MixingProcess_Detailed for editing.
  2. Draw a button on the canvas, type Alarms, and press Enter.
  3. Place the button to the right of the Tank and set its ElementStyle to Intensity2.
  4. Double-click the Alarms button to open Edit Animations.
  5. Add a Show Symbol animation. In Reference, enter Me.AlarmPopup.
  6. Click OK to close Edit Animations.
  7. Save and close the graphic.
  8. Save and close, then check in the $Mixer template.
The Alarms button placed on the mixer display, positioned to the right of the tank (page 359)
Show Symbol animation configured to display Me.AlarmPopup when the button is clicked (page 359)

Test at Runtime

  1. Click Runtime.
  2. In the ContentDisplay window, click the Alarms button to show the alarm popup.
  3. Verify that the displayed alarms are only for Mixer100.
  4. Close the alarm popup.
  5. Use the Select Mixer dropdown to select another mixer, then click Alarms again.
  6. Verify that the displayed alarms are now only for the selected mixer.
  7. Close the alarm popup.
  8. Click Development! to return to WindowMaker.
The alarm popup showing Mixer100-specific alarms — the script dynamically built the query to filter to this mixer only (page 360)
Key Concept: The Me. relative reference is what makes this popup context-aware. Each mixer instance gets its own AlarmPopup symbol, and the Me. references automatically resolve to the correct mixer's tagname and area at runtime. The MyPlatform. reference resolves to the WinPlatform hosting the mixer.

27.4 Historical Alarm Data Sources (Overview)

Reference: Module 5, Section 3 (pages 361–362)

Alarm and event data can be logged using three different mechanisms:

Data SourceDescription
History BlocksProprietary format files that store plant data, alarm data, and event data
A2ALMDB DatabaseSQL Server database storing Application Server alarm and event data
Legacy Alarm DB LoggerLogs data to SQL Server (default database name: WWALMDB)

To enable historization, you must configure the host AppEngine with storage to historian and specify a Historian server. In the Alarms and Events dialog box (accessed via Galaxy → Configure → Alarms and Events Configuration), you can enable historizing alarms by severity level and events by type. Modifying historize options does not require a redeployment.

Knowledge Check

1. What happens when you acknowledge an alarm with the Ack Selected option?

Click to reveal answer

The Ack Comment dialog appears, allowing you to enter a comment (e.g., "ticket issued"). The alarm transitions from UNACK to ACK state. If the alarm has also returned to normal, it disappears from the Current Alarms view entirely.

2. Why does the UpdateQuery script in Lab 15 use Me. references instead of hardcoded tagnames?

Click to reveal answer

Because the AlarmPopup symbol is embedded within each $Mixer template instance, the Me. references automatically resolve to the correct mixer's area, tagname, and child object tagnames at runtime. This makes the popup context-aware — one symbol works for all mixers.

3. What is the difference between the MyPlatform. and Me. relative references?

Click to reveal answer

MyPlatform. references the WinPlatform that hosts the object (used here for NodeName). Me. references the current object instance itself (used for Area, Tagname, and child object references like Me.Agitator.Tagname).

4. What three data sources can be used to log historical alarm and event data?

Click to reveal answer

History Blocks (proprietary format files), A2ALMDB (SQL Server database for Application Server data), and the legacy Alarm DB Logger (SQL Server database, default name WWALMDB).
