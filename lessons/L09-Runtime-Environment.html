Lesson 9: Runtime Environment and Application Design

Module 2 — Getting Started | Pages 81–86

Up to this point, you have been working in WindowMaker — the InTouch development environment where you design windows, embed graphics, and configure animations. But an HMI application is only useful when it is running and displaying real-time data to operators. That runtime environment is provided by WindowViewer. In this lesson, you will learn how WindowViewer works, how InTouch applications integrate with Application Server through the InTouchViewApp and ViewEngine objects, and how to deploy and launch your application in production.

WindowViewer: The InTouch Runtime

WindowViewer is the program that executes InTouch applications at runtime. When an application is started in WindowViewer, the windows that are configured to open by default are launched automatically — unless WindowViewer was called from WindowMaker for testing purposes, in which case only the windows that were open in the editor are displayed.

Key Concept: WindowViewer
WindowViewer is the InTouch runtime engine. It displays real-time application data from the plant floor, allowing operators to monitor processes and interact with controls by clicking items on the screen. WindowViewer reads tag values, executes scripts, and renders the windows you designed in WindowMaker.

The information displayed in WindowViewer comes from real-time data sources — PLCs, Historians, Application Server, and other industrial data providers. Operators can interact with the application by clicking buttons, entering values, acknowledging alarms, and navigating between screens. All of these interactions happen through the runtime, not the development environment.

WindowViewer Properties

WindowMaker provides extensive options for customizing the WindowViewer runtime experience. These configuration settings are accessed through File → Configure → WindowViewer and are organized into several tabs, each controlling a different aspect of runtime behavior.

The WindowViewer configuration screen with the Preferences tab — startup and inactivity timeout settings are visible.

The available tabs and their purposes are:

TabPurpose
PreferencesMiscellaneous runtime behavior: I/O settings, automatic logoff, and virtual keyboard access. Controls how WindowViewer behaves during normal operation.
Application typeLicensing support options — determines how the application is licensed in production.
WindowControls which runtime menu items are visible, which keyboard keys are locked down, and the application's display resolution.
MemorySets minimum RAM allocations for storing and retrieving Industrial Graphics. Important for performance tuning.
StartupSpecifies which windows open automatically when WindowViewer starts. This is where you define your "home" windows.
Advanced formatRuntime data precision and numerical formatting options — controls how values are displayed to operators.
Managed applicationSettings for the deployed application's working directory, script timeout period, and how operator changes to deployed applications are handled.
Tip: The Startup tab is one of the most important configuration areas. The windows you select here will open automatically when WindowViewer launches in production. Make sure to select all the windows that form your application layout — in our case, Navigation, Logon, AlarmDisplay, OverviewDisplay, and ContentDisplay.

InTouchViewApp: The Galaxy Integration Object

When InTouch is used within an AVEVA Application Server Galaxy, the InTouch application is represented by an InTouchViewApp object. This object serves as the bridge between the InTouch HMI layer and the Application Server's model-driven architecture. It manages the entire lifecycle of the InTouch application within the Galaxy.

Key Concept: InTouchViewApp
The InTouchViewApp object represents an InTouch application in the Application Server environment. It manages check-in, check-out, and deployment of the application, and enables InTouch tags to be referenced as Application Server attributes through Message Exchange.

The InTouchViewApp object inherits from the standard $InTouchViewApp template and includes the capabilities of the Application Server InTouchProxy object. This means that InTouch tags can be browsed through the standard Application Server attribute browser and referenced by any Message Exchange client as InTouchViewApp attributes.

The Application Server Galaxy manages the InTouchViewApp through several key capabilities:

  • Check-in and check-out for multiple users — enabling collaborative development with version control
  • Security restrictions on who can deploy the application and when deployment is permitted
  • Backup and restore — protecting against accidental data loss
  • Deployment and undeployment — pushing applications to target platforms and pulling them back

To create an InTouch application, you derive an InTouchViewApp object from the $InTouchViewApp template. You can start from scratch or use an application template — exported .aaPKG files stored in the InTouch installation directory (typically C:\Program Files(x86)\Wonderware\InTouch\ApplicationTemplates). You can also create an application by copying an existing one. In all cases, opening the InTouchViewApp object launches WindowMaker for editing.

Pitfall: There is no dedicated object editor for the InTouchViewApp in the IDE. Double-clicking the object opens WindowMaker directly. This can be confusing if you expect to see an attribute editor like other Application Server objects.

ViewEngine: The Runtime Host

The ViewEngine object is a simplified version of the AppEngine, purpose-built to host InTouchViewApp objects (and ViewApp objects) at runtime. It is the execution environment that runs your InTouch HMI application within the Application Server Galaxy.

Key Concept: ViewEngine
A ViewEngine object is assigned to and deployed on a WinPlatform. It hosts and executes InTouchViewApp objects, and its scan rate determines the scan rate of all hosted InTouch applications. One ViewEngine can host multiple InTouchViewApp objects, but only one InTouch application can run per node at a time.

The ViewEngine provides a rich set of capabilities beyond simply hosting InTouch windows:

  • Deployment management — it supports deploy, undeploy, startup, and shutdown of hosted applications
  • Initialization logic — it contains the logic to set up InTouchViewApp objects when they are first deployed, enabling communication with other objects in the Galaxy
  • Application Server scripts and UDAs — these continue to run even when WindowViewer is shut down, providing a persistent execution layer
  • Historian integration — the ViewEngine can include Historian configuration, and all hosted InTouchViewApp objects share this historian
  • Runtime diagnostics — the ViewEngine provides its own diagnostic attributes that can be monitored, alarmed, and historized
Tip: The ViewEngine's scan rate directly affects how quickly tag value updates appear in the runtime. A shorter ScanPeriod on the ViewEngine minimizes latency when referencing InTouch tags as InTouchViewApp attributes. However, keep in mind that Historization and alarm options of the ViewEngine may not always be necessary since InTouchViewApp objects primarily serve as tag proxies.
Pitfall: The ViewEngine does not support redundancy. If high availability is required for your visualization layer, this limitation must be considered in your system architecture.

Deploying a Managed InTouch Application

Before a Managed InTouch application can be used in a production environment, it must be deployed to the target platform(s) from the IDE. An instance of the $InTouchViewApp template is treated like any other Application Server object — it must be assigned to a target node and deployed.

Deployment can target either the local node or a remote node. After deployment, you can run the application in WindowViewer on the remote node. The deployment process pushes the application files, configuration, and graphics to the target platform where the ViewEngine will host and execute them.

Right-clicking the InTouchViewApp instance in the IDE reveals deployment and management options including Deploy, Undeploy, and Upload Changes.

One important detail: associating all Galaxy graphics with the InTouchViewApp template means that deployed and published InTouch applications can render any graphic from the Galaxy without having to embed it directly in the application. This significantly reduces application file size and ensures visual consistency across the entire system.

AVEVA Application Manager

The AVEVA Application Manager is the standalone tool for managing and launching InTouch applications on a local machine. When first run, it scans the computer for existing InTouch applications and displays them as tiles in a visual interface.

The Application Manager showing the $MixerView application tile — note that WindowMaker is grayed out because this is a Managed InTouch application.
Key Concept: Managed vs. Unmanaged Applications
In the Application Manager, some toolbar buttons — such as WindowMaker, DBLoad, and DBDump — are grayed out for Managed InTouch applications. This is because Managed InTouch applications are controlled through the Application Server Galaxy (IDE), not through the local Application Manager. To run a Managed application, click the WindowViewer button on its tile.

The Application Manager displays important information about each application tile, including the resolution setting (for example, "Managed 1920×975") and the available launch options. For Managed InTouch applications, the primary action is to launch WindowViewer — all development and configuration work happens through the Galaxy IDE.

The Development-to-Runtime Workflow

Understanding the complete workflow from design to production is essential for any InTouch developer. Here is the typical sequence:

  1. Design in WindowMaker — create windows, embed Industrial Graphics, configure animations and scripts
  2. Test with fast-switch — use the Runtime button to jump into WindowViewer for quick testing
  3. Check in to Galaxy — commit your changes to the InTouchViewApp template in the IDE
  4. Configure WindowViewer settings — set startup windows, lock window size, configure runtime options
  5. Deploy from IDE — push the InTouchViewApp and ViewEngine instances to the target platform
  6. Launch in production — use Application Manager on the target machine to start WindowViewer
Tip: When using the Runtime fast-switch from WindowMaker, the windows that were open in the editor are the ones displayed in runtime — not the startup windows you configured. This is useful for testing specific windows during development, but always verify the final result by launching through Application Manager in production mode.

Quiz — Test Your Understanding

Q1: What is WindowViewer, and how does it differ from WindowMaker?

WindowMaker is the development environment where you design windows, embed graphics, and configure animations. WindowViewer is the runtime environment that executes the application, displaying real-time data from plant floor sources and allowing operators to interact with the HMI.

Q2: What role does the InTouchViewApp object play in the Application Server Galaxy?

The InTouchViewApp object represents the InTouch application within the Galaxy. It manages check-in/check-out, deployment, security, and backup/restore. It also enables InTouch tags to be referenced as Application Server attributes through Message Exchange, bridging the HMI and automation layers.

Q3: What is the ViewEngine object, and what determines its scan rate?

The ViewEngine is a simplified AppEngine that hosts and executes InTouchViewApp objects at runtime. It is assigned to a WinPlatform, and its configured scan rate determines the scan rate of all InTouchViewApp objects it hosts. A shorter ScanPeriod reduces tag value update latency.

Q4: Why are WindowMaker, DBLoad, and DBDump buttons grayed out for Managed InTouch applications in the Application Manager?

Managed InTouch applications are controlled through the Application Server Galaxy IDE, not through the local Application Manager. Development tasks like editing (WindowMaker), database load (DBLoad), and database dump (DBDump) are performed in the IDE. The Application Manager is used primarily to launch WindowViewer for production use.

Q5: What happens to Application Server scripts and UDAs hosted by a ViewEngine when WindowViewer is shut down?

Application Server scripts and UDAs (User Defined Attributes) hosted by the ViewEngine continue to run even when WindowViewer is shut down. The ViewEngine provides a persistent execution layer independent of the HMI runtime.
