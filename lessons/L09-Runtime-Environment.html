Lesson 9: Runtime Environment and Application Design

Source: AVEVA InTouch for System Platform 2023 Training Manual, Module 2, Section 4 (pages 81–86)

This lesson provides an overview of InTouch WindowViewer, the runtime environment, and explains how to deploy and run a Managed InTouch application.

9.1 WindowViewer Overview

Reference: Module 2, Section 4 (pages 81)

WindowViewer is the InTouch runtime environment — it is what operators see and interact with on the plant floor. When an application is started in WindowViewer, the windows configured to open by default are loaded automatically, displaying real-time application data from the plant. Operators can then control industrial processes by interacting with on-screen elements like buttons, switches, and data entry fields.

There is one exception: when WindowViewer is launched from WindowMaker for testing purposes, the windows that are open in WindowMaker are the ones that appear in WindowViewer — not the default startup windows.

Key Concept: WindowViewer displays real-time plant-floor data. Operators interact with the HMI by clicking items on screen to control industrial processes. The windows that open at startup are configured through WindowMaker's settings.

9.2 WindowViewer Configuration Properties

Reference: Module 2, Section 4 (pages 81)

WindowMaker includes extensive options for customizing the WindowViewer runtime environment. Access these by selecting File > Configure > WindowViewer from the backstage menu. The configuration is organized into several tabs:

TabConfiguration Options
PreferencesMiscellaneous runtime behavior: I/O settings, automatic logoff, virtual keyboard access, inactivity timeout and warning values
Application typeLicensing support options
WindowRuntime menu visibility, keyboard key locking, application display resolution
MemoryMinimum RAM allocation for storing and retrieving Industrial graphics
StartupSelection of windows to open when WindowViewer starts
Advanced formatRuntime data precision and numerical formatting options
Managed applicationWorking directory for deployed applications, script timeout period, operator experience for deployed changes
WindowViewer configuration Preferences tab — settings for startup behavior, access levels, inactivity timeouts, and blink frequencies (page 81)
Tip: The Inactivity timeout values (Warning and Timeout in seconds) control automatic logoff. Setting both to 0 disables automatic logoff. For production environments, consider setting reasonable timeout values for security.

9.3 InTouchViewApp in the Runtime Context

Reference: Module 2, Section 4 (pages 82–83)

The InTouchViewApp object represents the InTouch application within the Application Server Galaxy. It manages the check-in, check-out, and deployment lifecycle, and includes InTouchProxy capabilities so that InTouch tags can be referenced by other Galaxy objects as InTouchViewApp attributes.

You can create an InTouch application in two ways:

  • From scratch or template — Derive from the standard $InTouchViewApp template. Application templates (.aaPKG files) are available from the Application Templates folder in the InTouch installation directory.
  • From an existing application — Copy an existing InTouch application to create a new managed application.

Either way, opening an InTouchViewApp object launches WindowMaker — there is no separate object editor. You can assign multiple InTouchViewApp objects to the same ViewEngine to make multiple InTouch applications available on a node, but only one application can run at a time.

9.4 Deploying and Running an InTouch Application

Reference: Module 2, Section 4 (pages 82–84)

To run an InTouch application on an Application Server Platform, follow these steps:

  1. Create and assign a ViewEngine to a Platform (e.g., WinPlatform).
  2. Create an instance of the InTouchViewApp template that defines your application.
  3. Assign the InTouchViewApp instance to the ViewEngine instance.
  4. Deploy the ViewEngine and InTouchViewApp objects to the target node.
  5. Run Application Manager to select the application and start WindowViewer (only needed the first time).
Key Concept: Associating Galaxy graphics with an InTouchViewApp template allows deployed applications to request any graphic from the Galaxy without having to embed them directly in the InTouch application.
Pitfall: Before a Managed InTouch application can be used in production, it must be deployed to the target platform(s) from the IDE. The InTouchViewApp instance is treated like any other object and must be deployed.
Deploying an InTouchViewApp — right-click the object in the Deployment view and select Deploy (page 84)

9.5 AVEVA Application Manager

Reference: Module 2, Section 4 (pages 85)

The AVEVA Application Manager is the utility used to launch InTouch applications in runtime. When first opened, it scans the computer for existing InTouch applications and displays them as tiles.

For Managed InTouch applications, several toolbar buttons are grayed out — including WindowMaker, DBLoad, and DBDump — because these functions are not available for managed applications. To run a managed application, simply click the WindowViewer button on the application's tile.

AVEVA Application Manager — click the WindowViewer button on the $MixerView tile to launch the application in runtime (page 85)
Key Concept: The Application Manager only needs to be used the first time to select which application to run. After that, WindowViewer launches automatically when the deployed application starts on the target node.

Knowledge Check

1. What is the difference between launching WindowViewer from WindowMaker (for testing) versus launching it from Application Manager (in production)?

Click to reveal answer

When launched from WindowMaker for testing, the windows open in WindowMaker are shown in WindowViewer. In production (via Application Manager), the windows configured as startup windows in the WindowViewer settings are loaded.

2. What happens when you double-click an InTouchViewApp template in the IDE?

Click to reveal answer

WindowMaker launches automatically. There is no separate object editor for InTouchViewApp — WindowMaker is the development environment for InTouch applications.

3. Why are WindowMaker, DBLoad, and DBDump buttons grayed out for Managed InTouch applications in Application Manager?

Click to reveal answer

Managed InTouch applications are controlled through the Galaxy (check-in, check-out, deploy). Direct editing through Application Manager is not supported — you must use the IDE and WindowMaker instead.

4. What are the five steps to deploy and run an InTouch application on an Application Server Platform?

Click to reveal answer

(1) Create and assign a ViewEngine to a Platform, (2) Create an InTouchViewApp instance from the template, (3) Assign the InTouchViewApp to the ViewEngine, (4) Deploy both objects to the target node, (5) Use Application Manager to select and run the application (first time only).
