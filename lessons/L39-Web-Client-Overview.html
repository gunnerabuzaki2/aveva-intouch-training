Lesson 39: Web Client Overview

Source: AVEVA InTouch for System Platform 2023 Training Manual, Module 8 — Web Client, Section 1 (pages 527–536)

This lesson introduces the InTouch Web Client, covering its architecture, configuration, navigation, licensing model, and known limitations.

39.1 Module Objectives

Reference: Module 8, Section 1 (page 528)

This lesson covers the following topics:

  • Describe the Web Client features and dependencies
  • Explain how to enable the Web Client
  • Explain how to navigate graphics in the Web Client
  • Explain the use of the Web Client fast switch

39.2 What Is the Web Client?

Reference: Module 8, Section 1 (pages 529–530)

The InTouch Web Client allows you to view Industrial Graphics from a managed InTouch for System Platform application in any HTML5-capable web browser. A built-in web server handles the delivery of application graphics to browsers on any Microsoft Windows client or server operating system — no Remote Desktop Protocol (RDP) or Internet Information Services (IIS) is required. Only Managed applications can be served through the Web Client.

Key Concept: The Web Client is a built-in web server that streams InTouch Industrial Graphics to HTML5 browsers, eliminating the need for RDP or IIS.
Web Client architecture showing Design Time (WindowMaker, IDE publishing to Web Server), the InTouch Web Server (licensing, security, browser connections), and Run Time (WindowViewer, Application Server providing iData) (page 529)

39.3 Enabling the Web Client

Reference: Module 8, Section 1 (pages 529–530)

By default, the Web Client is disabled to protect systems where no decision has been made to expose a web server. You must have administrative privileges to enable or disable the feature. The Web Client is enabled from the AVEVA Application Manager on the Web Client tab. When the Web Client is not started, its toolbar icon displays a small clock overlay. Clicking the icon starts the Web Server, and the clock disappears once the server is running.

Application Manager with the Web Client button showing a clock icon, indicating the Web Server has not yet been started (page 529)
Tip: Always start the Web Client from Application Manager after your InTouch application is deployed. The clock icon is a quick visual indicator of whether the server is running.

39.4 Configuring Graphics for Web Viewing

Reference: Module 8, Section 1 (page 530)

To control which graphics are accessible through the Web Client, you configure a root folder in the Industrial Graphic Toolbox within WindowMaker. Only graphics stored in the folder set as the Web Client root folder will appear in the browser. Within that root folder, you can designate a single symbol as the Home Symbol by enabling the Set Web Client Home Symbol option. This symbol is automatically displayed when a user opens the Web Client, and the interface provides a Home button to return to it at any time.

Key Concept: Only Industrial Graphics placed in the designated Web Client root folder are visible in the browser. A single Home Symbol can be set as the default display.

39.5 The Web Client Fast Switch

Reference: Module 8, Section 1 (page 531)

When using the Web Client with InTouch for System Platform, WindowViewer must be running to supply live data to the Web Client. The Web Client fast switch in WindowMaker provides a convenient toggle to open the Web Client directly in your default browser. Initially, this fast switch may be disabled if the InTouch application path does not match the current application — for example, after importing a restored application or switching between applications.

Starting WindowViewer with the Runtime fast switch accomplishes three things:

  • Sets the application path to the current application and builds the application's Web folder
  • Enables the Web Client fast switch in WindowMaker (if it was disabled)
  • Starts WindowViewer, which provides the live data needed for Web Client animations
The IDE toolbar showing the Web Client and Runtime toggle buttons alongside the Industrial Graphics toolbox (page 531)

39.6 Web Client Page Elements

Reference: Module 8, Section 1 (pages 532–533)

The Web Client page is organized into several distinct areas that control navigation, display, and user interaction. The following table summarizes each element:

ElementDescription
1 — Title BarDisplays the InTouch application name and a hamburger button. Clicking the hamburger button opens the folder navigation overlay, listing all folders and symbols under the Web Client root folder.
2 — Action IconsHome, Share, Notifications, Full Screen, and Profile icons appear on the title bar. Click Notifications to view messages related to connection and licensing issues.
3 — Canvas AreaDisplays the selected graphic from the navigation overlay. The graphic scales to fit the browser viewport while maintaining its aspect ratio. Navigation and notification overlays do not consume canvas space.
The Web Client running in a browser, showing the $MixerView HMI interface with navigation tabs, alarm table, and process schematic (page 532)

39.7 Pan and Zoom Support

Reference: Module 8, Section 1 (page 533)

The Web Client supports pan and zoom gestures on all supported browsers and devices, providing behavior similar to Frame Window support in WindowViewer. When zooming from a non-mobile device, the current zoom percentage is displayed in the lower-left corner of the horizontal scrollbar. Zoom is limited to a maximum of 500%.

Key Concept: Pan and zoom in the Web Client mirror the Frame Window experience in WindowViewer. Hold Ctrl and scroll to zoom; hold the middle mouse button to pan.

39.8 Home Button Behavior

Reference: Module 8, Section 1 (page 533)

Standalone application windows can be viewed in the Web Client, but they must first be converted to Industrial Graphics. For InTouch applications, you can use the Home Windows tab in WindowMaker to set which windows appear by default in WindowViewer. Similarly, you can designate a graphic as the default symbol for the Web Client.

The Home button behavior depends on which configuration is in place:

  • Home Windows configured: Even if a Home Symbol is also configured, only the Home Windows are displayed by default. Clicking the Home icon shows the Home Windows.
  • Home Symbol configured (no Home Windows): The home symbol displays by default when the Web Client launches.
  • Neither configured: No graphic or window appears by default, and clicking the Home button has no effect.

39.9 Automatic Symbol Refresh and URL Access

Reference: Module 8, Section 1 (pages 533–534)

Changes made to graphics through the Graphic Editor are automatically refreshed in the browser. However, changes to Quality and Status Styles, Element Styles, and Formatting Styles (for InTouch) are only propagated after specific graphic operations occur, such as updating and saving symbol content, creating or deleting a symbol, moving a symbol to a different toolset folder, or updating the root folder or home symbol assignment. Re-launching WindowViewer also forces these style changes to propagate.

To display an individual symbol directly via URL, you can use the following syntax:

http://<hostname>/InTouchWeb/api/symbol/<graphicname>

For example, http://production/InTouchWeb/api/symbol/tankfarm would display a mixer symbol representing a production vessel in a tank farm.

Tip: Direct URL access is useful for embedding specific process graphics in third-party dashboards or custom portals without navigating through the full Web Client interface.

39.10 Known Limitations

Reference: Module 8, Section 1 (page 534)

A number of features common to InTouch for System Platform applications are not yet supported by the Web Client. Notably, no ActiveX or .NET controls — such as the Alarm Client — are supported at this time. For a detailed list and description of known limitations, consult the PDF file entitled Viewing InTouch Application Graphics in a Web Browser.

Warning: ActiveX and .NET controls (including Alarm Client) are not supported in the Web Client. Plan your application accordingly if browser-based viewing is a requirement.

39.11 Licensing

Reference: Module 8, Section 1 (pages 534–535)

Without a license installed, the Web Client operates in Single Session Mode — a single browser can connect to the Web Server (multiple tabs within that browser count as one session), but a second browser cannot access the server. For full unrestricted use, you need a valid Web Server license.

The Web Server provides two license types:

License TypeCapability
Unlimited Read-OnlyUnlimited sessions for viewing application graphics in a browser
Unlimited ReadWriteUnlimited sessions for viewing and interacting with application graphics in a browser
Key Concept: A ReadWrite license is required to write values or acknowledge alarms from the browser. Read-Only licenses only allow viewing.

39.12 Authentication and Authorization

Reference: Module 8, Section 1 (pages 534–535)

After the Web Server starts, it first authenticates the user and then determines authorization. During authentication, the server validates whether the user belongs to the aaInTouchUsers or aaInTouchRWUsers security groups — both are created during installation. The login user at the time of Web Client installation is automatically added to both groups. Additional users can be added later, but they must log off and re-login for the change to take effect.

The server then attempts to acquire the ReadWrite license first. If that is unavailable, it tries the Read-Only license. If neither license is available, the Web Client falls back to Single Session Mode.

ModeRequirementsCapabilities
ReadWriteReadWrite license acquired and user in aaInTouchRWUsersWrite to attributes, acknowledge alarms with operator details
Read-OnlyReadWrite license with user not in RW group, or Read-Only license, or first session in Single Session ModeView only — no writes or alarm acknowledgments
Single SessionNo license acquired at startupOne browser session only, Read-Only
Warning: If a Web Server loses its license after startup, sessions are given a grace period. However, if the license is not restored, new sessions will be denied.

39.13 Web Client Application Windows

Reference: Module 8, Section 1 (page 535)

The Web Client converts all Frame Windows in your application and makes them available from the hamburger menu. If Home Windows are configured and they are Frame Windows, they display when the Web Client is launched. Any navigation configured in your application — such as buttons to show windows or scripting using ShowGraphic(), ShowHome, or Show script functions — behaves normally, just as it would in WindowViewer.

Replace window functionality also works as expected, closing any window the Replace window touches. When using ShowGraphic(), the identity property must use the InTouch:WindowName syntax to properly identify the XY position and dimensions of the window for replacement purposes. If the identity property does not use the InTouch: prefix, a brand new popup window is created in the Web Client that floats on top of other windows instead of replacing them.

Warning: When using ShowGraphic() in the Web Client, always use the InTouch:WindowName syntax for the identity property. Without the InTouch: prefix, the graphic opens as a separate popup rather than replacing the current window.

Knowledge Check

1. What is the primary advantage of the Web Client over using RDP or IIS to view InTouch graphics?

Click to reveal answer

The Web Client uses a built-in web server, so no RDP or IIS configuration is needed. Graphics are served directly to any HTML5 browser on any Windows OS.

2. What does the small clock icon on the Web Client button in Application Manager indicate, and what happens when you click it?

Click to reveal answer

The clock icon indicates the Web Server is not started. Clicking the button starts the Web Server, and the clock disappears once the server is running.

3. What is the difference between the aaInTouchUsers and aaInTouchRWUsers security groups?

Click to reveal answer

aaInTouchUsers grants Read-Only access to the Web Client. aaInTouchRWUsers grants ReadWrite access, allowing users to write to attributes and acknowledge alarms.

4. Why must the ShowGraphic() identity property use the InTouch:WindowName syntax?

Click to reveal answer

Without the InTouch: prefix, the Web Client creates a new popup window instead of replacing the current window. The InTouch: syntax allows proper XY positioning and size identification for replace behavior.

5. What happens when the Web Server starts with no license available?

Click to reveal answer

The Web Server enters Single Session Mode, allowing only one browser session with Read-Only access. A second browser connection will be denied.
