Lesson 17: The OwningObject Property + Lab 8 – Multiple Mixers

Module 3 — Symbols | Pages 203–213

In the previous lab, you created a complete mixer display for a single mixer instance. But in a real plant, an operator often needs to monitor many similar units — eight mixers on two production lines, for example — without opening a separate screen for each one. The OwningObject property solves this problem elegantly: it lets a single graphic dynamically switch which automation object it references at runtime. Combined with a Combo Box drop-down list, an operator can select any mixer from a list and see its live data displayed instantly.

The OwningObject Property

Key Concept — OwningObject: OwningObject is a runtime property that replaces all Me. references within an embedded graphic. When you set OwningObject to a different object name (for example, switching from Mixer100 to Mixer500), every relative reference in that graphic — Me.Level.PV, Me.Temperature.PV, Me.Area, and so on — automatically resolves to the newly referenced object. The graphic itself does not change; only the data source behind it changes.

Think of OwningObject as a "pointer swap." The graphic contains relative references like Me.Pump1\Pump_Detailed, Me.Level\Level_Detailed, and text animations referencing Me.Area. By default, Me resolves to the object that hosts the graphic (for example, Mixer100). When you reassign OwningObject to Mixer500, every Me. reference in the graphic now resolves to Mixer500's attributes — the pumps, valves, level, temperature, and area all switch to Mixer500's data without any code changes.

The OwningObject concept: a Combo Box changes which object the graphic's Me. references resolve to.

Windows Common Controls

InTouch's Graphic Editor provides several standard Windows controls that you can place on your canvas just like any other graphic element. These controls enable user interaction at runtime:

  • Radio button group — mutually exclusive selection
  • Check box — toggle on/off
  • Edit box — free-text input
  • Combo Box — drop-down selection from a list
  • Calendar control — date picker
  • DateTime picker — date and time selection
  • List box — scrollable list selection

Each control is selected from the Tools pane and placed on the canvas by clicking and dragging to define its size. After placement, you can configure its background color, text color (except DateTime Picker), control-specific properties, and control-specific animations. Of these controls, the Combo Box is the one you will use in this lab to build the mixer selector.

Combo Box Control and Animation

A Combo Box presents users with a foldable drop-down list from which they can select a value. It supports three modes:

TypeBehavior at Runtime
SimpleUser can type a value or select one using up/down arrow buttons.
DropDownUser can type a value or select one from the drop-down list.
DropDownListUser can only select from the list — no typing allowed.

The Combo Box animation is configured through the Edit Animations dialog, where you define:

  • Reference — the property the Combo Box writes to when the user makes a selection (in this lab, MixingProcess.OwningObject)
  • Static Values and Captions — the list of items displayed in the drop-down
  • Submit Value Changes — whether changes take effect immediately or on a trigger
Key Concept — Combo Box-Specific Runtime Properties: At runtime, Combo Box controls expose properties you can use in scripts: Count returns the number of items in the list, and NewIndex returns the index of the last item added. These are accessible through the attribute/tag browser for the HMI.
Pitfall — OwningObject Must Match Object Names Exactly: The values you enter in the Combo Box's Static Values list must match the exact tag names (hierarchical names) of the automation objects in your Galaxy. If you type "Mixer100" but the object is actually named "MIXER100", the OwningObject assignment will fail silently and the graphic will not update.

Lab 8 — Adding a Mixer Selector with OwningObject

Lab Objectives: Upon completing this lab, you will be able to:
  • Use the OwningObject property to dynamically change instance references of a graphic
  • Add and configure a Combo Box control
  • Embed a switched graphic in WindowMaker for runtime testing

Step 1: Create the SwitchedMixingProcess Symbol

In the IDE, open the $Mixer template and add a new content item called SwitchedMixingProcess. Click Edit Content to open the Graphic Editor. Right-click the canvas, select Embed Industrial Graphic, and from the Relative References, embed Me.MixingProcess_Detailed — the mixer display you built in Lab 7. Place it on the canvas and name the embedded graphic MixingProcess.

The SwitchedMixingProcess canvas with the embedded mixer display and placeholder label.

Step 2: Add the "Select Mixer" Label and Combo Box

Add a Text element to the right of the $Mixer label with the text Select Mixer. Name it SelectMixer and set its ElementStyle to Heading. This provides a clear visual cue for the operator.

The Select Mixer label positioned above where the Combo Box will go.
Configuring the SelectMixer text element properties.

Now select the Combo Box tool from the Tools pane. Below the "Select Mixer" label, click and drag diagonally to draw the Combo Box. Name it MixerDropDown.

The Combo Box tool in the Tools palette.
Placing the Combo Box on the canvas below the label.
Configuring the MixerDropDown element properties.

Step 3: Configure the Combo Box Animation

Double-click the Combo Box on the canvas to open Edit Animations. In the Reference field, enter MixingProcess.OwningObject. This is the critical link: when the user selects a mixer from the drop-down, the selected value is written to the OwningObject property of the MixingProcess graphic, causing all its Me. references to switch to the new mixer.

Tip — Understanding the Reference Path: MixingProcess is the name you gave to the embedded MixingProcess_Detailed graphic on the canvas. OwningObject is its built-in runtime property. So MixingProcess.OwningObject means "the OwningObject property of the MixingProcess element on this canvas."

In the Static Values and Captions table, click the Add a row button eight times — one row for each mixer. Before entering values, configure the table settings:

  • Check Use Values as Captions — this means the caption text automatically matches the value, so you only need to fill in one column
  • Uncheck AllowDuplicates — each mixer name must be unique
  • Set Type to DropDown — users can type or select from the list

In the Value column, enter the eight mixer tag names: Mixer100, Mixer200, Mixer300, Mixer400, Mixer500, Mixer600, Mixer700, Mixer800. Press Tab after each entry to move to the next row. Click OK to close Edit Animations, then save, close, and check in the $Mixer template.

Setting the Combo Box Reference to MixingProcess.OwningObject.
The Static Values table with eight empty rows ready for mixer names.
Table settings: Use Values as Captions enabled, AllowDuplicates unchecked, Type = DropDown.
The fully configured Combo Box animation with all eight mixer entries.

Step 4: Replace the Symbol in WindowMaker

Switch to WindowMaker and open the ContentDisplay window. Right-click and select Embed Industrial Graphic. In the Galaxy Browser, click Instances, scroll down to Mixer100, and select SwitchedMixingProcess. Click OK, then confirm the replacement by clicking Yes.

Selecting SwitchedMixingProcess from the Galaxy Browser.
The SwitchedMixingProcess symbol now embedded in the ContentDisplay window.

Step 5: Test in Runtime

Click Runtime to start the application. The mixer display appears with the Select Mixer drop-down list. By default, it shows Mixer100's data (since that is the instance the symbol is hosted on). Now use the drop-down to select Mixer500. Watch as the entire graphic updates — the object name changes to Mixer500, the area name updates, and all process values (level, temperature, agitator speed, pump statuses) switch to Mixer500's live data.

Try selecting different mixers from the list to see the display update each time. This demonstrates the power of OwningObject: one symbol serves all eight mixers. Click Development! to return to WindowMaker when you are finished testing.

Runtime view after selecting Mixer500 — all values and labels have updated to reflect the new instance.
Key Concept — Why This Matters: Without OwningObject, you would need eight separate graphics in WindowMaker — one for each mixer — each hard-coded to its own object. OwningObject eliminates that redundancy. You build one graphic, and operators can switch between any number of similar objects at runtime. This dramatically reduces development time and simplifies maintenance.

Quiz — Test Your Understanding

Q1: What does the OwningObject property do when set to a different automation object name at runtime?

It replaces all Me. references in the embedded graphic so they resolve to the newly referenced object. For example, if OwningObject is set to "Mixer500", then Me.Level.PV resolves to Mixer500.Level.PV instead of Mixer100.Level.PV.

Q2: What is the difference between the Simple, DropDown, and DropDownList Combo Box types?

Simple lets users type or use arrow buttons. DropDown lets users type or select from a list. DropDownList restricts users to only selecting from the list — no free-text typing is allowed. DropDownList is often preferred for OwningObject selectors to prevent invalid object names.

Q3: In the Combo Box animation, what does the Reference field "MixingProcess.OwningObject" mean?

MixingProcess is the name of the embedded graphic element on the canvas (the MixingProcess_Detailed symbol). OwningObject is its built-in runtime property. The Reference tells the Combo Box: "When the user selects a value, write it to the OwningObject property of the MixingProcess element."

Q4: Why must the Combo Box values exactly match the tag names in the Galaxy?

OwningObject requires the exact hierarchical name of an automation object to resolve references correctly. If the value does not match (e.g., a typo or case mismatch), the OwningObject assignment fails silently — the graphic will not update and may continue showing stale data from the previous object, which could mislead operators.

Q5: What is the benefit of checking "Use Values as Captions" when configuring the Combo Box Static Values table?

It automatically populates the Caption column with whatever you type in the Value column, so you only need to fill in one column instead of two. This reduces data entry errors and keeps the display text consistent with the actual object names sent to OwningObject.
