Lesson 21: Lab 10 – Creating a Navigation Symbol

Module 3 — Symbols | Pages 255–276

Lab Objectives
In this lab, you will build a complete navigation system for switching between different process views at runtime. You will create reusable navigation buttons and a container that uses scripting to show and hide mixer display graphics dynamically.

21.1 Why Navigation Matters

In any real-world HMI, operators need to move quickly between different areas of the plant — a production line, a specific mixer, or a system overview. Rather than creating a separate window for each view, InTouch provides a more elegant approach: pop-up graphics that appear within a single content area. By using the ShowGraphic() and HideGraphic() scripting functions, you can build a navigation symbol that swaps content on demand, giving operators a seamless, application-like experience.

Key Concept: Pop-Up Graphics as Navigation
Instead of switching between full windows, navigation symbols display pop-up graphics inside a designated content area. The pop-ups have no title bar, close button, or border, making them appear as if they are part of the main window. This technique gives your application a modern, unified feel.

21.2 Creating the Mixer Display Container

The first step is to create a symbol that will serve as a container for displaying mixer process graphics. This symbol uses a rectangle as a placeholder and two custom properties to control which graphics appear and where they are positioned.

Begin by opening the Line1_MixingProcess symbol you created in an earlier lab. Set the canvas zoom to 75% so you can see the full layout. You will duplicate the existing mixer graphic four times, each time selecting a different mixer instance (Mixer100, Mixer200, Mixer300, Mixer400) using the Select Alternate Instance option from the right-click menu. This gives you a 2×2 grid of mixer graphics.

The canvas at 75% zoom with the Mixer100 graphic and its duplicate, ready for instance assignment

Use the alignment toolbar to arrange the four graphics neatly: select the top two and click Align Top, then select the bottom two and click Align Bottom. Repeat for the left and right pairs using Align Left and Align Right. Save and check in the symbol when finished.

Next, create a new symbol called Line2_MixingProcess. This symbol will contain a rectangle named MixerDisplay (width 1490, height 670) that acts as the display area for the pop-up mixer graphics. Add a custom property called ShowTrigger with a String data type — this property will be the trigger that tells the symbol which mixers to display.

The MixerDisplay rectangle configured in Properties with its dimensions set

21.3 Writing the Show/Hide Scripts

The real magic happens in the scripts. Right-click the canvas and open Scripts. You will add two types of scripts: a predefined On Hide script and a named ShowMixer script.

The On Hide script runs whenever the symbol is hidden (for example, when the operator navigates away). It loops through View5 to View8 and calls HideGraphic() on each, ensuring all previously displayed pop-up windows are closed before new ones appear:

'Hide all graphics before displaying new graphics.
dim i as integer;
for i = 5 to 8
  'Identities: View1, View2, View3, View4.
  HideGraphic( "View" + StringFromIntg(i, 1) );
next;
Tip: The HideGraphic() function takes an identity string as its argument. Each pop-up graphic is given a unique identity (View5, View6, View7, View8) when it is shown, so you can target it individually for hiding.

The ShowMixer named script is triggered by a DataChange event on the ShowTrigger property. When ShowTrigger changes, the script loops from i = 5 to 8, building a GraphicInfo object for each mixer pop-up. For each iteration, it sets the identity, graphic name, size (half the MixerDisplay dimensions), and position (top-left, top-right, bottom-left, or bottom-right quadrant of the MixerDisplay area). Finally, it calls ShowGraphic() to display each pop-up:

'Show 4 Mixer display using ShowGraphic() function.
dim i as integer;
dim graphicInfo as aagraphic.GraphicInfo;
for i = 5 to 8
  graphicInfo.Identity = "View" + StringFromIntg(i, 1);
  graphicInfo.GraphicName = "Mixer" + StringFromIntg(i, 1) +
    "00.MixingProcess_Detail";
  graphicInfo.HasTitleBar = false;
  graphicInfo.HasCloseButton = false;
  graphicInfo.RelativeTo = aaGraphic.RelativeTo.CustomizedWidthHeight;
  graphicInfo.Width = (MixerDisplay.Width)/2;
  graphicInfo.Height = (MixerDisplay.Height)/2;
  graphicInfo.WindowLocation = aaGraphic.WindowLocation.TopLeftCorner;
  graphicInfo.WindowRelativePosition =
    aaGraphic.WindowRelativePosition.ClientAreaXY;
  if (i == 5) then
    graphicInfo.X = MixerDisplay.X;
    graphicInfo.Y = MixerDisplay.Y;
  elseif (i == 6) then
    graphicInfo.X = MixerDisplay.X + (MixerDisplay.Width)/2;
    graphicInfo.Y = MixerDisplay.Y;
  elseif (i == 7) then
    graphicInfo.X = MixerDisplay.X;
    graphicInfo.Y = MixerDisplay.Y + (MixerDisplay.Height)/2;
  elseif (i == 8) then
    graphicInfo.X = MixerDisplay.X + (MixerDisplay.Width)/2;
    graphicInfo.Y = MixerDisplay.Y + (MixerDisplay.Height)/2;
  endif;
  ShowGraphic( graphicInfo );
next;
Key Concept: The GraphicInfo Object
The aagraphic.GraphicInfo structure controls every aspect of how a pop-up graphic appears: its identity, name, position, size, and whether it has a title bar. By programmatically setting these properties, you can place multiple graphics precisely within a container — no manual positioning required at runtime.

21.4 Building the Navigation Button

With the display container ready, the next step is to create a reusable navigation button. Create a new symbol called NavigationButton. This button needs several custom properties:

PropertyData TypePurpose
GetButtonTextStringTracks which button is currently active
GraphicNameStringThe graphic to display when clicked
WindowModalBooleanWhether the pop-up is modal
WindowNameStringTarget InTouch window (blank for pop-up)
WindowResizableBooleanWhether the pop-up can be resized

Add a button element to the canvas and attach an Action Scripts animation. The script, triggered on Left/Key/Touch Up, reads the custom properties and calls ShowGraphic() to display the target graphic. It also checks whether the WindowName is set — if it is, the graphic targets an InTouch window; if blank, it opens as a pop-up:

dim graphicInfo as aaGraphic.GraphicInfo;
GetButtonText = Text;
if (WindowName <> "") then
  graphicInfo.Identity = "InTouch:" + WindowName;
else;
  graphicInfo.Identity = GraphicName;
endif;
graphicInfo.GraphicName = GraphicName;
if (WindowModal) then
  graphicInfo.WindowType = aaGraphic.WindowType.Modal;
endif;
graphicInfo.Resizable = WindowResizable;
ShowGraphic( graphicInfo );

To provide visual feedback, add a Disable animation that grays out the button when it matches the currently active view (GetButtonText == Text), and an Element Style animation that changes the button's appearance to indicate selection.

21.5 Assembling the SystemNavigation Symbol

Now you bring everything together. Create a new symbol called SystemNavigation and add a custom property called LastButton (String type). Embed three copies of your NavigationButton side by side:

  • Mixers — targets Mixer100.SwitchedMixingProcess in the ContentDisplay window
  • Line 1 — targets Line1_MixingProcess
  • Line 2 — targets Line2_MixingProcess

Each button's GetButtonText is bound to the LastButton property, and each has its own GraphicName configured. When a button is clicked, it writes its text to LastButton, which triggers the Disable animation on all buttons — disabling the one that matches. This creates a simple "active tab" effect.

Important: The runtime resolution must be set to 1920 × 1080 for this lab to work correctly. If you are connecting to a remote VM, ensure the display is set to Full Screen. At other resolutions, Microsoft's popup window behavior may cause graphics to overlap or shift position.

21.6 Testing in Runtime

Embed the SystemNavigation symbol into your Navigation window in WindowMaker. Click Runtime to switch to WindowViewer. When you click the Line 1 button, the button highlights and four mixer process displays appear in the content area. Click Line 2 and the view switches to show the Line 2 mixers.

Runtime view with Line 1 selected — four mixer process graphics displayed in a 2×2 grid

The mixers you see are actually four borderless, titlebar-free popup windows positioned to look like part of the ContentDisplay window. When you navigate away from Line 2, the On Hide script automatically closes these pop-up windows, keeping the interface clean.

Line 2 view — the pop-up windows float over WindowViewer, giving the illusion of a single unified display
Tip: Remember that the popup windows have no close button. The only way to dismiss them is through the On Hide script, which fires when you navigate to a different view. This is by design — it prevents operators from accidentally closing individual mixers and losing their place.

Quiz — Test Your Understanding

Q1: What scripting function is used to display a pop-up graphic, and what function closes it?

The ShowGraphic() function displays a pop-up graphic, and HideGraphic() closes it.

Q2: What custom property on the NavigationButton tracks which button is currently active?

The GetButtonText property, which is bound to the LastButton property on SystemNavigation.

Q3: Why do the pop-up mixer graphics have no title bar or close button?

The title bar and close button are hidden so the pop-ups appear to be part of the main content area, giving the application a unified, seamless look.

Q4: What happens to the pop-up windows when you navigate away from the Line 2 view?

The On Hide script fires, which loops through View5–View8 and calls HideGraphic() on each, closing all pop-up windows automatically.
