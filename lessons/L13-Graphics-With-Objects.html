Lesson 13: Graphics with Objects

Module 3 — Symbols | Pages 145–148

Key Takeaway: This lesson explains how symbols interact with automation objects — specifically the difference between linking external symbols and adding symbols directly to an object, how relative references resolve at runtime, and how you can swap alternate symbols within the same object instance.

In the previous lessons, you learned how to create symbols on the Graphics tab and embed them into windows. But in a real AVEVA System Platform project, the primary consumers of those symbols are automation objects — the templates and instances that model your physical plant equipment. Understanding how symbols relate to objects is essential for building reusable, maintainable HMI graphics.

Inserting Symbols from an Automation Object

From the graphics perspective, symbols that are added to automation objects are referenced just like any other attribute within that object — you simply use the symbol's name. When you want to embed a symbol, either inside another symbol or on a WindowMaker window, you have two choices in the Galaxy Browser: you can select the Template Toolbox button to browse symbols from templates, or the Instances button to browse symbols from specific instances.

Templates vs. Instances: A template is the blueprints; an instance is the real thing. When you embed a symbol from a template, you are prompted for a unique instance name because the software will attempt to create a new Galaxy instance. If you want to use an existing instance's symbol, select Instances instead and pick the one you need.

The Galaxy Browser will only display objects — templates or instances — that actually contain symbols. If an object has no associated symbol, it simply will not appear in the browser. When you select an object, all of its symbols are displayed for you to choose from.

Warning: If you embed a symbol from a template in WindowMaker, you will be prompted to name a new instance. The name must be unique within the Galaxy, because the software will attempt to create that instance automatically. If you actually wanted to use an existing instance's symbol, make sure you selected Instances rather than Templates.

Linked Symbols vs. Added Symbols

This is one of the most important conceptual distinctions in InTouch graphics management. There are two fundamentally different ways to associate a symbol with an automation object, and choosing the wrong one can lead to maintenance headaches down the road.

Linked Symbols

Linking an external symbol to an automation object creates an association between the object and a symbol that lives on the Graphics tab. The symbol on the Graphics tab can be linked to multiple automation objects — even objects that are not part of the same derivation hierarchy. When you make changes to the symbol on the Graphics tab, those changes automatically propagate to every automation object to which the symbol is linked.

Linked Symbol Behavior:
  • Symbols on the Graphics tab can be linked to automation objects
  • Linked symbols can be embedded into other symbols
  • Changes to a linked symbol propagate to all linked automation objects and all graphics in which it is embedded
  • Relative references in a linked symbol are resolved through the object hierarchy at runtime
  • Deleting the automation object does not delete the symbol — it still exists on the Graphics tab

Added Symbols

Adding a symbol to an automation object creates a symbol that is natively hosted by that object. You can edit the symbol within the object — for example, by embedding a symbol from the Graphics tab and then customizing it specifically for that object. When the symbol in a template object changes, those changes propagate to all child (derived) objects and wherever those child objects are referenced.

Warning — Deletion Behavior: Deleting an automation object that has added symbols will also delete those symbols, because the symbols only exist within the automation object. In contrast, deleting an automation object with linked symbols does not delete the symbols, because they still exist independently on the Graphics tab. Choose your approach carefully based on whether you want the symbol to be disposable with the object or persistent.

When you are editing an object in the Object Editor, you can add or link symbols from the Content pane on the Attributes tab. Clicking the Add button creates a new symbol natively within the object, while clicking the Link Content button opens the Galaxy Browser so you can browse for and select a symbol from the Graphics tab to link. You can also select a symbol from the list and click Edit Content to open it in the Graphic Editor.

The Object Editor Content pane with Add (for native symbols) and Link Content (for external symbols) buttons.
An object with a linked content item — MixingProcess_Detailed — ready for editing.
When to use which? Use linked symbols when you want a single graphic to be shared across many objects and maintained in one place. Use added symbols when you need a graphic that is unique to a specific object and will evolve independently from other objects.

Relative References in Graphics

When you configure graphics that will be reused across multiple objects, you cannot hard-code absolute attribute paths like Mixer100.Totalizer1.PV. Instead, you use relative references — keywords that navigate up the object hierarchy to resolve to the correct attribute at runtime. This is what makes a single symbol usable across Mixer100, Mixer200, Mixer300, and every other mixer instance.

Available Relative References: The following keywords refer to the current object and can be used in attribute bindings within graphics:
  • Me — the current object
  • MyArea — the area (parent) of the current object
  • MyContainer — the container of the current object
  • MyEngine — the engine of the current object
  • MyHost — the host of the current object
  • MyPlatform — the platform of the current object

Relative references are resolved at runtime when the graphic is displayed in the running application. This means that the same symbol can display completely different data depending on which object instance it is embedded in — the references adapt automatically based on context.

Practical example: If you bind a temperature indicator to Me.Temperature.PV inside a symbol, that same symbol will show Mixer100's temperature when embedded in Mixer100, Mixer200's temperature when embedded in Mixer200, and so on — without any changes to the graphic itself.

Selecting Alternate Symbols

An automation object can include more than one symbol. The Select Alternate Symbol feature allows you to replace an embedded symbol with a different symbol that belongs to the same instance. This is useful when you want to swap between different visual representations of the same equipment — for example, switching from a detailed P&ID view to a simplified mixer view without changing the underlying data binding.

This feature is available in both the Graphic Editor and WindowMaker. In the Graphic Editor, select the embedded symbol and go to Edit > Embedded Symbol > Select Alternate Symbol (or right-click and select the same). In WindowMaker, the menu path is slightly different: Edit > Industrial Graphic "<current graphic name>" > Select Alternate Symbol.

The Galaxy Browser displays multiple symbols belonging to the same instance — you can select an alternate symbol from this list.
Mixer100 has two alternate symbols: MixingProcess_Detailed (full P&ID) and SwitchedMixingProcess (simplified view). Selecting an alternate swaps the visual representation while keeping the same data bindings.
Note: The Select Alternate Symbol feature cannot be used with symbols that originate from the Graphics tab, because those symbols are not part of any automation object. It only works with symbols that are natively hosted by or linked to an automation object.

When you select an alternate symbol in a non-frame window, if the new symbol is a different size than the original, you will be prompted with a message asking whether to retain the current size. Choosing Yes keeps the current dimensions, while choosing No updates the size to match the new symbol.

Summary

This lesson established the critical relationships between symbols and automation objects. You learned that linked symbols live on the Graphics tab and can be shared across multiple objects with changes propagating everywhere, while added symbols are natively hosted by an object and are deleted when the object is deleted. You explored how relative references like Me and MyArea allow a single graphic to display context-appropriate data at runtime. And you saw how the Select Alternate Symbol feature lets you swap between different visual representations of the same equipment. These concepts are the foundation for building scalable, maintainable HMI graphics in AVEVA System Platform.

Quiz — Test Your Understanding

Q1: What is the fundamental difference between a linked symbol and an added symbol?

A linked symbol lives independently on the Graphics tab and is associated with an object via a link — changes propagate to all linked objects. An added symbol is natively hosted by the object and exists only within that object — it is deleted when the object is deleted.

Q2: Why are relative references important when creating graphics for reuse?

Relative references (Me, MyArea, MyContainer, etc.) allow a single graphic to display different data depending on which object instance it is embedded in. This eliminates the need to create separate graphics for each piece of equipment — one graphic template serves all instances.

Q3: What happens when you delete an automation object that has added symbols versus linked symbols?

Deleting an object with added symbols also deletes those symbols, since they exist only within the object. Deleting an object with linked symbols does NOT delete the symbols, because they still exist independently on the Graphics tab.

Q4: What does the Select Alternate Symbol feature do, and when can it NOT be used?

Select Alternate Symbol replaces an embedded symbol with another symbol belonging to the same instance, allowing you to swap between different visual representations (e.g., detailed vs. simplified view). It cannot be used with symbols that originate from the Graphics tab — only with symbols that are part of an automation object.

Q5: When embedding a symbol from a template in WindowMaker, why are you prompted for a name?

Because the software will attempt to create a new instance in the Galaxy from that template. The name must be unique. If you wanted to use an existing instance's symbol instead, you should select the Instances button in the Galaxy Browser rather than Templates.
