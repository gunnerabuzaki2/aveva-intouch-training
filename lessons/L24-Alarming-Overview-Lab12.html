Lesson 24: Alarming Overview + Lab 12 – Adding Alarm Notifications to Graphics

Source: AVEVA InTouch for System Platform 2023 Training Manual, Module 5 – Alarms and Events Visualization, Section 1 (pages 293–320)

This lesson covers the fundamentals of alarms and events in System Platform, explains how the Platform acts as an Alarm Provider, introduces Alarm Border animation and aggregation, and walks you through enabling alarm notifications in a graphic.

24.1 Alarms vs. Events

Reference: Module 5, Section 1 (pages 295–296)

System Platform's alarm and event capabilities automate the detection, notification, historization, and viewing of process conditions. However, alarms and events are fundamentally different concepts, and the system treats them separately.

An alarm represents an abnormal condition that requires immediate operator attention. Examples include a temperature exceeding a high limit, or a pump that should be running but has stopped. An event is simply the occurrence of a condition at a specific point in time — for example, a system engineer deploying a new AutomationObject, or a Platform coming back online after a shutdown.

Key Concept: Not everything that gets logged is an alarm or event. Configuration actions within the Galaxy Repository (like imports or checkouts) and messages sent to the system logger (SMC Logger) are not alarms or events. Logger messages may be generated by software events, but they are not events themselves.

The Platform as an Alarm Provider

A WinPlatform acts as a single Alarm Provider, feeding alarms into the InTouch Distributed Alarm subsystem. Each Platform's AutomationObject has a Boolean checkbox that controls whether it subscribes to Areas for alarm and event reporting. The Platform routes all alarms and events from its subscribed Areas to the distributed alarming infrastructure.

The alarm fields generated by AutomationObjects are mapped to InTouch Distributed Alarm fields as follows:

InTouch as an Alarm Consumer

The InTouch Alarm Client subscribes to alarms by specifying a node name and provider name. For System Platform, one provider exists per Platform (node). The client can also filter subscriptions to specific Alarm Areas using query filters. Operators can acknowledge unacknowledged alarms directly from InTouch — the Platform forwards the acknowledgment to the target AutomationObject's acknowledge attribute. Similarly, alarms can be enabled or disabled from InTouch by writing to the AlarmMode attribute.

24.2 Alarm Queries

Reference: Module 5, Section 1 (pages 298–299)

Alarm queries are the mechanism that Alarm Clients use to retrieve real-time alarm information and event reports from the Galaxy. The fully qualified syntax for querying a single alarm within an object is:

\\nodename\Galaxy!area!object.attribute

If the WinPlatform is on the local computer, you can omit the node name:

\Galaxy!area!object.attribute

Alarm QueryReturns
\\nodename\Galaxy!areaAll alarms/events from the area, all hosted objects, and all subareas
\\nodename\Galaxy!area!objectAll alarms/events from the object itself
\\nodename\Galaxy!winplatformAll alarms/events from the WinPlatform object
\\nodename\Galaxy!engineAll alarms/events from the engine object (AppEngine or ViewEngine)
\\nodename\Galaxy!dioAll alarms/events from the device integration object

The asterisk (*) acts as a wildcard in the object.attribute portion of the query, but only one wildcard is allowed per query. Multiple queries can be submitted at once — for example, to retrieve alarms from two mutually exclusive areas: \Galaxy!Area1 \Galaxy!Area2.

24.3 Alarm Border Animation

Reference: Module 5, Section 1 (pages 300–303)

Alarm Border animation is a visual feature that draws a highly visible colored border around a graphic element when an alarm condition occurs. The border's color and fill pattern communicate the alarm's severity and current state, allowing plant operators to instantly recognize alarm conditions without needing to check a separate alarm list.

Key Concept: Alarm Border animation applies to all symbol types except embedded symbols and nested groups. It can be applied to individual graphic elements or grouped elements.

Alarm Border Behavior

The border appearance changes based on the alarm lifecycle:

Alarm StateBorder Behavior
Alarm activatesBorder appears with blinking, matched to severity color
Alarm acknowledgedBorder remains visible but stops blinking
Returns to normal (not acknowledged)Border remains in "Return to Normal" visual style, no blinking
Returns to normal + acknowledgedBorder disappears entirely
Alarm suppressedBorder remains in "Suppressed" style; indicator icon hidden
Important: Alarm Border animation is not used for alarms in an inhibited or shelved state. These alarm modes do not produce a visual border.

Configuring Alarm Border

Alarm Border can be configured for either aggregated alarms (using Standard Alarm-Urgency References) or individual alarms (using Customized Alarm-Urgency References). For aggregated alarms, you enter an object name and the system maps to the AlarmMostUrgentInAlarm, AlarmMostUrgentAcked, AlarmMostUrgentMode, AlarmMostUrgentSeverity, and AlarmMostUrgentShelved attributes automatically.

The alarm border's appearance is defined by Element Styles that cannot be reassigned — only the Outline properties (line pattern, weight, color) can be modified. Each combination of severity (1–4) and state (Unacknowledged, Acknowledged, Return to Normal) has a dedicated Element Style.

24.4 Alarm Aggregation and Shelving

Reference: Module 5, Section 1 (pages 304–306)

Alarm aggregation provides an efficient way to understand the alarm status of an object and all its descendants at a glance. Five read-only attributes aggregate alarm information: AlarmCntsBySeverity (counts by severity level), AlarmMostUrgentAcked, AlarmMostUrgentInAlarm, AlarmMostUrgentMode, and AlarmMostUrgentSeverity. These attributes roll up alarm counts from children to parents, making it possible to follow a trail from one containment level to the next to find the root cause of complex alarms.

Alarm aggregation is enabled on Area objects by checking the Aggregate Alarms checkbox in the Area object configuration editor. Only alarms in the UNACK_ALM, ACK_ALM, or UNACK_RTN states are aggregated — alarms in the ACK_RTN state are not.

Alarm shelving temporarily hides alarms from displays for a fixed period while continuing to historize them. Shelving differs from silencing in important ways: shelved alarms have a built-in timeout (they auto-unshelve when the period expires), shelving applies only to individual alarms (not entire areas), and the system enforces role-based limits on who can shelve, which severity levels can be shelved, and how many alarms a user can shelve at once.

24.5 Lab 12 – Adding Alarm Notifications to Graphics

Reference: Module 5, Lab 12 (pages 307–311)

Lab Objectives:
  • Enable the built-in Alarm Border feature in the Level_Detailed graphic
  • Observe Alarm Border animations at runtime

Step 1: Enable AlarmBorder on Level_Detailed

  1. In the IDE, on the Graphics tab, in the Training toolset, open Level_Detailed for editing.
  2. On the canvas, right-click LevelMeter and select Custom Properties. Notice that no alarm properties exist yet.
  3. Click OK to close Edit Custom Properties.
  4. On the Properties tab, set SymbolMode to Advanced and AlarmBorder to True.
Custom Properties for LevelMeter before enabling AlarmBorder — only standard properties are present (page 308)
Setting AlarmBorder to True and SymbolMode to Advanced in the Wizard Options (page 308)
  1. Open Edit Custom Properties again — alarm properties have now been automatically added.
  2. Select the AlarmMostUrgentShelved property.
  3. In Default Value, enter Me.PV.AlarmMostUrgentShelved.
  4. Click OK to close Edit Custom Properties.
  5. Save and close, then check in the graphic.
After enabling AlarmBorder, the Custom Properties dialog now includes alarm aggregation attributes (page 309)
Configuring the AlarmMostUrgentShelved property to reference the PV's shelved status (page 309)

Step 2: Test at Runtime

  1. In WindowMaker, ensure the ContentDisplay window is open, then click Runtime.
  2. In the ContentDisplay window, observe the Alarm Border animations on the mixer level meters. The borders change color and flash to indicate alarm severity.
  3. In the Navigation window, click Line 1 to navigate to the Line1 area. Observe the Alarm Border animations on the Line1 mixers.
  4. Click Development! to return to WindowMaker.

Knowledge Check

1. What is the difference between an alarm and an event in System Platform?

Click to reveal answer

An alarm is an abnormal condition requiring immediate operator attention (e.g., high temperature, stopped pump). An event is an occurrence at a point in time that does not necessarily require attention (e.g., deploying a new object, a Platform coming back online).

2. What must you set on a symbol's properties to enable Alarm Border animation, and what property controls the shelved status?

Click to reveal answer

You must set SymbolMode to Advanced and AlarmBorder to True. The AlarmMostUrgentShelved custom property is configured with the expression Me.PV.AlarmMostUrgentShelved to expose shelved status.

3. Which alarm states are included in alarm aggregation, and which are excluded?

Click to reveal answer

Alarms in UNACK_ALM, ACK_ALM, and UNACK_RTN states are aggregated. Alarms in the ACK_RTN state are NOT aggregated. Silenced alarms ARE aggregated even though they may not appear in runtime clients.

4. What are three key differences between shelving and silencing an alarm?

Click to reveal answer

1) Shelved alarms have a built-in timeout and auto-unshelve; silenced alarms do not. 2) Shelving applies to individual alarms only (not entire areas); it cannot propagate through the hierarchy. 3) The system enforces role-based limits on shelving permissions, severity levels, and maximum count per user.
