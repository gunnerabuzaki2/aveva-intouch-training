Understanding the Difference Between Alarms and Events

When a production line is running smoothly, operators rarely think about notifications. But the moment a tank level climbs too high, a motor stops unexpectedly, or a temperature drifts beyond its safe range, the operator needs to know — instantly, clearly, and with enough context to act. This is where alarms and events come in, and understanding the distinction between the two is foundational to building effective visualization in System Platform.

Not all runtime notifications carry the same weight. System Platform draws a sharp line between alarms, which signal abnormal conditions that demand an operator's immediate attention, and events, which are informational records of things that happened at a point in time. A pump that should be running but has stopped generates an alarm. A configuration change that was committed by an engineer generates an event. The difference matters because alarms have severity levels and require acknowledgment, while events are simply logged for reference.

Key Concept: Alarms vs. Events
Alarms represent abnormal conditions requiring immediate operator attention — such as a temperature exceeding a high limit, a pump that should be running but is stopped, or a tank level dropping below its safe threshold. Alarms carry severity levels (from Low to Critical) and require acknowledgment. Events, by contrast, are occurrences logged at a specific point in time — such as a configuration change, a platform coming back online, or a data value changing beyond a tracking threshold. Events are informational and do not require acknowledgment. Configuration actions within the Galaxy Repository (like imports or checkouts) and messages sent to the SMC Logger are not considered events, even though they may log messages.

How System Platform Routes Alarms: The Provider/Consumer Model

Behind the scenes, System Platform uses a Provider/Consumer architecture to deliver alarms from the process to the operator's screen. Think of it as a supply chain: the automation objects in your Galaxy generate alarm conditions, the Platform object collects and routes those alarms, and InTouch subscribes to them for display. This separation of concerns means that alarm generation and alarm visualization are decoupled — you can change how alarms are displayed without affecting how they are generated, and vice versa.

A Platform object acts as an Alarm Provider, collecting alarms from all the AutomationObjects it hosts and routing them into the distributed alarming infrastructure. InTouch, running on the operator's workstation, acts as an Alarm Consumer, subscribing to alarms from specific Platforms and displaying them in whatever form the engineer has configured. The Platform is responsible for routing all alarms and events for subscribed Areas, and each alarm generated by Application Server carries a rich set of fields — timestamp, tagname, name, comment, area, type, state, priority, value, limit, category, and class — that map directly to fields in InTouch's distributed alarm subsystem.

This architecture gives you remarkable flexibility. The Alarm Client in InTouch can be configured to subscribe to alarms from specific Platforms (identified by node name and provider), filter alarms by Area using query syntax, acknowledge unacknowledged alarms, and even enable or disable alarming on specific AutomationObjects. The query syntax follows a hierarchical pattern: \\nodename\Galaxy!area!object.attribute. You can use wildcards — for example, \\nodename\Galaxy!area!*.attribute — to match all objects in an area, and you can submit multiple queries simultaneously from different nodes.

Alarm Border Animation: Visual Feedback at a Glance

One of the most intuitive ways to draw an operator's attention to an alarm is through the Alarm Border animation. Rather than requiring the operator to scan a table of text, this feature draws a colored, blinking border directly around the graphic element that is in an alarm state. The color and fill pattern of the border indicate the alarm's severity and its current acknowledgment state, and an indicator icon in the top-left corner shows the severity level (1 through 4) or the current alarm mode (Shelved, Silenced, or Disabled).

The Alarm Border communicates state through a simple visual language that operators can learn in minutes. A blinking border means the alarm just occurred and has not yet been acknowledged. A solid border without blinking means the alarm has been acknowledged but the condition is still active. When the value returns to normal but the operator hasn't acknowledged the alarm yet, a special return-to-normal style is shown. Once the alarm is acknowledged and the value returns to normal, the border disappears entirely. If the operator has suppressed the alarm, a distinct suppressed style is displayed.

Key Concept: Alarm Border Behavior States
The Alarm Border animation cycles through five distinct visual states:
  • Blinking border — The alarm just occurred and the operator has not yet acknowledged it. This is the highest-urgency visual state.
  • Solid border (no blink) — The alarm has been acknowledged but the condition is still active. The operator knows about it and is presumably working on it.
  • Return-to-Normal style — The process value has returned to normal range, but the alarm has not yet been acknowledged. This is a transitional state.
  • No border — The alarm has been acknowledged and the value has returned to normal. The situation is resolved.
  • Suppressed style — The operator has actively suppressed the alarm, typically during maintenance or planned work.

The Alarm Border animation can be applied to all symbol types (with the exception of embedded symbols and nested groups) and it supports aggregated alarms — meaning it will show the highest current alarm state across all attributes on a symbol. This is particularly powerful when applied to complex symbols like a mixer, where multiple attributes (level, temperature, agitator speed) can each have independent alarms.

The appearance of each alarm state is controlled by AlarmBorder Element Styles. Every combination of severity (1 through 4) and state (Unacknowledged, Acknowledged, Return to Normal) has a dedicated Element Style whose Outline properties — line pattern, weight, and color — can be customized to match your facility's visual standards.

Tip: You can customize the Alarm Border indicator icons by replacing the default images in the XML file located in the global data cache. Supported formats include .bmp, .gif, .jpg, .png, .ico, and .emf. This lets you match the indicator icons to your organization's alarm management standards.

Alarm Aggregation and Severity Roll-Up

In a facility with hundreds of automation objects, operators cannot afford to check each one individually for alarm states. Alarm aggregation solves this problem by automatically rolling up alarm counts from child objects to parent objects in the Galaxy hierarchy. If Mixer100 has 12 Critical alarms and Mixer200 has another 12 Critical alarms, the Line1 node will show 24 Critical alarms. This hierarchical roll-up continues all the way up to the Production or Plant level, giving operators an instant summary of where problems are concentrated.

Alarms are included in aggregation when they are in one of three states: UNACK_ALM (unacknowledged and active), ACK_ALM (acknowledged and still active), or UNACK_RTN (unacknowledged but returned to normal). Importantly, alarms in the ACK_RTN state — acknowledged and returned to normal — are not aggregated, because they represent resolved situations. Silenced alarms are still included in aggregation counts, even though they do not appear in runtime clients.

Alarm Shelving: Temporary Suppression with a Safety Net

During planned maintenance or system modifications, operators often need to temporarily suppress specific alarms without permanently disabling them. Alarm shelving provides exactly this capability, and it differs from alarm silencing in several important ways. Shelving has a built-in timeout — alarms automatically return to visibility when the shelving period expires. This is a critical safety feature: it ensures that alarms are not forgotten after maintenance is complete.

Shelving applies to individual alarms only, not to entire areas or hierarchies. Role-based permissions control who can shelve alarms, which severity levels they are allowed to shelve, and how many alarms they can shelve simultaneously. The system also maintains a complete audit trail — recording who shelved the alarm, from which workstation, the reason for shelving, and when it expires.

Pitfall: Alarm shelving must be enabled at the Area level before individual alarms can be shelved. If you find that you cannot shelve an alarm at runtime, the first place to check is the Area object configuration. This is a common source of confusion during initial setup.

Lab 12 — Enabling Alarm Border Notifications

Lab Objectives:
  • Enable the Alarm Border feature on the Level_Detailed graphic
  • Configure the Shelved custom property to expose alarm shelving state
  • Observe Alarm Border animations at runtime

Step 1: Enable the AlarmBorder Wizard Option

Open the Level_Detailed symbol for editing. Select the LevelMeter element, right-click, and choose Custom Properties. You will notice that there are no alarm-related properties yet — this is because the AlarmBorder feature has not been enabled. Close the dialog and return to the Properties tab, where you will set two critical properties:

PropertyValue
SymbolModeAdvanced
AlarmBorderTrue

Setting AlarmBorder to True is the switch that activates the entire alarm border subsystem for this symbol. It tells InTouch to monitor alarm states for this element and render the appropriate visual feedback.

The AlarmBorder option is set to True in the LevelMeter Wizard Options panel.

Step 2: Configure the Shelved Property

Open Edit Custom Properties again. You will now see that additional alarm properties have been automatically added as a result of enabling AlarmBorder. Select AlarmMostUrgentShelved and set its Default Value to Me.PV.AlarmMostUrgentShelved. This binding connects the symbol's shelved state to the process variable's alarm system, enabling the shelving indicator to display correctly at runtime. Save, close, and check in the graphic.

After enabling AlarmBorder, six new alarm properties are automatically added to the custom properties list.

Step 3: Test in Runtime

Open the ContentDisplay window and click Runtime to see your changes in action. Observe the Alarm Border animations on the mixer's level sensor — you will see the border change color and blink based on the alarm severity as the level value changes. Navigate to Line 1 to see Alarm Borders on all four mixers simultaneously, giving you a visual overview of alarm states across the entire production line.

Quiz — Test Your Understanding

Q1: What is the key difference between an alarm and an event in System Platform?

Alarms represent abnormal conditions requiring immediate operator attention (e.g., high temperature, stopped pump) and have severity levels and acknowledgment states. Events are informational occurrences logged at a point in time (e.g., configuration changes, platform recovery) that do not require acknowledgment.

Q2: What happens visually when an operator acknowledges an active alarm?

The Alarm Border animation changes from blinking to a solid (non-blinking) border. The border remains visible until the alarm value returns to normal and the operator acknowledges it, at which point the border disappears entirely.

Q3: How does shelving differ from silencing an alarm?

Shelved alarms have a built-in timeout and automatically return after the configured period. Shelving applies to individual alarms only (not entire areas), enforces role-based permissions, and tracks audit information (who, where, reason, duration). Silenced alarms have no automatic timeout.

Q4: Which alarm states are included in alarm aggregation?

UNACK_ALM, ACK_ALM, and UNACK_RTN states are aggregated. Alarms in ACK_RTN are not aggregated. Silenced alarms are aggregated even though they don't appear in runtime clients.
