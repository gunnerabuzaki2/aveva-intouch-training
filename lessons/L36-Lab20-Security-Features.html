Lesson 36: Lab 20 – Security Features in Graphics

Module 7 — Security | Pages 473–496

36.1 Introduction

In this lab, you will test InTouch's Secured Write and Verified Write functions by writing to attributes configured with these security classifications. To do this, you will modify the mixer agitator valves with security classifications, create new security roles and users representing operator and supervisor accounts, and then verify that the permission model works correctly at runtime.

Objectives — Upon completion, you will be able to:
  • Use the Secured Write and Verified Write functions in InTouch
  • Enable the Operate and Can Verify Write permissions

36.2 Background: Signed Writes Theory

36.2.1 Assigning Users to Roles

After creating users and roles in the Galaxy Security settings, you assign them on the Users tab. Every user in the Galaxy and their role assignments appear there. By default, each new user is automatically associated with the Default role — but not the Administrator role. Users can be assigned to more than one role at a time.

Best Practice: Limit the Default role's permissions to what every user should have, then add users to additional roles with elevated permissions for specific parts of Application Server. This gives you a clean baseline security posture.

The Administrator user can log on under any authentication mode (except when security is disabled). When logged on as Administrator on the Galaxy Repository node, you can change any Galaxy user's password without providing the old password.

Security settings Users tab showing user list (Administrator, DefaultUser, TUser) with role assignments

36.2.2 OS Group-Based Security and Access Levels

If you use OS Group-based Authentication Mode, each node within the Galaxy must have the same OS Users, Groups, and user-group mappings to ensure consistent access levels across all nodes.

Important: A newly-added user who has never logged on to a remote node cannot write to attributes on that node, even with sufficient runtime operational permissions. You must log on to the remote node at least once first.

The Roles page includes an Access Level column — an InTouch-specific function ranging from 0 (minimum) to 9999 (maximum). When a user is assigned to multiple roles with different access levels, the higher value is passed to InTouch.

36.2.3 Secured and Verified Writes

Secured Write and Verified Write are security classifications you assign to configurable attributes. They require authentication to perform runtime writes — meaning any attempt to change the attribute's value (via script, animation graphic, or the SignedWrite() function) triggers a security dialog.

Secured Write requires a single authenticated user. Verified Write requires two: an operator who initiates the write and a verifier who must have the "Can Verify Writes" permission.

Operators can write to these attributes even when another user is logged on — this does not affect the other session. However, the operator must have "Can Modify 'Operate' Attributes," and for Verified Write, the verifier needs "Can Verify Writes."

Note: Secured and Verified Writes only work when security is enabled on both the Galaxy and InTouch. If either is set to None, these features will not function.

36.2.4 The Dialog Boxes and Audit Trail

At runtime, the Secured Write or Verified Write dialog appears when an operator attempts to write to a protected attribute. The dialog displays the attribute, the value, and configurable input fields. You can use the SignedWrite() script function to configure these inputs — but only for symbol scripts, not application object scripts.

The dialog includes a Reason Description area (explaining the attribute's purpose), a Predefined Comment list for consistent auditing, and an Editable Comment box. These fields appear only in InTouch WindowViewer, not in Object Viewer.

Secured Write dialog for Agitator_001.Cmd — Value: true, Comment: start the motor, authenticated as operator1

Following a write, a security Event is logged with the signee name, verifier name, write type, comment, reason description, and field attribute description.

Side-by-side comparison: Secured Write (supervisor1, single-user auth) and Verified Write (dual-operator/verifier auth for Speed.SP value 25.0)

36.3 Lab Steps: Creating Roles and Users

First, close WindowMaker and check in the $MixerView template. Security configuration requires all objects to be checked in — the Find tool helps with this.

In the IDE, navigate to Galaxy > Configure > Security. On the Roles tab, select the Default role. Under Operational permissions, uncheck Default, expand it, and under Alarms check "Can Acknowledge Alarms." This gives the base role alarm acknowledgment while keeping other permissions restricted.

Security settings Roles tab — Default role selected, showing IDE Permissions and Operational permissions with Can Acknowledge Alarms checked

Create an Operators role with Access Level 1000. Under Operational permissions, check "Can Modify 'Operate' Attributes." This gives operators the ability to modify operational attributes — nothing more.

Operators role permissions showing Can Modify 'Operate' Attributes checked (access level 1000)

Create a Supervisors role with Access Level 5000. Enable Can Shelve Alarms, Can Modify Alarm Modes, Can Modify Plant States, and Can Verify Writes. This role has significantly more power than Operators.

Supervisors role (access level 5000) with Can Shelve Alarms, Can Modify Alarm Modes, Can Modify Plant States, and Can Verify Writes all checked

On the Users tab, create three users: Operator1 (Operator One, Operators role, password: ww), Operator2 (Operator Two, Operators role, password: ww), and Supervisor1 (Supervisor One, Supervisors role, password: ww).

Operator1 (Operator One) assigned to Default and Operators roles with access levels 0 and 1000
Change Password dialog for Operator1 — New password and Confirm fields filled with masked characters

36.4 Configuring Attributes with Security Classifications

Open the $Mixer.Agitator template. Select the Cmd attribute and assign it a SecuredWrite classification via the Security Classification icon. Lock the initial value — this propagates the classification to all instances during deployment.

Select the Speed.SP attribute and assign it VerifiedWrite. Lock it as well. Save, close, and check in the template.

$Mixer.Agitator attribute editor — Security Classification dropdown showing FreeAccess, Operate, SecuredWrite, VerifiedWrite, Tune, Configure, ViewOnly

Reopen the template, select Cmd, and unlock the initial value. This allows runtime changes while preserving the security classification. Repeat for Speed.SP. Save, close, and check in once more.

Why lock then unlock? Locking first propagates the classification to all derived instances during deployment. Unlocking afterward allows runtime changes while keeping the classification intact.

36.5 Redeploying the Agitator Objects

In the Derivation view, expand $UserDefined > $Motor > $Agitator > $Mixer.Agitator. All eight instances show pending changes. Select them all, right-click, and Deploy with default settings.

Derivation tree view with Deploy option highlighted for the eight agitator instances

36.6 Testing in Runtime

36.6.1 Testing Alarm Mode Permissions

Open $MixerView and start Runtime. With no user logged in, navigate to Alarm Overview and attempt to disable alarms on the Production node. Click Apply — nothing happens, because disabling alarms requires the "Can Modify Alarm Modes" permission.

Log on as Supervisor1. Navigate to Alarm Overview, set the Production node Mode to Disable, and click Apply. Alarms are now disabled across the entire hierarchy. Re-enable them and close the overview.

AlarmAggregationOverview — Disable option being selected on Production node with alarm counts visible

36.6.2 Testing Secured Write

Open Agitator_001's command panel and click Start. The Secured Write dialog appears with supervisor1 as the current user. Enter a comment and password, then click OK. The write fails — Supervisor1 lacks the "Can Modify 'Operate' Attributes" permission.

Secured Write dialog — supervisor1 with password entered, Comment: Attempt to start the motor

Log off, then log on as Operator1. Open Agitator_001 and click Start. Enter "Start motor" and the password, then click OK. The motor starts — Operator1 has the Operate permission.

36.6.3 Testing Verified Write

Open Agitator_001 again and change the Speed setpoint to 25. The Verified Write dialog appears with Operator and Verifier sections. Enter Operator1's credentials, then Operator2's credentials. The write fails — Operator2 lacks "Can Verify Writes."

Verified Write dialog — Value: 25.0, Operator: operator1, Verifier section awaiting credentials

Try again using Supervisor1 as the verifier. Enter Supervisor1's credentials in the Verifier section and click OK. The write succeeds — the Speed setpoint updates to 25 rpm.

Agitator_001 control panel — Speed setpoint updated to 25.00 rpm after successful verified write

36.6.4 Reviewing the Audit Log

Navigate to Historical Alarms and click Historical Events. The Alarm Client shows entries for both denied and successful write attempts, providing a complete audit trail of who attempted what and whether it was authorized.

AlarmHistDisplay — Historical Events showing Secured Write access denied, success, Verified Write access denied, and success

Click Development! to return to WindowMaker and end the lab.

36.7 Quiz — Test Your Understanding

Quiz

Q1: What is the difference between Secured Write and Verified Write?

Secured Write requires a single authenticated user to confirm the write. Verified Write requires two users: an operator who initiates and a verifier who confirms, where the verifier must have the "Can Verify Writes" permission.

Q2: Why must security be enabled on both the Galaxy and InTouch for these features to work?

The Galaxy defines the roles, users, and permissions, while InTouch provides the runtime authentication dialogs. If either is set to None, the authentication infrastructure is incomplete and Secured/Verified Writes cannot function.

Q3: Why do you lock an attribute's initial value before unlocking it?

Locking first propagates the security classification change to all derived instances during deployment. Unlocking afterward allows runtime changes while preserving the classification assignment.

Q4: What happens when a user without "Can Modify Alarm Modes" tries to disable alarms?

The operation silently fails — alarms are not disabled. The Apply button may appear, but clicking it has no effect without the required permission.

Q5: Where can you find a record of all secured and verified write attempts?

In the Historical Alarms display under Historical Events. Each attempt is logged as a security Event including the signee, verifier, write type, comment, and reason description.
