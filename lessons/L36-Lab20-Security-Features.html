Lesson 36: Lab 20 – Security Features in Graphics

Source: AVEVA InTouch for System Platform 2023 Training Manual, Module 7, Section 2 + Lab 21 (pages 473–500)

Completing Lab 20 with user-role assignments, then covering Signed Writes concepts and Lab 21 — configuring Secured Write and Verified Write security classifications on agitator attributes and testing them at runtime with multiple user accounts.

36.1 Assigning Users to Roles

Reference: Module 7, Section 2 (pages 473–474)

After creating users and roles, the next step is assigning users to their appropriate roles. The Users page in the Security editor shows all users in the Galaxy and the roles they are assigned to. By default, every new user is associated with the Default role but not the Administrator role. Users can be assigned to more than one role simultaneously.

Key Concept: All users are automatically assigned to the Default role in addition to any newly created roles. The best approach to managing permissions is to restrict the Default role to basic access, then add users to other roles that grant the specific permissions they need.

The Administrator user can log on under any authentication mode (except when security is disabled). When logged on as Administrator on the Galaxy Repository node, you can change any Galaxy user's password without providing the old password.

The Users tab showing authorized users and their role assignments (page 473)

36.2 OS Group-Based Security and Access Levels

Reference: Module 7, Section 2 (pages 474)

When using the OS Group-based Authentication Mode, each node within the Galaxy must have the same OS Users, Groups, and user-group mappings to ensure consistent access across the deployment. This is because the system maps Windows groups directly to security roles in the Galaxy.

For InTouch access levels, the Roles page includes an Access Level column. Access levels are an InTouch-specific function — in the security model, they only map to InTouch values and have no prioritizing characteristics. The maximum value is 9999 and the minimum is 0. If a user is assigned to multiple roles with different access levels, the higher value is passed to InTouch.

Important: A newly added user who has never logged on to a remote node cannot write to attributes on that node, even if they have sufficient runtime permissions. The user must log on to the remote node at least once to enable remote writing capabilities.

36.3 Secured and Verified Writes Overview

Reference: Module 7, Section 2 (pages 474–476)

Assign Secured Write or Verified Write security classification to a configurable attribute when you need to require authentication before runtime writes can occur. These classifications affect any action that changes the attribute's value — whether through script assignments (like A=B), animation interactions (user inputs, sliders, up/down buttons), or the SignedWrite() script function.

Operators can write to Secured or Verified Write attributes even when another user is logged on — this does not affect the other user's session. However, the operator must have the Can Modify "Operate" Attributes permission, and for Verified Write, the verifier must have Can Verify Writes permission.

The Secured Write dialog — requires the operator to authenticate with their password before completing the write (page 475)

Dialog Box Features

At runtime, the Secured Write or Verified Write dialog boxes appear when an operator attempts to write to a protected attribute. These dialogs include several configurable elements:

  • Reason Description — specific to each attribute, explaining the purpose of the attribute and the impact of changing it
  • Predefined Comment List — enables the operator to select from a predefined list of comments for consistent auditing
  • Editable Comment Box — allows free-form operator comments (displayed only in WindowViewer, not Object Viewer)
Key Concept: The SignedWrite() script function can only be used in symbol scripts for attributes configured with Secured Write or Verified Write — not for application object scripts.

Logged Information

After every Secured or Verified Write, a security event is written to the event log that includes the signee name, verifier name (if any), write type, comment, reason description, and the field attribute description. This creates a complete audit trail of all protected write operations.

Knowledge Check

1. In Lab 21, why did the Verified Write operation fail when Operator2 was used as the verifier?

Click to reveal answer

Operator2 only had the Operators role, which grants Can Modify "Operate" Attributes but does not include Can Verify Writes. The verifier must have the Can Verify Writes operational permission, which was only granted to the Supervisors role (assigned to Supervisor1).

2. What happens when a user without the Can Modify Alarm Modes permission tries to disable alarms through the AlarmAggregationOverview?

Click to reveal answer

The operation fails silently — the alarms are not disabled. This was demonstrated when no user was logged in and the Disable command was applied. Only users with the Can Modify Alarm Modes permission (like Supervisor1) can successfully change alarm modes.

3. What is the purpose of locking and then unlocking attribute initial values during the template configuration process?

Click to reveal answer

Locking the initial value propagates the security classification change to all derived instances when deployed. After deployment, unlocking the initial value allows the attribute values to be changed at runtime by authorized users, while the security classification remains in effect.

4. Why must all objects be checked in before security configuration can proceed?

Click to reveal answer

Security configuration affects the entire Galaxy database. If objects are checked out (being edited by another user), the system cannot safely apply security changes that might affect those objects. All objects must be checked in to ensure a consistent and complete security update.
