Lesson 38: Runtime Customization + Lab 22

Module 7 — Security | Pages 511–526

38.1 Introduction

This lesson completes Lab 22 — Locking Down Your Application. You will test the inactivity settings from L37 in runtime, then create a MenuBarCover window to block access to the WindowViewer menu bar, and finally implement the EnableDisableKeys() script function to restrict keyboard shortcuts. Together, these techniques lock down the application against unauthorized access.

Objectives — Upon completion, you will be able to:
  • Test inactivity warning and timeout settings in runtime
  • Use negative window dimensions to cover the menu bar
  • Use the EnableDisableKeys() script function
  • Create DataChange and Condition scripts

38.2 Testing Inactivity in Runtime

Switch to WindowViewer and log on as TUser (password: ww). Wait 15 seconds — the Inactivity window automatically opens with a blinking warning message. TUser is still logged in at this point. Click the mouse or press a key — the $InactivityWarning tag is cleared, triggering the On False condition, and the Inactivity window is hidden.

InTouch runtime — red Inactivity Warning with yellow text "Click mouse or use keyboard to prevent system log off"

Now wait 30 seconds without interacting. The timeout expires and you are automatically logged off. The Inactivity window shows "You have been logged off!" and the Logon panel shows User: None with Access Level: 0.

InTouch runtime — Inactivity Timeout: "You have been logged off!" after the 30-second timeout

Return to WindowMaker and reset the inactivity settings (Warning: 0, Timeout: 0) to disable them for the remaining lab steps.

38.5 Understanding EnableDisableKeys()

The EnableDisableKeys() script function takes three parameters — one each for the Alt, Esc, and Win keys — where 0 enables and 1 disables:

CallEffect
EnableDisableKeys(0,0,0)Enables all three keys
EnableDisableKeys(1,1,1)Disables all three keys
EnableDisableKeys(0,0,1)Disables only the Win key
Dynamic vs. Static: WindowViewer offers two methods to disable keys. The static method (Window tab checkboxes) permanently disables keys for all users. The EnableDisableKeys() script function is dynamic — it adapts based on which user is logged in, making it far more flexible for real-world applications.

38.6 Creating the Application On Startup Script

Create an Application script with Condition Type On Startup. This script runs once when WindowViewer starts and immediately disables the Alt, Esc, and Win keys. This ensures the application is locked down before any user logs in.

Application Script editor — On Startup trigger with EnableDisableKeys(1,1,1) to disable keys at startup

38.7 Updating Home Windows and Testing

In WindowViewer settings, update the Startup tab to include MenuBarCover in the Home Windows. This ensures the cover window is displayed when WindowViewer starts.

In runtime, verify the following behavior:

  • No user logged in: Menu bar is covered, Alt+F doesn't open the File menu, Ctrl+Shift+Esc doesn't open Task Manager, and the Win key doesn't open the Start menu.
  • Operator1 logged in: Menu bar is still covered (access level 1000 < 5000).
  • Administrator logged in: Menu bar is visible, and all keyboard shortcuts work normally (access level 9999 > 5000).
WindowViewer runtime — menu bar covered for non-admin user (Access Level: 0)

38.8 Quiz — Test Your Understanding

Quiz

Q1: Why is a negative Y position used for the MenuBarCover window?

Setting Y to -20 positions the window above the visible area at the top of the screen, precisely covering the WindowViewer menu bar. Combined with a narrow 20-pixel height, this blocks only the menu bar without affecting other UI elements.

Q2: What is the difference between the static key disable method and EnableDisableKeys()?

The static method (Window tab checkboxes) permanently disables keys for all users at runtime — even administrators. EnableDisableKeys() is dynamic: it can adapt based on the logged-in user's access level, allowing administrators full access while restricting operators.

Q3: Why is an On Startup script used to disable keys?

The On Startup script ensures the application is locked down before any user logs in. Without it, there would be a brief window after WindowViewer starts where keyboard shortcuts are available to anyone, including unauthorized users.

Q4: What access level threshold determines whether the menu bar is shown or hidden?

The threshold is 5000. Users with an access level greater than 5000 (like Administrator at 9999) see the menu bar and have full keyboard access. Users at or below 5000 (like Operators at 1000) have the menu bar covered and keys disabled.
