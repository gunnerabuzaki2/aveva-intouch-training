Lesson 38: Lab 22 – Locking Down Your Application

Source: AVEVA InTouch for System Platform 2023 Training Manual, Module 7, Section 3 + Lab 22 (pages 511–526)

Completing Lab 22 with condition scripts for inactivity windows, runtime testing, creating the MenuBarCover overlay, $AccessLevel Data Change scripts, and the EnableDisableKeys() function for keyboard lockdown.

38.1 Completing the Inactivity Condition Scripts

Reference: Module 7, Lab 22 (pages 511–512)

Continuing from Lesson 37, you will now complete the two condition scripts that control the Inactivity warning window.

Steps 31–32. In the Condition Scripts editor, configure the script for $InactivityWarning with the On True trigger. The script body opens the Inactivity window only when a user is logged in:

IF $Operator <> "None" THEN
    Show "Inactivity";
ENDIF;
The On True condition script — opens the Inactivity window when a user is logged in and the warning activates (page 511)
The Condition Type dropdown — On True fires once when the condition transitions to true (page 511)

Steps 33–34. Add a second condition for the same tag with the On False trigger. The script body hides the window when the user interacts:

Hide "Inactivity";

Click OK to close Condition Scripts. Expanding the Condition folder in the Scripts pane shows $InactivityWarning listed with both triggers.

The On False script — hides the Inactivity window when the user interacts (page 512)
The Scripts pane confirms the condition script is in place (page 512)

Reference: Module 7, Lab 22 (pages 516–517)

Now you will create a popup window designed to cover the WindowViewer menu bar, preventing unauthorized users from accessing menu options like File, Logic, and the Development! fast switch.

Steps 47–49. Create a new Frame window with these properties:

38.4 Creating the $AccessLevel Data Change Script

Reference: Module 7, Lab 22 (pages 518–519)

You need a script that shows or hides the MenuBarCover based on the current user's access level. This is done with a Data Change script on the $AccessLevel tag — it fires whenever the access level changes (i.e., when a user logs on or off).

Steps 50–53. In Scripts, double-click Data change. Configure the script:

{ Menu Access }
IF $AccessLevel > 5000 THEN
    { Allow menu access }
    Hide "MenuBarCover";
ELSE
    { Deny menu access }
    Show "MenuBarCover";
ENDIF;
The Data Change script — hides MenuBarCover for users with access level above 5000 (page 519)
Key Concept: The threshold of 5000 means that only users with the Supervisors role (access level 5000) or Administrator (access level 9999) can access the menu bar. Operators (access level 1000) and unauthenticated users (access level 0) are blocked.

38.7 Denying Access to Alt, Esc, and Win Keys

Reference: Module 7, Section 3 + Lab 22 (pages 498, 521–522)

The MenuBarCover handles mouse-based access, but keyboard shortcuts like Alt+F (File menu), Ctrl+Shift+Esc (Task Manager), and the Win key (Start menu) could still bypass the protection. The EnableDisableKeys() script function addresses this by dynamically disabling these keys based on the user's access level.

The EnableDisableKeys() Function

The function receives three Boolean parameters in this order: EnableDisableKeys(AltKey, EscKey, WinKey) where 0 enables the key and 1 disables it.

CallEffect
EnableDisableKeys(0, 0, 0)Enable all three keys
EnableDisableKeys(1, 1, 1)Disable all three keys
EnableDisableKeys(0, 0, 1)Enable Alt and Esc, disable only Win key

Steps 63–65. Open the existing $AccessLevel Data Change script and add EnableDisableKeys calls:

{ Menu Access }
IF $AccessLevel > 5000 THEN
    { Allow menu access }
    Hide "MenuBarCover";
    EnableDisableKeys( 0, 0, 0);
ELSE
    { Deny menu access }
    Show "MenuBarCover";
    EnableDisableKeys( 1, 1, 1);
ENDIF;
The updated Data Change script — now controls both menu bar visibility and keyboard shortcuts (page 522)

38.8 Application On Startup Script

Reference: Module 7, Lab 22 (pages 523–524)

When WindowViewer first starts, before any user logs on, the keys should be disabled by default. An Application On Startup script ensures this.

Steps 66–69. In Scripts, double-click Application. Set the Condition Type to On Startup. Enter the script body:

{ Disable the Alt, Esc, Win keys }
EnableDisableKeys( 1, 1, 1);
Selecting the On Startup trigger for the Application script (page 523)
The On Startup script — disables all three keys when WindowViewer starts (page 523)
Key Concept: The Application On Startup script runs once when WindowViewer starts. By disabling the keys immediately, you ensure that unauthorized users cannot use keyboard shortcuts before any user has logged in and the Data Change script has had a chance to evaluate.

38.9 Final Runtime Test — Keyboard Lockdown

Reference: Module 7, Lab 22 (pages 524–526)

Steps 70–75. Close WindowViewer to allow the On Startup script to run. Reopen WindowMaker, open the MenuBarCover window, and click Runtime. Verify no user is logged in. Test the keyboard shortcuts:

  • Press Alt+F — the File menu does not open
  • Press Ctrl+Shift+Esc — Task Manager does not open
  • Press the Win key — the Start menu does not open

Steps 76–80. Log in as Administrator (no password). Test the same shortcuts:

  • Press Alt+F — the File menu opens
  • Press Ctrl+Shift+Esc — Task Manager opens
  • Press the Win key — the Start menu opens
The File menu is accessible for the Administrator — keys are re-enabled (page 524)

Knowledge Check

1. Why does the $InactivityWarning condition script include an IF $Operator <> "None" THEN check before showing the Inactivity window?

Click to reveal answer

This check ensures the Inactivity window only appears when a user is actually logged in. If no user is logged in ($Operator equals "None"), showing the warning window would be meaningless — there's no active session to protect. This prevents the warning from appearing when the application starts or after the user has already been logged off.

2. How does the MenuBarCover window use negative dimensions to achieve its purpose?

Click to reveal answer

The MenuBarCover has a Y position of -20, which places the top of the window 20 pixels above the visible screen area. With a height of 20 pixels, the bottom edge of the window overlaps the WindowViewer menu bar. Since the window has no title bar, no frame, and is white (matching the menu bar background), it's invisible to the user but intercepts mouse clicks before they reach the actual menu bar.

3. What is the difference between the static keyboard disable method (Window Viewer settings) and the dynamic EnableDisableKeys() method?

Click to reveal answer

The static method uses checkboxes in WindowViewer Settings to permanently disable keys for all users at runtime — it cannot differentiate between users. The dynamic EnableDisableKeys() method uses scripts to disable or enable keys based on the current user's access level, allowing administrators to access all keys while operators are restricted.

4. Why is an Application On Startup script needed in addition to the $AccessLevel Data Change script?

Click to reveal answer

The Data Change script only fires when $AccessLevel changes. When WindowViewer first starts, no logon event has occurred yet, so the Data Change script hasn't run. The Application On Startup script ensures that the Alt, Esc, and Win keys are disabled from the very beginning, before any user has a chance to use keyboard shortcuts to bypass security.

5. In the $AccessLevel Data Change script, why is the threshold set to 5000 rather than 1000?

Click to reveal answer

The threshold of 5000 ensures that only supervisors (access level 5000) and administrators (access level 9999) can access the menu bar. Operators have access level 1000, which is below the threshold, so they cannot access the Development! fast switch or other menu options. This creates a clear security boundary between operational users and supervisory/administrative users.
