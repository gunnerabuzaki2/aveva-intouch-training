Lesson 14: Lab 6 – Creating the Mixer Components

Module 3 — Symbols | Pages 149–172

Lab Objective: Create individual symbols for each piece of mixer equipment — level meter, temperature meter, valve, agitator, and pump — using Situational Awareness library graphics. Configure each with Wizard Options, Substitute References, and Custom Properties using relative Me. references. Then link the symbols to their corresponding template objects so they can be reused across all mixer instances.

14.1 Creating the Level Meter Symbol

Create a new symbol called Level_Detailed in the Training toolset. Open it in the Graphic Editor, then use the Toolbox tab to search for "meters" and drag the SA_Meters graphic onto the canvas.

Searching the Toolbox for "meters" reveals the SA_Meters symbol from the Situational Awareness Library.
The SA_Meters symbol placed on the canvas, ready for configuration.

Rename the embedded graphic to LevelMeter and configure its Wizard Options: set Type to Level, LabelType to ObjectTagname, EngUnits to True, and EngUnitsType to CustomPropertyLabel. This configures the meter to show a level reading with the object's tag name as its label and engineering units from a custom property.

The LevelMeter Wizard Options configure the SA_Meters symbol to display level data with object tagname labels and engineering units.

14.2 Creating the Temperature Symbol by Duplication

Instead of starting from scratch, duplicate the Level_Detailed symbol and rename the copy as Temperature_Detailed. Open it for editing, select the LevelMeter element, and rename it to TemperatureMeter. Then simply change the Wizard Options Type from Level to Temperature. The graphic instantly updates to show a temperature gauge instead of a level gauge — demonstrating how powerful symbol duplication and Wizard Options are for rapid development.

Key Concept: Duplicate and Modify for Speed
When creating similar symbols (like level vs. temperature meters), duplicate an existing symbol and change only the Wizard Options. This is much faster than building each symbol from scratch and ensures visual consistency across your displays.

14.3 Creating the Valve Symbol

Create a new symbol called Valve_Detailed. Search the Toolbox for "valve_a" and drag the SA_Valve_And_Damper graphic onto the canvas. Configure the Wizard Options to match your valve type: set DiscreteCmdType to CmdOpenAndClose, Actuator to Control, LimitSwitches to Dual, LabelType to ObjectTagname, and SymbolMode to Advanced. Also enable the DirectionOfTravelIndicator so operators can see which way the valve is moving.

Now configure the Custom Properties to use relative references:

  • CmdClose → not Me.Cmd
  • CmdOpen → Me.Cmd
  • LimitSwitchClosed → Me.LSClosed
  • LimitSwitchOpen → Me.LSOpen

The not keyword inverts the Boolean value of Me.Cmd, which is how a single command attribute can drive both the open and close states of the valve display.

Valve Custom Properties use relative Me. references so the same symbol works for every valve instance in the Galaxy.

14.4 Creating the Agitator Symbol

Create Agitator_Detailed and embed the SA_Agitator_Settler graphic. Configure Wizard Options: LabelType to ObjectTagname, PVNumericDisplay to True, EngUnits to True, EngUnitsType to CustomPropertyLabel, SymbolMode to Advanced, and SetPoint to True.

The agitator's default references point to Me.PV and Me.SP, but the actual agitator data lives under a Speed sub-object. Use the Substitute References tool to remap these: right-click the graphic, select Substitute > Substitute References, then use Find & Replace to change Me.PV to Me.Speed.PV and Me.SP to Me.Speed.SP.

The Substitute Reference dialog shows the remapped references — Me.PV becomes Me.Speed.PV, Me.SP becomes Me.Speed.SP, and so on.
Agitator Custom Properties after remapping — all PV and SP references now point to the Speed sub-object.

Finally, configure the Custom Properties with the remapped references: PV → Me.Speed.PV, SP → Me.Speed.SP, EquipState → Me.PV, and so on.

14.5 Creating the Pump Symbol

Create Pump_Detailed and embed the SA_Pump_Blower_RotaryValve graphic. Configure LabelType to ObjectTagname and set the EquipState custom property to Me.PV. This is the simplest of the five symbols — just one custom property drives the pump's running/stopped state display.

14.6 Linking Symbols to Template Objects

Now comes the crucial step: associating each symbol with its corresponding template. Open each template ($Agitator, $Level, $Pump, $Temperature, $Valve) in the Object Editor, navigate to the Content pane, and click Link Content. In the Galaxy Browser, select the matching symbol from the Training toolset and click OK.

The Galaxy Browser shows all symbols in the Training toolset — select the matching symbol for each template.
After linking, each template shows its associated symbol in the Content pane.
Pitfall: Template Changes Propagate to All Derived Objects
When you link a symbol to a template and check it in, the change propagates to every derived object. In a multi-developer environment, you must notify other developers before checking in template changes. In this lab, you can proceed since only one developer is working, but in production always check the Derivation view first.

After linking all five symbols, each template now has its visual representation. When you later create instances of these templates, they'll automatically inherit the correct symbol with all the relative Me. references resolving to the instance's actual data.

14.7 Testing in Runtime

Switch to WindowMaker, open the ContentDisplay window, and use Embed Industrial Graphic to place a Level_Detailed symbol from the Instances view (not the Templates view). Select Level_001 from the Instances list — this ensures you get the instance version with actual I/O links, not the template version with Me. references. Click Runtime to verify the graphic displays live data.

At runtime, the Level_001 instance displays live data — the Me. references have been resolved to actual I/O links.

At runtime, you can use Ctrl + mouse wheel to zoom in and out of the ContentDisplay frame window. The zoom control bar appears at the bottom when zoomed in, offering zoom in, zoom out, fit, and pan controls. This zoom capability is one of the key benefits of using frame windows.

Key Concept: Instance vs. Template Graphics at Runtime
Template graphics use Me. relative references that only resolve correctly when embedded within an automation object instance. If you embed a template graphic directly in a WindowMaker window (outside any object), the Me. references won't resolve. Always use Instance graphics when embedding in WindowMaker for testing.

Quiz — Test Your Understanding

Q1: Why use the Substitute References tool instead of manually editing each reference?

Substitute References lets you remap all occurrences of a reference pattern in one operation. For the agitator, changing Me.PV to Me.Speed.PV affected 4 strings at once. Manually editing each one would be time-consuming and error-prone, especially in symbols with many data-bound elements.

Q2: What does the not keyword do in the valve's CmdClose custom property?

The not keyword inverts a Boolean value. Since the valve has a single Me.Cmd attribute (True = open, False = closed), using not Me.Cmd for CmdClose means when Cmd is True (open), CmdClose is False, and vice versa. This lets one command attribute drive both the open and close visual states.

Q3: Why must you select from Instances (not Templates) when embedding a graphic in WindowMaker for testing?

Template graphics use Me. relative references that only resolve when the graphic is embedded inside an automation object instance. When placed directly in a WindowMaker window (outside any object context), Me. has no meaning and the references won't resolve. Instance graphics have the Me. already replaced with actual I/O links to specific equipment.

Q4: What happens when you link a symbol to a template and check it in?

The change propagates to all derived objects that inherit from that template. Every instance of the template will now include the linked symbol. In a multi-developer environment, this can affect other developers' work, so you should always check the Derivation view and notify colleagues before checking in template changes.
