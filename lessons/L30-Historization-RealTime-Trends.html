Lesson 30: Historization & Real-Time Trending

Module 6 — Trend Visualization | Pages 381–386

30.1 Module Overview

This lesson marks the beginning of Module 6 — Trend Visualization. Trending is one of the most critical capabilities in any HMI system because it gives operators the ability to see not just where a process value is now, but where it has been and where it is heading. This module covers the historization architecture that makes trending possible, the Trend Pen element for real-time visualization, the Situational Awareness trend symbols, and the Multi Pens Trend control for comparing multiple process variables on a single chart.

30.2 Historization Architecture in Application Server

The history system in AVEVA Application Server uses a distributed history architecture. Each engine in a Galaxy is configured to send its historical data to a specific Historian storage node. The configuration is stored as an attribute within the engine object itself. Key architectural rules:

  • A single engine sends its history to only one Historian
  • A single Historian can receive data from only one Galaxy
  • Each object attribute that needs historization requires a corresponding Historian tag in the database — a strict one-to-one relationship

When an object attribute updates, the host engine pushes the value, timestamp, and quality of the update to Historian, which then stores it. The historized object itself is responsible for change detection and value deadbanding — not Historian.

Key Concept: Historizable Data Types
Not all data types can be historized. The supported types are: Float, Double, Integer, Boolean, String (Unicode), CustomEnum (maps to Historian Integer), and ElapsedTime (maps to Historian Float, converted to seconds). Arrays — entire or partial — are not supported. All numerical attributes are sent in their engineering units form; Historian does not perform any scaling.
AppEngine1 configuration — the History section enables storage to the Historian node (S01ENG) and controls the scan period for data collection.
Tip: The AppEngine's scan period (shown as 500ms in the screenshot) determines how frequently the engine samples attribute values. This directly affects the resolution of your historical data — a shorter scan period gives finer detail but generates more data to store.

30.3 The Trend Pen Element

The Trend Pen is a lightweight visualization element that you add to a graphic to display a single process value trend in real time. It is found in the Tools pane of the Graphic Editor and is the building block behind both the SA_Trend_SinglePen and SA_Trend_MultiPen symbols.

Once you place a Trend Pen on the canvas, the Edit Animations dialog opens where you configure:

  • Reference — The analog data source (e.g., HeatEx100.TT1.PV)
  • Historical Source — Historian (auto-detected) or InTouch Log History
  • Time Period — Moving (rolling window) or Fixed, with a duration in minutes
  • Y-Axis Range — Clip out-of-range values or Auto-Range, with Min/Max references
  • Plot Type — Step Line (default) or Line
Key Concept: Backfill
At runtime, the Trend Pen immediately fills itself with historical data from Historian — a process called backfill. This means the operator sees a populated trend chart the moment the graphic loads, rather than waiting for new data to accumulate. Once historical retrieval is complete, the pen transitions to live data, scrolling left as each new value is plotted.
Tools pane — the Trend Pen icon is located in the fifth row of the Graphic Editor toolbar.
Trend Pen configuration — the reference is HeatEx100.TT1.PV with a 5-minute moving window and Step Line plot type.

Quiz — Test Your Understanding

Q1: In Application Server's distributed history architecture, what is the relationship between engines, Galaxies, and Historian nodes?

Each engine sends history to only one Historian, and each Historian receives data from only one Galaxy. This is a strict one-to-one relationship at both levels.

Q2: What does "backfill" mean in the context of Trend Pen behavior at runtime?

Backfill is the process where the Trend Pen immediately populates itself with historical data from Historian when the graphic loads, so the operator sees a complete trend chart right away instead of waiting for new data to accumulate.

Q3: Which data types are NOT supported for historization?

Arrays (entire or partial) are not supported for historization. All other common types — Float, Double, Integer, Boolean, String, CustomEnum, and ElapsedTime — are supported.

Q4: What is the maximum number of pens supported by SA_Trend_MultiPen?

SA_Trend_MultiPen supports a maximum of four pens, displayed using the predefined element styles Trend_PenA through Trend_PenD.
