Lesson 3: Lab 1 – Creating and Deploying a Galaxy

Source: AVEVA InTouch for System Platform 2023 Training Manual, Module 1, Lab 1 (pages 11–30)

A hands-on lab where you create a Galaxy, import prebuilt Mixer objects, configure multi-node deployment and Historian, deploy the Galaxy, and verify data connections.

3.1 Lab Introduction

Reference: Module 1, Lab 1 (pages 11–12)

In this lab, you will create a Galaxy and connect to it using the System Platform IDE. Then you will import some prebuilt automation objects that represent a Mixer application and configure them for a multi-node environment. You will also configure an AppEngine to historize data to Historian.

Once the Galaxy is created and configured, you will deploy it and use Object Viewer to verify that attributes are receiving data at runtime. The Galaxy you create in this lab will be used for all other labs in this course.

Lab Objectives: Upon completion, you will be able to create a Galaxy, import objects into it, deploy those objects, and verify data connections using Object Viewer.

3.2 Creating the Galaxy

Reference: Module 1, Lab 1 (pages 12–15)

First, you will create a Galaxy based on the Blank Galaxy template and connect to it.

Step 1. Open the System Platform IDE. The Galaxies dialog box appears.
The Galaxies dialog box in the System Platform IDE, showing available Galaxy templates on the Home tab (page 1-12)
Step 2. In the New area, select the Blank Galaxy template. Be sure to select Blank Galaxy — this creates a Galaxy containing only base template objects. You will import prebuilt objects in a later step.
Step 3. In Galaxy name, enter TrainingGalaxy. The Connect to this Galaxy option is checked by default.
The Create Galaxy dialog before entering a name — the Create button is grayed out (page 1-13)
After entering TrainingGalaxy, the Create button becomes active (page 1-13)
Step 4. Click Create. The dialog shows the Galaxy creation progress and informational messages. Review the messages to ensure no errors occurred. When you see “Successfully created TrainingGalaxy,” the Close button becomes enabled.
Galaxy creation in progress — the progress bar shows the restore steps (page 1-14)
Galaxy creation completed successfully — all restore steps finished (page 1-14)
Step 5. Click Close. After a few moments, the Galaxy opens in the System Platform IDE.
The System Platform IDE with TrainingGalaxy open and ready for work (page 1-15)

3.3 Importing Training Objects

Reference: Module 1, Lab 1 (pages 16–18)

Next, you will import the prebuilt objects for the mixer application. You will use these objects for the rest of this lab and throughout the course.

Step 6. On the ribbon, select the Galaxy menu, then select Import | Objects | From package. The Import Objects from package dialog box appears.
Selecting Import > Objects > From package from the Galaxy backstage menu (page 1-16)
Step 7. Navigate to C:\Training and select Lab 1 - InTouch for System Platform 2023.aaPKG.
Selecting the Lab 1 package file for import (page 1-16)
Step 8. Click Open to begin the import process. The Import preferences dialog box appears.
Import preferences dialog — leave the defaults and click Import (page 1-17)
Key Concept: The import preferences control how conflicts are resolved when objects with the same Tagname already exist in the Galaxy. The default settings (overwrite if higher version, migrate templates, skip different codebases) are suitable for this lab.
Step 9. Leave the default import preferences and click Import. The import progress dialog shows completion status. Review messages to ensure no errors occurred.
Import completed successfully — 162 objects were imported from the package file (page 1-18)
Step 10. When import is complete, click Close. Step 11. Click the back arrow in the top-left corner of the backstage view to return to the main IDE page.
The backstage view after import — click the back arrow to return to the IDE (page 1-18)

3.4 Configuring Platforms

Reference: Module 1, Lab 1 (pages 19–21)

Next, you will configure the network address for each platform. The engineering node (GRPlatform) will be used for project development and a production node (PROD_Platform1) will be used as a runtime environment for deployment and testing.

Step 12. In the Deployment view, expand TrainingGalaxy, and then double-click GRPlatform to open its configuration editor. Step 13. On the General tab, in the Network address field, enter the local computer name (your instructor will provide this — in the example, S01ENG). Step 14. Click the Save and Close button.
Deployment view with GRPlatform highlighted — double-click to open its configuration (page 1-19)
Setting the Network address for the GRPlatform (engineering node) (page 1-19)
Click Save and Close to save the GRPlatform configuration (page 1-19)

The Check-in dialog box appears.

Step 15. In Comment, enter a comment describing your changes (optional but recommended). Step 16. Click Check-in.
The Check-in dialog — enter an optional comment describing the changes (page 1-20)
Step 17. In the Deployment view, double-click PROD_Platform1 to open its configuration editor.
Deployment view with PROD_Platform1 selected for configuration (page 1-20)
Step 18. On the General tab, in Network address, enter your designated production server computer name (e.g., S01PROD). Step 19. Save, close, and check in the PROD_Platform1 object.

3.5 Configuring the Engine for Historian

Reference: Module 1, Lab 1 (pages 22–23)

Next, you will configure the Historian connection for the application engine.

Step 20. In the Deployment view, double-click AppEngine1 to open its configuration editor. Step 21. On the General tab, in Historian, enter the node name of the computer where Historian is installed (e.g., S01ENG). Step 22. Save, close, and check in AppEngine1.
Configuring the Historian connection for AppEngine1 (page 1-21)
The Deployment view showing the expanded AppEngine1 hierarchy (page 1-21)

3.6 Configuring the Data Source

Reference: Module 1, Lab 1 (pages 22–23)

You will now configure the simulated data source for the mixers. This data source will be used throughout this course.

Step 23. In the Deployment view, ensure AppEngine1 is expanded. Step 24. Double-click ProdPLC1 to open its configuration editor. Step 25. On the General tab, in Server node, enter the node name of the computer providing data (e.g., S01PROD). Step 26. Save, close, and check in ProdPLC1.
Configuring the ProdPLC1 data source with the production server node (page 1-22)

3.7 Deploying the Galaxy

Reference: Module 1, Lab 1 (pages 23–25)

Now you will deploy the Galaxy. Deployment sends all configured objects to their target platforms where they execute at runtime.

Step 27. In the Deployment view, right-click TrainingGalaxy and select Deploy.
Right-click TrainingGalaxy and select Deploy to begin deployment (page 1-23)
The deployment context menu with Deploy option selected (page 1-23)

The Deploy dialog box appears showing deployment options.

Tip: For a first-time deployment, the default settings are typically appropriate. Cascade deploy ensures all child objects are deployed along with their parent. The “On scan” initial state means objects begin collecting data immediately after deployment.
The Deploy dialog with default deployment settings (page 1-24)
Step 28. Leave the deployment default settings and click Deploy. Review the messages to ensure no errors occurred. Step 29. When deployment shows “Deploy complete,” click Close.
Deployment completed successfully — all 127 objects were deployed (page 1-25)

3.8 Verifying Data in Runtime

Reference: Module 1, Lab 1 (pages 25–27)

Finally, you will verify the data in runtime by viewing object attributes in Object Viewer. This confirms that your deployment is working and attributes are receiving live data.

Step 30. In the Deployment view, expand Line1, right-click Mixer100, and select View in Object Viewer.
Right-click Mixer100 and select View in Object Viewer to inspect live data (page 1-25)

The Object Viewer window appears, showing the list of attributes for Mixer100.

Step 31. At the bottom of the window, right-click in the Watch List 1 pane and select Open.
Object Viewer displaying live attribute data for Mixer100 (page 1-26)
Right-click the Watch List pane and select Open to load a saved watch list (page 1-26)
Step 32. Navigate to C:\Training, select Lab 1 - WatchList, and click Open. Step 33. Verify that the quality is Good for every attribute in the Mixer100 and ProdPLC1 watch windows. Notify your instructor if any attribute shows a different quality status. Step 34. When finished, close Object Viewer.
Key Concept: The “Good” quality code confirms that each attribute is receiving valid data from the simulated data source. If quality shows anything other than Good, it indicates a communication or configuration issue that needs investigation.

<End of Lab>

Knowledge Check

1. Why do you select the Blank Galaxy template instead of the Default Galaxy template when creating a new Galaxy?

Click to reveal answer

The Blank Galaxy template creates a Galaxy containing only base template objects, allowing you to import specific prebuilt objects into a clean environment. The Default Galaxy would include pre-existing objects that might conflict with the training objects.

2. What must be configured before deployment to ensure objects communicate on the network?

Click to reveal answer

Each platform must have its Network address set to the correct computer name (e.g., S01ENG for the engineering node, S01PROD for the production node). Without correct network addresses, the IDE cannot communicate with the target hosts.

3. What does a quality code of “Good” in Object Viewer indicate?

Click to reveal answer

It confirms that the attribute is receiving valid data from the data source. A quality code other than Good indicates a communication or configuration problem that needs investigation.

4. How many objects were imported from the training package and deployed to the Galaxy?

Click to reveal answer

162 objects were imported from the package file, and 127 objects were deployed to the Galaxy during the deployment step.
