Lesson 3: Lab 1 – Creating and Deploying a Galaxy

Module 1 — Introduction | Pages 11–30

3.1 Lab Overview

Lab Objectives
Upon completion of this lab, you will be able to:
  • Create a Galaxy
  • Import objects into a Galaxy
  • Deploy objects
  • Verify data connections using Object Viewer

In this foundational lab, you will create a Galaxy and connect to it using the System Platform IDE. You will then import prebuilt automation objects representing a Mixer application and configure them for a multi-node environment. Once the Galaxy is configured, you will deploy it and use Object Viewer to verify that attributes are receiving data at runtime. The Galaxy you create here will be used for every subsequent lab in this course.

Key Concept: What is a Galaxy?
A Galaxy is the central repository in Application Server that stores all automation objects, templates, graphics, and configuration for your industrial application. Think of it as the master database and project workspace that everything else is built from.

3.2 Creating the Galaxy

To begin, open the System Platform IDE. The Galaxies dialog box will appear, showing you a selection of templates to start from. For this lab, you will use the Blank Galaxy template, which creates a Galaxy containing only base template objects. This gives you a clean starting point into which you can import the prebuilt mixer objects.

In the New area of the Galaxies dialog, select the Blank Galaxy template and click Create. In the Create Galaxy dialog that appears, enter TrainingGalaxy as the Galaxy name. The "Connect to this Galaxy" option is checked by default, which means the IDE will automatically connect to your new Galaxy once it is created.

The Galaxies dialog box in the System Platform IDE, showing available templates including Blank Galaxy, InTouch Base, and Reactor Demo.
The Create Galaxy dialog with "TrainingGalaxy" entered as the name and the Blank Galaxy template selected.

Click Create and wait for the progress bar to complete. The dialog will display the steps as they execute — restoring the galaxy, getting database and file repository information, and deploying ASB services. When the process indicates "Successfully created TrainingGalaxy," click Close. The Galaxy will open in the System Platform IDE.

The Galaxy creation process completing successfully — all steps should show green checkmarks with no errors.
The System Platform IDE after Galaxy creation, showing the empty TrainingGalaxy ready for configuration.
Important: Always select the Blank Galaxy template for this lab, not the Default Galaxy template. The Blank Galaxy contains only base template objects, which ensures the prebuilt mixer objects you import later will not conflict with existing objects.

3.3 Importing the Training Objects

With your Galaxy created and connected, the next step is to import the prebuilt objects for the mixer application. On the ribbon, select the Galaxy menu and choose Import > Objects > From package. This opens the System Platform IDE backstage view with the Import Objects dialog.

Navigate to C:\Training and select the file Lab 1 - InTouch for System Platform 2023.aaPKG. Click Open to begin the import process. The Import preferences dialog will appear with default conflict resolution rules:

Conflict TypeDefault Action
Same Tagname and CodebaseOverwrite if higher version
Base Templates with different revisionMigrate
Same Tagname but different CodebaseSkip

Leave the default import preferences and click Import. Review the progress messages carefully to ensure no errors occurred. When the import indicates "Import completed," click Close and then use the back arrow in the top-left corner to return to the main IDE page.

The import completion dialog — note the summary showing 162 objects imported from the package file.

3.4 Configuring the Platforms

The training objects include two platforms: GRPlatform (the engineering node for development) and PROD_Platform1 (the production node for runtime deployment and testing). Each platform must be configured with the correct network address for your environment.

In the Deployment view, expand TrainingGalaxy and double-click GRPlatform to open its configuration editor. On the General tab, enter the local computer name in the Network address field — your instructor will provide the correct node name. Save and close the editor, then check in the object when prompted.

The GRPlatform configuration editor showing the Network address field where you enter your engineering node name.

Repeat the process for PROD_Platform1, entering the designated production server computer name. Save, close, and check in the object. These platform configurations are critical — they tell System Platform which computers will host the engineering and production environments.

Tip: Always enter a comment when checking in objects. While optional, these comments create a valuable audit trail that helps you and your team track what changes were made and why.

3.5 Configuring the Engine for Historian

Next, configure the Historian connection for the application engine. In the Deployment view, double-click AppEngine1 to open its configuration editor. On the General tab, ensure that "Enable storage to historian" and "Enable Tag Hierarchy" are both checked. In the Historian field, enter the node name of the computer where Historian is installed (your instructor will provide this). Save, close, and check in the object.

The AppEngine1 configuration editor showing Historian settings — both storage and Tag Hierarchy should be enabled.

3.6 Configuring the Data Source

The final configuration step is setting up the simulated data source for the mixers. This data source will be used throughout the entire course. In the Deployment view, expand AppEngine1 and double-click ProdPLC1 to open its configuration editor.

On the General tab, set the Server node to the computer that will provide data to the Galaxy. The Server name should already be set to MBTCP (Modbus TCP). Save, close, and check in the object.

ProdPLC1 configuration showing the Server node and Server name (MBTCP) fields.

3.7 Deploying the Galaxy

With all configurations in place, it is time to deploy the Galaxy. In the Deployment view, right-click TrainingGalaxy and select Deploy. The Deploy dialog box will appear with default deployment settings.

Key Concept: Deployment
Deployment is the process of pushing your configured objects from the IDE (development environment) to the runtime environment on the target nodes. Once deployed, objects begin executing — scanning for data, triggering alarms, and making information available to visualization clients.

Leave the deployment default settings (Cascade deploy checked, Force off scan checked, Preserve runtime changes checked, On scan selected) and click Deploy. The Deploying Objects dialog will show the progress of the deployment process. Review the messages to ensure no errors occurred. When deployment is complete, click Close.

The Deploy dialog box — leave the default settings as shown and click Deploy.
The deployment completion dialog showing all 127 objects deployed successfully with no errors.

3.8 Verifying Data in Runtime

The final step is to verify that your deployed Galaxy is actually receiving data at runtime. You will use Object Viewer for this purpose. In the Deployment view, expand the Line1 area and right-click the Mixer100 instance. Select "View in Object Viewer" from the context menu.

Object Viewer showing attributes for Mixer100 — all quality values should display "C0:Good" indicating healthy data connections.

The Object Viewer window will display the list of attributes for Mixer100. At the bottom of the window, right-click in the Watch List 1 pane and select Open. Navigate to C:\Training, select the Lab 1 - WatchList file, and click Open. The Mixer100 and ProdPLC1 watch windows will appear with pre-configured attributes.

Real-time attribute data in Object Viewer — verify that the Quality column shows "C0:Good" for every attribute. If any attribute shows a different quality status, notify your instructor.
Warning: If any attributes display a quality status other than "Good," this indicates a communication problem with the data source. Notify your instructor immediately rather than proceeding, as this will affect all subsequent labs.

Once you have verified that all attributes are receiving data with Good quality, close Object Viewer. This completes Lab 1.

Quiz — Test Your Understanding

Q1: What template should you select when creating the Galaxy for this lab, and why?

You should select the Blank Galaxy template because it creates a Galaxy containing only base template objects, ensuring no conflicts when you import the prebuilt mixer objects in later steps.

Q2: What file format is used when importing objects into the Galaxy?

Objects are imported from .aaPKG (AVEVA Application Package) files. In this lab, the file is "Lab 1 - InTouch for System Platform 2023.aaPKG".

Q3: How many objects were imported and deployed in this lab?

The import brought in 162 objects from the package file, and the deployment successfully deployed 127 objects out of 127 selected.

Q4: What tool do you use to verify that attributes are receiving data at runtime?

Object Viewer, which can be accessed by right-clicking any object in the Deployment view and selecting "View in Object Viewer."

Q5: What does a quality status of "C0:Good" indicate?

It indicates that the attribute is communicating properly with the data source and receiving valid, healthy data. Any other quality status suggests a communication problem.
